## Appendix A-1 — F-ENG-1 expanded (high severity)

- **Bug class**: backward-compatibility / improper handling of an unexpected enum value → CWE-754 (Improper Check or Handling of Exceptional Conditions). Adjacent: CWE-697 (incorrect comparison) in `all_done`.
- **Location**: `crates/sldo-common/src/runbook.rs:12` (`enum MilestoneStatus`), `:33-36` (`FromStr`, unknown→`Err`), `:58` (`status_re = (not_started|in_progress|done)`), `:69` (`.unwrap_or(NotStarted)`), `:113` (`all_done`), `:118` (`next_incomplete`).
- **Reachability**: `pub mod runbook` in `crates/sldo-common/src/lib.rs:9`; `sldo-common` is published on crates.io (v0.1.2) and depended on by `sldo-install` / `sldo-research`. Not *currently* called by those crates' runtime paths (so no live SLO-skill break), but it **is** public API surface: an external consumer or a future internal caller parsing a Secure-Value-Loop runbook gets wrong results.
- **Concrete exploit/defect**: a runbook with a `blocked_by_operator` (or even today's documented `blocked`) row → regex no-match → row dropped from `parse_tracker` → `all_done()` returns `true` over the remaining all-`Done` rows → "runbook complete" while a milestone is blocked. Silent false-positive completion is exactly the "no silent `done`" red line this runbook exists to enforce — violated by its own toolchain.
- **Class-elimination view**: the *class* (status value the parser doesn't know about) is **not eliminated** by the current runbook — M3 only edits Markdown/skill prose. To eliminate the class: make `MilestoneStatus` total over the documented set with an explicit `Unknown`/`Blocked`-defaulting arm and an exhaustive `all_done` that treats any non-`Done` (including unknown) as "not done".
- **Variant analysis** (per playbook): after fixing `runbook.rs`, grep the workspace for other status/enum string-matching that assumes the three-value set — `grep -rn "not_started\|in_progress\|done" crates/ xtasks/` — and any Markdown/structural test that hard-codes the old enum (e.g. `mloop_m3_plan.rs` status assertions) so the additive change is consistent everywhere.
- **Remediation**: in M3, extend `enum MilestoneStatus` with `Blocked` + the five new variants (or a single `Other(String)` carrier), update the regex to match them (or capture the whole status token and `FromStr` it), change `.unwrap_or(NotStarted)` → `.unwrap_or(Blocked)`, make `all_done` require every row `== Done` (already true — but ensure dropped-row path can't occur by capturing all rows), and add `runbook.rs` unit tests round-tripping every documented status. Record the crates.io version bump in the milestone (version discipline per the published-crates memory).
