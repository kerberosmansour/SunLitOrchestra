# Reversibility — slo-threat-model

Hard-to-change decisions, why each is hard to reverse, the reversibility
tactic, the rollback / migration path, and the proof required.

| Decision | Why hard to change | Reversibility tactic | Rollback / migration path | Proof required |
|---|---|---|---|---|
| `tm-<slug>-abuse-N` ID convention (reuse existing, no `AC-N`) | Once any `.slo.json` is committed and a runbook cites an `id`, renaming the scheme breaks every citation — this is the exact failure the wedge exists to prevent | Reuse the convention already in `skills/slo-architect/references/threat-model-template.md`; never introduce a parallel scheme | If a scheme change is ever forced, it is a migration runbook that rewrites all citations atomically with a supersession map; never an in-place rename | Structural-contract test asserts the regex `^tm-<slug>-abuse-\d+$` and the frozen-ID invariant; grep shows no `AC-N` parallel scheme |
| SLO-owned minimal JSON schema (not OTM, not TM-BOM) | Downstream skills and the future producer bind to the shape; changing the canonical store later is a data migration across every committed `.slo.json` | Keep the schema minimal + additive; `slo_schema_version` (semver) gates evolution; OTM/TM-BOM are *exports*, never the store | Add an export adapter (additive, non-breaking). Replacing the store is a major-version migration with a converter and a deprecation window | `slo_schema_version` present and bumped on any breaking field change; export adapters (if added later) are additive-only, asserted by test |
| Frozen-ID invariant enforced by a Rust structural-contract test (not prose) | If the guard is later weakened to prose, the original wrong-test failure recurs silently | The invariant lives in a deterministic test under `xtasks/sast-verify/tests/`, run by `cargo test`; binds the rule, not the prose around it | Strengthening is additive. Weakening requires an explicit runbook + recorded residual risk acceptance; never a silent test deletion | The test fails on a renumbered/reused `id` in a fixture; deleting/relaxing the test is visible in diff and blocked by review |
| Schema doc location `references/security/threat-model-schema.md` | Skills cite the path; moving it breaks citations like the SAST manifest precedent | Place it in the established `references/security/` review/verification subtree from the start | A move is a tracked rename with citation updates in the same PR; never an orphaned path | `sap_imp_m1`-style citation test (paths resolve at HEAD) covers the new doc |
| Provenance = producing-skill SKILL.md git SHA + input SHAs (not content hash) | Switching provenance idiom later splits the repo into two inconsistent conventions | Match `references/sast/manifest-schema.md` from day one | If a content-hash is ever also wanted, add it as an *additional* field; do not replace the SHA idiom | Interfaces doc + test assert `provenance.producer_skill_sha` and `provenance.inputs[].sha` present |
| Consumers **halt** on absent/invalid JSON | Too-strict halt jams in-flight pre-schema runbooks (Top-risk outage) | The halt is scoped: a *documented degraded mode* for runbooks with no `.slo.json` yet (warn + proceed without ID-stability claim), a *hard halt* only when a JSON exists but is schema-invalid | Reverting to silent re-derive is forbidden (defeats the wedge); the safe rollback is widening the degraded-mode path, not removing the contract | `/slo-plan` milestone must specify the degraded-mode vs hard-halt boundary; `/slo-verify` exercises both a missing-file and an invalid-file path |

No `N/A` rows — every decision above is genuinely hard to reverse, which is
why the wedge is read-side-only and additive by construction.
