# Threat Model — scanner-orchestration

> Generated by `/slo-architect` on 2026-04-26.
> Scope: the new `/slo-sast` skill emitted by the v1 runbook plus the artifacts it writes into a target repo (`.semgrep/`, `.github/workflows/sast.yml`, `.semgrep/manifest.json`).
> This file is the upstream artifact `/slo-plan`, `/slo-critique`, and `/slo-verify` cite for this runbook.
> User-provided strings render inside `~~~text` fences so raw Markdown / YAML / HTML metacharacters are literal text, not interpretable.

## System description

~~~text
/slo-sast is a Markdown Claude Code skill (skills/slo-sast/SKILL.md). When invoked it: (1) parses CWE references from the user's threat-model file, (2) detects target repo stack from manifest files, (3) clones github.com/semgrep/semgrep-rules at a pinned SHA into ~/.cache/sldo/semgrep-rules/<SHA>/, (4) filters rules by metadata.cwe ∧ metadata.technology, (5) writes selected rule files to .semgrep/rules/, .semgrep.yml config, .github/workflows/sast.yml workflow, .semgrep/manifest.json audit-defense artifact, and (6) opens a PR.

Trust boundaries: User ↔ Claude Code (agent interprets intent); Claude Code ↔ skill SKILL.md (read-only Markdown); skill ↔ semgrep-rules upstream (network — git/GitHub, integrity via SHA pin); skill ↔ target repo filesystem (writes into .semgrep/ and .github/workflows/); target repo ↔ GitHub Actions runner (out of SLO scope, but the workflow's safety properties are the load-bearing reason this whole skill exists).
~~~

## Top risks (from idea doc)

~~~text
Top risks captured by /slo-ideate scanner-orchestration on 2026-04-26:

- Breach: pull_request_target footgun in the auto-emitted workflow. The skill (or a future contributor / Claude run) writes on: pull_request_target, granting forked PRs the upstream secret context. Adversary: drive-by malicious PR contributor exfiltrating GITHUB_TOKEN or any repo secret. Surface: the auto-emitted CI YAML.

- Compliance fine: False coverage claim against PCI DSS 6.2.3 (v4.0.1 — bespoke and custom software code review) or SOC 2 CC7.1. The skill emits a coverage doc claiming CWE-77 / CWE-78 / CWE-89 are covered because the threat model named them, but the chosen rule packs don't actually have rules for the user's specific framework. User cites the doc in audit; auditor disproves. Data class: whatever the regulated app processes (cardholder data for PCI, customer PII for SOC 2 CC7.1 scope). Note: v3.2.1 numbering for code review was 6.3.2; v4.0.1 renumbered to 6.2.3 and v4.0.1's 6.3.2 is now the bespoke / third-party software inventory mandate (different scope, out of v1).

- Prolonged outage: Baseline-day CI jam in existing repos. Skill onboards a mature project, baseline doesn't capture all 200 legacy critical findings, fail-on-new-critical degrades to fail-on-everything. First noticer: the next maintainer trying to merge a PR (within hours of install). Defection: team disables the workflow within 1–3 days; SLO skill abandoned by precisely the user who most needed it.
~~~

## Residual risks

- **Between-install SKILL.md tampering.** Inherited from slo-security-embedding. The `sldo-install` manifest detects tampering at install/re-install boundaries but not continuously. Mitigation requires either filesystem-level integrity monitoring (out of SLO scope) or a runtime SHA check (deferred — see open questions in research dossier).
- **`semgrep-rules` upstream key compromise.** SHA pin defends against tag rewriting but not against an attacker who can reissue commits with the same SHA prefix (SHA-1 collision attacks are computationally costly but documented). Mitigation: GitHub's planned SHA-256 migration (no firm date as of 2026-04). Track the [Git SHA-256 transition issue](https://github.com/git/git/blob/master/Documentation/technical/hash-function-transition.adoc).
- **Cache pruning unmanaged.** `~/.cache/sldo/semgrep-rules/<SHA>/` grows unbounded; multiple pin bumps accumulate snapshots. Disk-quota DoS is theoretical for solo OSS maintainers but real for shared CI environments. Pruning skill is out of v1 scope.
- **Initial-baseline preview is advisory, not enforced.** The emitted workflow correctly uses GHA auto-baseline so PR blocking is diff-aware, but a user who disables the preview confirm step (or doesn't read it) could still ship a workflow they don't fully understand. Mitigation is documentation + the structural-contract test suite that runs before the PR is opened.
- **`/slo-rulegen` integration is contract-defined but not exercised in v1.** A v1 user who never invokes `/slo-rulegen` will never test the integration boundary. Mitigation: M5 includes a fixture that simulates a `/slo-rulegen`-authored rule alongside registry rules and asserts the manifest distinguishes them correctly.
