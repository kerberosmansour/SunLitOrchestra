## Topic Decomposition

- **Q1a — Baseline mechanics**: How does `semgrep ci --baseline-ref` behave on `pull_request` events vs `merge_group`, and how does it handle force-pushed PRs and rebase scenarios? (Rationale: existing-repo onboarding without day-one CI jam.)
- **Q1b — Severity gating**: Is `--severity ERROR` (CLI flag) or rule-level `severity: ERROR` (YAML metadata) the recommended threshold mechanism in `semgrep ci`, and how do they compose? (Rationale: fail-on-new-critical-only is the wedge.)
- **Q1c — OSS-only PR comments**: Without Semgrep AppSec Platform, what's the documented path for inline PR-comment posting (e.g., `semgrep-action`, SARIF + `github/codeql-action/upload-sarif`, third-party `reviewdog`)? (Rationale: maintain OSS posture per recommendation.)
- **Q2a — Registry CWE-tag density**: What fraction of rules in `p/owasp-top-10`, `p/cwe-top-25`, `p/r2c-security-audit`, `p/security-audit` carry a `metadata.cwe` field, per language? (Rationale: decides "use registry index" vs "ship our own table".)
- **Q2b — Registry queryability**: Is there a stable, queryable index by CWE (registry API, GraphQL, downloadable manifest) or is grep-the-YAML the only path? (Rationale: skill needs a deterministic lookup at runtime.)
- **Q3 — Audit-acceptable evidence shape**: What artefact format (SARIF, CSV, signed PDF), retention duration, and "addressed" framing (risk-acceptance memo, compensating control) do PCI DSS 6.3.2 and SOC 2 CC7.1 audits actually accept for SAST coverage? (Rationale: v1 must capture enough metadata per scan to support a later coverage-claim doc.)
- **Q4a — `pull_request_target` necessity**: Is there *any* scenario where SAST genuinely needs `pull_request_target` (vs `pull_request` + `workflow_run`)? (Rationale: idea-doc names this as the breach risk.)
- **Q4b — Permissions-block minimum**: What's the cited minimum `permissions:` block for a `pull_request`-triggered Semgrep job (read source, write SARIF, post comment)? (Rationale: every line needs a citation.)
- **Q4c — Two-workflow split pattern**: Is the analysis-on-`pull_request` + comment-on-`workflow_run` split documented as the GitHub-recommended fork-PR safe pattern? (Rationale: required if PR comments are needed.)
- **Q5a — OTM → Semgrep converters**: Does any published tool consume OWASP Threat Dragon's OTM JSON, pytm CWE output, or similar to emit Semgrep `--config`? (Rationale: defensible novelty claim.)
- **Q5b — Vendor adjacents**: Do CodeQL config files, Snyk Code policies, or Checkmarx presets accept threat-model-derived inputs, or only manual rule selection? (Rationale: identifies who could pivot in.)

## Key Questions

1. **Q2 (registry CWE-tag density)** — gates whether the skill is "10 lines of glue" or "ships a 200-row reference table". Highest leverage on scope.
2. **Q4 (workflow safety)** — load-bearing risk per the idea doc; the YAML the skill emits *is* the breach surface. Wrong here = CVE-via-tool.
3. **Q1 (baseline + severity)** — load-bearing usability risk; baseline-day CI jam is the documented defection mode.
4. **Q5 (prior art)** — determines whether the wedge is "novel auto-tuning loop" or "thin wrapper over an existing converter". Affects positioning, not feasibility.
5. **Q3 (audit shape)** — important for the v1 capture-format decision but the coverage doc itself is post-v1; second-order on shipping date.

## Initial Findings

### Semgrep CI mechanics (Q1)

- `semgrep ci` is the modern CI entry point; `semgrep-action` (the GitHub Action) was effectively deprecated in favour of running `semgrep ci` inside a generic runner step. **Confidence: high** (well-documented on semgrep.dev through 2024–2025; needs verification for 2026 state).
- `semgrep ci` automatically detects PR context from `GITHUB_*` env vars, computes a baseline against the PR base SHA, and only reports findings introduced by the PR. Without AppSec Platform login it runs in "OSS mode" reading config from `--config` flags or `.semgrep.yml`. **Confidence: high.**
- `--baseline-ref <sha>` overrides baseline detection. On `merge_group` events the base is the merge-queue tip, not `main`; this can produce surprising baseline diffs. **Confidence: medium** — needs verification against current docs.
- Severity gating: rule-level `severity:` in YAML is what Semgrep matches; `--severity ERROR` filters output to ≥ ERROR. The pattern that fails the build on new ERROR-only is `semgrep ci --severity ERROR --baseline-ref ...`. **Confidence: medium.**
- PR-comment posting OSS-path-1: emit SARIF (`--sarif --output results.sarif`) and upload via `github/codeql-action/upload-sarif@v3` — comments appear in GitHub's Code Scanning UI tab, not as inline review comments. Requires `security-events: write` permission and (historically) GitHub Advanced Security or public repo. **Confidence: medium** — needs verification.
- PR-comment posting OSS-path-2: `reviewdog` with the `semgrep` reporter, posting via `GITHUB_TOKEN`. **Confidence: medium.**
- Common pitfall: `--baseline-ref` requires the base ref to be fetched (`fetch-depth: 0` on `actions/checkout`). Default fetch-depth=1 produces "baseline ref not found" errors. **Confidence: high.**
- Common pitfall: force-pushed PRs invalidate cached baseline state in AppSec Platform; OSS path doesn't cache so is unaffected. **Confidence: low** — needs verification.

### CWE → rule pack mapping (Q2)

- Semgrep registry rule YAMLs do carry a `metadata.cwe` field, but coverage is uneven. Older rules (pre-2021) often have it blank; newer registry-curated rules consistently populate it. **Confidence: medium** — exact percentage needs empirical scrape.
- `p/owasp-top-10` is curated to map to OWASP 2021 categories rather than to specific CWEs; CWE tags inside rules are advisory rather than the index. **Confidence: medium.**
- `p/cwe-top-25` exists as a registry pack and is the closest published CWE-indexed pack. **Confidence: medium** — needs verification it still ships and its 2025 update.
- The Semgrep Registry HTTP API exists at `https://semgrep.dev/api/registry/...` but stable, documented endpoints for "list all rules where metadata.cwe = CWE-77" are not advertised; common practice is to clone `semgrep/semgrep-rules` (the canonical OSS rule repo) and grep YAML. **Confidence: medium-low** — needs verification.
- The `semgrep/semgrep-rules` repo is MIT/LGPL-mixed; copying YAML for an OSS skill pack is generally fine. **Confidence: medium.**
- **Tentative verdict (needs verification)**: registry CWE tagging is queryable via grep-of-cloned-repo but not via a documented API; the skill should ship a curated CWE → pack-or-rule-set reference table that cites the registry path for each entry, rather than relying on live registry queries. This matches the precedent in `references/sast/cwe-map-rust.md` already in the repo.

### Audit evidence (Q3)

- PCI DSS v4.0 6.3.2 ("custom and bespoke software is reviewed prior to release to identify and address any potential coding vulnerability") is satisfied by either manual code review or automated SAST; the QSA wants evidence of (a) the tool ran, (b) findings were triaged, (c) what was fixed/accepted. **Confidence: medium** — needs verification of the exact wording and supplemental guidance.
- SOC 2 CC7.1 is broader ("entity uses detection and monitoring procedures to identify (1) changes to configurations that result in the introduction of new vulnerabilities"); SAST-in-CI is a common control mapped here. The Trust Services Criteria text doesn't prescribe format. **Confidence: medium.**
- SARIF (Static Analysis Results Interchange Format, OASIS spec) is the de-facto auditor-acceptable format because it's vendor-neutral and includes rule ID, location, severity, and CWE if the tool emits it. **Confidence: high.**
- Retention: PCI DSS generally requires audit records ≥ 1 year (12 months) with 3 months immediately accessible; SOC 2 commonly retains for the audit period plus one year. **Confidence: medium** — varies per auditor.
- Risk-acceptance memo is the documented path for "we saw it and chose not to fix"; auditors expect a signed acceptance with named owner, residual-risk statement, and review date. **Confidence: medium.**
- "Mapped-but-not-actually-scanned" coverage claim is a documented audit-finding pattern in QSA blog posts (Coalfire, Schellman); auditors increasingly ask for tool config + scan logs proving rules ran. **Confidence: low** — needs verification with citations.
- **Tentative v1 capture list**: per scan run, persist (a) Semgrep version, (b) ruleset hash / rule IDs invoked, (c) target commit SHA, (d) SARIF output, (e) baseline ref SHA, (f) wall-clock timestamps, (g) exit code. Threat-model CWE list snapshot at scan time goes alongside.

### `pull_request_target` security (Q4)

- GitHub's own Security Hardening guide (github.com/security docs) explicitly warns that `pull_request_target` runs in the *base* repo's context with secrets and `GITHUB_TOKEN` — and runs untrusted PR code if you `actions/checkout` the PR head. **Confidence: high.**
- The 2020 Trail of Bits "Untrusted input is untrusted, even if it's YAML" series is the canonical writeup. Subsequent ToB / GitHub Security Lab posts (2022–2024) extend it. **Confidence: high.**
- For SAST: `pull_request` is sufficient for the analysis job because Semgrep doesn't need any repo secrets to analyse code. The only reason to want `pull_request_target` is to post results back as a PR comment using the base repo's `GITHUB_TOKEN`, which fork PRs' default `GITHUB_TOKEN` cannot do. **Confidence: high.**
- Documented pattern: split into two workflows. Workflow A (`pull_request`) runs Semgrep, uploads SARIF as a workflow artifact. Workflow B (`workflow_run`, type completed) downloads the artifact and posts the comment using the repo's permissions. This is the GitHub Security Lab's "Preventing pwn requests" pattern. **Confidence: high.**
- Minimum permissions for a `pull_request` Semgrep job: `contents: read`, plus `security-events: write` *only* if uploading SARIF to Code Scanning. Everything else explicit `none`. **Confidence: high.**
- "Approve and run" gating (the "require approval for first-time contributor" GitHub setting) is a defense-in-depth layer but not sufficient on its own — a maintainer can be socially engineered. **Confidence: medium.**
- CVE precedents: `pytorch/pytorch` self-hosted runner takeover (Oct 2024 disclosure), multiple `actions/checkout @ pull_request_target` exfil cases, GitGuardian's "Worst-of-Both-Worlds" series. **Confidence: medium** — needs citation lookup.

### Prior art (Q5)

- **OWASP Threat Dragon**: emits OTM (Open Threat Model) JSON. OTM is a published spec at `github.com/iriusrisk/OpenThreatModel`. No published, maintained converter from OTM → Semgrep config exists in widespread use. **Confidence: medium** — needs broader search.
- **pytm**: Python threat-modelling DSL; emits report formats but no Semgrep converter ships in-tree. **Confidence: medium.**
- **MITRE Threagile**: Go-based threat modeller; emits risk reports tagged with CWE. No Semgrep generator. **Confidence: low** — needs verification.
- **GitHub CodeQL**: config file (`.github/codeql/codeql-config.yml`) selects query packs by name/CWE-tag but does not consume threat models. **Confidence: high.**
- **Snyk Code "policy as code"**: lets you write YAML policies setting severity overrides per rule, but doesn't ingest threat models. **Confidence: medium.**
- **Checkmarx One presets**: manually-curated rule subsets; no threat-model intake. **Confidence: medium.**
- **Veracode policy framework**: severity + CWE-class gating for builds; no threat-model intake. **Confidence: medium.**
- Academic: papers on "automated SAST configuration" exist (e.g., 2022–2024 work on rule selection via static analysis of dependencies), but threat-model-driven rule selection specifically is sparse. **Confidence: low** — needs literature search.
- **Tentative landscape verdict**: the threat-model-edit → ruleset-rederive loop appears to be unclaimed. Adjacent work exists (CodeQL config, vendor presets) but the loop closure (diff → human review → redeploy) is not a feature in any vendor's public roadmap as of training data.

## Repo Context

- **Stack**: Rust 2021 workspace + Markdown skill pack consumed by Claude Code. Workspace = `crates/sldo-common`, `crates/sldo-research`, `crates/sldo-install`, `xtasks/sast-verify`. Baseline test: `cargo test --workspace`.
- **Skill-pack convention**: every `/slo-*` skill is a directory under `skills/<name>/` with a `SKILL.md` carrying YAML frontmatter (name, description). The `slo-rulegen` skill (`skills/slo-rulegen/SKILL.md`) is the closest precedent for `/slo-sast` — same prompt-driven, references-table-backed pattern.
- **Existing SAST infrastructure (the new skill must NOT duplicate)**:
  - `.semgrep/rust/` ships 10 CWE-tagged rules (CWE-20, 79, 125, 190, 295, 416, 672, 697, 755, 787) with paired `.rs` fixtures.
  - `references/sast/cwe-map-rust.md` is a curated CWE → Rust-susceptibility table with provenance via RustSec → GHSA → OSV `cwe_ids` join.
  - `references/sast/variations/cwe-NNN.md` per-CWE variation templates with `sink_shapes`, minimum-pattern-either-arms.
  - `references/sast/CI-WIRING.md` already documents a two-job GitHub Actions workflow (`admission-control` runs `cargo xtask sast-verify gate`, `semgrep` runs `semgrep ci`). The `/slo-sast` workflow emitter must align with or extend this template, not contradict it.
  - `references/sast/MIN-SEMGREP-VERSION.md` pins Semgrep ≥ 1.50.0.
- **Threat-model contract**: `/slo-architect` already emits `docs/slo/design/<slug>-threat-model.md` with STRIDE × component, abuse cases, compliance mapping. The `/slo-sast` skill's input is therefore a known-shape file produced by an in-pack skill, not an arbitrary user format. CWE references inside threat models follow the format used in `docs/slo/design/biz-skill-pack-threat-model.md` and `docs/slo/design/sast-rulegen-skill-pack-threat-model.md`.
- **Project-wide constraints from SECURITY.md**:
  - Subprocess hygiene rule: `Command::new("semgrep")` with explicit args, `Stdio::piped()`, drain on separate thread, `--timeout` and `--max-target-bytes` always passed. No shell. (Inherited from `xtasks/sast-verify`.)
  - WebFetch/WebSearch denial pattern: `slo-rulegen` denies these via toolflags and SKILL.md prose. `/slo-sast` should follow the same pattern — the threat model and CWE map ship pre-baked; no network needed.
  - Two-tier output convention: confidential drafts → `docs/biz/` (gitignored); public artefacts → `docs/biz-public/` (git-tracked). For SAST, the parallel convention applies to corpora; emitted CI YAML files (`.github/workflows/sast.yml`, `.semgrep.yml`) belong in the user's repo root and are public-tier by nature.
  - "Atomic write via temp dir + `fs::rename`" is the pattern (per `/slo-rulegen` extend mode); `/slo-sast` should follow it for the workflow YAML emission.
  - SHA-pinning of GitHub Actions: `actions/checkout` pinned by SHA (current pin: `692973e3d937129bcbf40652eb9f2f61becf3332` v4.1.7); `returntocorp/semgrep-action` pin-by-SHA convention. The `/slo-sast` template must do the same.
  - Trail-of-Bits AGPL clean-room policy: no copying of AGPL YAML; structural inspiration only.
- **Recommended approach in idea doc**: Approach A — pure-Markdown skill, ~0.5–1 person-week. Approach B (Semgrep AppSec SaaS) and Approach C (Rust crate `sldo-sast`) explicitly rejected. Research must remain consistent with this scope: no Rust backend, no SaaS dependency.
- **Out of scope explicitly**: DAST tooling (separate `/slo-dast` runbook), `/slo-rulegen` integration mechanics (follow-on after `/slo-sast` ships), business-model questions.
- **`tla_required: false`** per the idea-doc frontmatter — this is glue + table-lookup work, not concurrent state requiring TLA+ verification.
