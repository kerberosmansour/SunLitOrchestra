## Deepened Findings

### F1 — RustSec advisories do NOT carry CWE in their native schema; GHSA join is the data path
- **EVIDENCE**: The TOML schema in `rustsec/advisory-db/README.md` defines `id`, `package`, `categories`, `keywords`, `aliases`, `cvss`, etc. — **no `cwe` field**. Source: https://raw.githubusercontent.com/rustsec/advisory-db/main/README.md
- **EVIDENCE**: OSV's `RUSTSEC-*` records expose `database_specific.categories` and `database_specific.cvss` but **no `cwe_ids`**. The same vuln fetched by its **GHSA alias** does carry `database_specific.cwe_ids` (e.g. `RUSTSEC-2024-0421` → `GHSA-h97m-ww89-6jmq` → `["CWE-1289","CWE-697"]`). Source: https://api.osv.dev/v1/vulns/RUSTSEC-2024-0421 + https://api.osv.dev/v1/vulns/GHSA-h97m-ww89-6jmq
- **Implication**: the bootstrap M0 task to drive `references/sast/cwe-map-rust.md` from data must do a two-hop fetch (RUSTSEC → aliases[] → GHSA-OSV → cwe_ids), with a category-to-CWE fallback for advisories without a GHSA alias (informational/unmaintained advisories are almost always GHSA-less and CWE-less).

### F2 — Refined Rust CWE frequency (sample, 2024-2026)
Confidence: **MEDIUM** — derived from a 14-advisory spot-check, not a full DB tally.
- **CWE-416 (use-after-free)** appears as one of the highest-frequency classes (RUSTSEC-2024-0017 cassandra-rs, RUSTSEC-2024-0019 mio, RUSTSEC-2026-0090 wasmtime). This was **not in the user's prior top-7** and should be promoted in.
- **CWE-787 / CWE-125 (OOB write/read)** confirmed (RUSTSEC-2026-0080 scaly, RUSTSEC-2024-0400 ruzstd). Consistent with RustXec.
- **CWE-190 (integer overflow)** confirmed (RUSTSEC-2026-0070 hpke-rs).
- **CWE-248 panic-DoS** is **NOT directly tagged** by GHSA — RustSec/GHSA tag the root cause, not the panic symptom. Panic-DoS shows up as `categories=["denial-of-service"]` plus a root-cause CWE (CWE-190, CWE-20, CWE-697). The user's CWE-248 framing remains valid as a *sink-shape* class but does not rank natively from advisory data.
- **CWE-191 / CWE-770 / CWE-1284**: INCONCLUSIVE — not directly hit in the sample. RustXec's CWE-400 prominence is reproducible only via the category crosswalk, not native tags.

### F5 — Repo-state constraints that bound the architecture
- `Cargo.toml` workspace has **no `xtasks/` directory and no `.cargo/config.toml`** — adding `cargo xtask sast-verify` is net-new infrastructure (workspace member + alias + dispatcher).
- The matklad xtask convention requires `[alias] xtask = "run --package xtask --"` in `.cargo/config.toml` for the alias to work from any subdirectory. Source: https://github.com/matklad/cargo-xtask/blob/master/README.md
- The `--workspace` test baseline is parked; a new `xtasks/sast-verify/` crate must be added explicitly to the `cargo test -p ...` line in CLAUDE.md (per the existing constraint).
- All current `crates/` use `clap 4`, `anyhow 1`, `thiserror 2`, `serde 1` — any verifier should reuse these workspace deps, not introduce new ones.

### F6 — Publication-risk default (refined from previous round)
- No web evidence of takedown notices against semgrep-rules-style repos for vulnerable test snippets. Convention is **inline `// ruleid:` annotation in tracked source files**.
- The compliance-fine framing in the idea doc is **not corroborated** by precedent. Default conservative posture (gitignored corpus) is *defensible-but-overcautious*; matching upstream convention (tracked + `// ruleid:` annotation) is *defensible-and-portable*. Pick based on whether portability to `semgrep ci` matters more than legal-team comfort.

## Library Evaluations

### Semgrep CLI (verifier engine)
- **Version**: not pinned in this research; Rust frontend in production use for SCA reachability in 2026.
- **License**: LGPL-2.1 (engine), Semgrep Rules License (rules).
- **Pros**: stable `--validate` and `--test` exit codes; first-party `pre-commit` hook repo; SARIF output; Rust taint mode now usable.
- **Cons**: only **10 Rust rules** in the upstream first-party tree — pack starts thin, rule writers cannot copy from a deep prior-art well; macro/proc-macro parsing is brittle (multiple open issues); `metavariable-type` semantics on Rust generics are under-documented.
- **Fit**: required. Verifier and pack runtime both bind to it.

### `prek` (j178/prek) — Rust-native pre-commit replacement
- **Version**: new as of late 2025/early 2026 (HN signal, no major-version tag).
- **License**: TBD (not surfaced).
- **Pros**: drop-in for `.pre-commit-config.yaml`; no Python bootstrap; matches a `cargo`-shaped repo's tooling story.
- **Cons**: maturity unproven; users without `prek` still get the canonical `pre-commit` path. Adopting `prek` in CI of a generated project requires the user to install it, which `pre-commit` does not.
- **Fit**: **document as alternative, default to `pre-commit`**. The skill should emit a `.pre-commit-config.yaml` that works under either runner.

### `cargo-audit` 0.22.1 + SARIF
- **License**: Apache-2.0 OR MIT.
- **Pros**: confirmed JSON+SARIF output; CWE-via-GHSA path is the same join `/slo-rulegen` already needs.
- **Cons**: emits findings against `Cargo.lock`, not source AST — does not directly drive a Semgrep rule. The tie-in is "CI-time advisory hit → triggers rulegen against the patch diff."
- **Fit**: **CI integration partner**, not engine.

### `xtaskops` / `xtasks` / `tracel-xtask` (helper crates)
- **Versions**: not surfaced reliably.
- **Pros**: pre-built helpers (coverage, ci, fmt) reduce hand-rolling.
- **Cons**: NOT canonical — matklad's spec doesn't bless any helper crate; introduces an unpinned dep into the project verifier; current SLO crates use bare `clap 4` everywhere with no helper crate, so adopting one is inconsistent.
- **Fit**: **do not adopt**. Hand-roll a 200-line `xtask/src/main.rs` matching the OpenVMM single-binary-with-subcommands shape.

### Clippy (`unwrap_used`, `indexing_slicing`, `arithmetic_side_effects`)
- **Status**: stable, default `Allow`, restriction group.
- **Pros**: zero new infrastructure; covers CWE-248/CWE-190 sink shapes broadly.
- **Cons**: no taint, no per-handler context, no CWE tags, fires on every occurrence — noisy without Semgrep layered on top.
- **Fit**: **integrate as coarse fence**, not replacement.

### `pre-commit` (Python-based, canonical)
- **Status**: ecosystem default; first-party Semgrep hook repo at `semgrep/pre-commit`.
- **Pros**: Python install is universal; users' existing pre-commit configs accept the new hook with no rewrite.
- **Cons**: Python runtime in a Rust-only repo introduces a non-Rust dep.
- **Fit**: **default**.

## Architecture Options

### Option α — Skill-only (Approach A revisited)
- **Shape**: `/slo-rulegen` and `/slo-ruleverify` ship as `skills/<name>/SKILL.md` files plus `references/sast/` reference content. No new Rust crate. The skill drives Claude to shell out to `semgrep --validate <rule>` and `semgrep --test <rule-dir>` via Bash, parsing exit codes inline.
- **Pros**: matches the existing `slo-*` skill pack pattern (no new crate); no `xtasks/` infrastructure to introduce; `prek` / `pre-commit` still emit-able as project artifacts; ships in ~1 week.
- **Cons**: verifier gate runs by *prompted Bash sequence* — Claude can skip a step. The fire/silent gate is "policy" not "code." If `semgrep --test` is unavailable on the user's PATH the skill silently degrades. Variation-coverage minimum becomes prompt-policed (the breach-mitigation risk in the idea doc).
- **Best fit**: speed-of-shipping wins; user accepts that the gate is enforced by skill discipline + `semgrep` exit codes alone.

### Option β — Hybrid skill pack + `xtasks/sast-verify/` Cargo crate
- **Shape**: skills do all LLM-heavy work. New `xtasks/sast-verify/` crate added to the workspace plus `.cargo/config.toml` declaring `[alias] xtask = "run --package xtasks/sast-verify --"`. Crate is a single binary with subcommands `validate`, `test`, `check-coverage`. Each subcommand returns a structured exit code. Skills shell to `cargo xtask sast-verify <subcommand>`.
- **Pros**: deterministic gate — `/slo-rulegen` cannot append a rule to the pack until `cargo xtask sast-verify check-coverage --rule <path>` exits 0. Verifier can enforce variation-coverage minimums (e.g. CWE-248 needs ≥7 sink shapes) deterministically. Reuses workspace deps (`clap 4`, `anyhow 1`).
- **Cons**: net-new infrastructure (`.cargo/config.toml`, `xtasks/` dir, alias) the repo has never had; must be added to the explicit `cargo test -p` baseline list in CLAUDE.md per the parked-Tauri constraint; xtask scope-creep risk (idea-doc explicitly flagged this).
- **Best fit**: when the variation-coverage minimum and gate determinism matter more than ship-speed.

### Option γ — Hybrid skill pack + `crates/sldo-sast/` first-party crate
- **Shape**: skills are thin wrappers shelling out to a `sldo-sast` binary, mirroring the `sldo-research` precedent. Crate owns advisory-db ingestion, GHSA join for CWE, snippet-corpus management, verifier loop.
- **Pros**: full deterministic backstop; CWE map can be regenerated from data on a schedule; matches the `sldo-research` shape so it's familiar in this codebase.
- **Cons**: ~3-4 weeks; couples bug-found→rule-extended loop to a `cargo build` cycle. The user's pain explicitly is "fix the regression now, not after a release tag." Idea doc rejected this.
- **Best fit**: only if the project commits to the maintenance tax of an advisory-db pipeline.

### Option δ — Skill-only + `semgrep-test-action` reuse (no xtask, no new crate)
- **Shape**: `/slo-rulegen` writes rules, paired snippets, and a `.github/workflows/semgrep-test.yml` that invokes Semgrep's first-party `--test` runner in CI. Local dev uses `pre-commit` with the `semgrep/pre-commit` hook. The "gate" lives in the upstream Semgrep tooling, not a new component.
- **Pros**: minimum surface area; matches semgrep-rules upstream convention exactly (tracked corpus + `// ruleid:` annotation + `semgrep --test` runner); zero new infrastructure in this repo. Variation-coverage minimum becomes a Bash check the skill runs locally before commit, but the *correctness gate* is canonical Semgrep tooling.
- **Cons**: pushes verification responsibility to the user's CI provider; no in-repo deterministic fail-fast for the variation-coverage minimum.
- **Best fit**: when "match upstream convention" outranks "in-repo gate" — best for portability of the generated pack.
