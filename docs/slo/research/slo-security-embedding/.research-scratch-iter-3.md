## Deepened Findings

### Q1 — Capability manifest granularity (resolved, important caveat)

**High confidence, from primary sources.** The CycloneDX 1.6 **declarations / definitions** surface expresses *catalog- and standard-level* conformance claims (attestations → claims → evidence, signed), not parametric primitive-level claims. A declaration like "component X conforms to ASVS 5.0 V2.4" is schema-native; a claim like "implements Argon2id with ≥3 iterations, ≥64 MiB memory, ≥4 parallelism" is **not** schema-enforced — it has to ride in the free-form `properties` array, a `cryptographic-asset` component, or an `externalReferences` link ([CycloneDX 1.6 JSON reference](https://cyclonedx.org/docs/1.6/json/); [Interlynk — All about CycloneDX 1.6](https://medium.com/@interlynkblog/all-about-cyclonedx-1-6-6af35df675de)).

Implication for SLO: "Hulumi implements Argon2id" is a legitimate CycloneDX declaration; "Hulumi implements Argon2id with these parameters" needs either a CycloneDX Property Taxonomy extension (there is such a public registry) or an external JSON sidecar. OSCAL's `implemented-requirements.statements.by-component.props/remarks` is the closest competing surface, but has the same "controlled vocabulary vs. free prose" cliff.

### Q1 — Rust tooling gap (new, load-bearing)

**High confidence, from `gh api` on the CycloneDX org.** The canonical Rust crate `cyclonedx-bom` is at **0.8.1 (2026-03-19)** and still targets **CycloneDX spec 1.5** ([cyclonedx-rust-cargo releases](https://github.com/CycloneDX/cyclonedx-rust-cargo/releases); [CHANGELOG](https://github.com/CycloneDX/cyclonedx-rust-cargo/blob/main/cyclonedx-bom/CHANGELOG.md)). No upstream support for 1.6 `declarations` / `definitions` exists in the Rust ecosystem today. Any Rust-native emitter in SLO would either (a) upstream 1.6 support to `cyclonedx-bom`, (b) hand-roll the declarations JSON against the 1.6 schema, or (c) shell out to the Node.js/Python toolchain. This materially weakens Option A's claim of "drop-in" Rust fit.

### Q2 — Compliance framework default set (resolved)

**High confidence, multi-source.**
- Commercial consensus baseline (Vanta 15k+ customers; Drata 8k+): **SOC 2, ISO 27001, HIPAA, PCI DSS, GDPR** — the common five across both leaders ([prior round sources still authoritative]).
- Maintainer-survey data stops at *awareness of security practices*, not per-framework compliance adoption: Tidelift 2024 reports OpenSSF Scorecard awareness at **40%**, NIST SSDF at **39%**, SLSA at **23%** — no SOC 2 / ISO 27001 numbers published ([Tidelift state of maintainers](https://sdtimes.com/open-source/the-state-of-open-source-maintainers/)). The earlier round's concern that per-framework OSS-user stats don't exist publicly is confirmed — no further survey will close this gap.
- **Recommended SLO default column set:** SOC 2 + ISO 27001 + GDPR + HIPAA + PCI DSS + NIST 800-53 (moderate baseline) + OWASP ASVS 5.0. Render only SOC 2 + ASVS by default; others selectable by project flag. Medium-confidence inference from vendor consensus + SLO's plausible user distribution.

### Q3 — LLM threat-modelling state of the art (tightened)

**High confidence, from `gh api` on SecOpsTM.** SecOpsTM now has **real tagged releases**: v1.0.0 (2025-08-13), v1.1.0 (2025-08-26) ([ellipse2v/SecOpsTM releases](https://github.com/ellipse2v/SecOpsTM/releases)). That is a much stronger maturity signal than the previous round implied — it's shippable, not a perpetual alpha. pytm at **v1.3.1 (April 2024)** is still stagnant; rely on it through SecOpsTM's wrapper only.

Output-stability caveat from prior rounds still stands: LLM-emitted STRIDE (STRIDE-GPT, SecOpsTM component-level engine) shows inconsistent categorization across runs. Any `/slo-threat-model` that ingests LLM-produced threats must validate against a schema (OTM 0.2.0 is the natural choice — see below) before persisting.

### Q3 — Interchange format (resolved)

**OTM 0.2.0** (JSON/YAML, Apache 2.0, IriusRisk) is the only platform-independent threat-model interchange with a published schema ([iriusrisk/OpenThreatModel/otm_schema.json](https://github.com/iriusrisk/OpenThreatModel/blob/main/otm_schema.json)). Schema has not moved in ~2 years; treat as stable rather than stalled.

### Q4 — Variant analysis (confirmed)

No new findings beyond iteration 2: **Semgrep CE (LGPL core) + ast-grep 0.42.0 (Rust, MIT, SARIF since 0.40.0)** is the pragmatic pair for a <60 s interactive budget. **CodeQL** for nightly only. **Opengrep** is the licensing hedge. This conclusion is stable; further research would be diminishing returns.

### Q5 — gh-CLI filing (resolved with concrete numbers)

**High confidence, GitHub Docs.** Secondary rate limit: **80 content-creating req/min, 500/hour**, returning **403 or 429 with a `retry-after` header**, applied to issues, PRs, and comments across REST + GraphQL + web UI ([GitHub REST rate limits](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api); [GitHub discussion #32120](https://github.com/orgs/community/discussions/32120)). Primary: 5,000/hr user token; 1,000/hr `GITHUB_TOKEN`. Per-endpoint point cost is **not publicly disclosed** — empirical probing is the only way to measure.

Combined with the iteration-2 finding that org-owned repos inconsistently enforce `validations: required: true` on issue forms ([community #43859](https://github.com/orgs/community/discussions/43859)), the operational picture is: an agent filing cross-repo issues needs a **client-side rate guard** (e.g. max-40/hr per session as a defensive half of budget) and **must assume template validation is advisory, not enforcing**.

### Cross-cutting resolution: OSCAL↔CycloneDX crosswalk

**Medium-low confidence.** NIST's control-mapping model page describes OSCAL's *internal* crosswalks between control catalogs, not an OSCAL-to-CycloneDX interchange ([NIST OSCAL Control Mapping](https://pages.nist.gov/OSCAL/learn/concepts/layer/control/mapping/)). NIST CSWP 53 ipd asserts conceptual alignment but publishes no lossless mapping table. Sbomify maintains a **SPDX↔CycloneDX** field crosswalk ([Sbomify schema crosswalk](https://sbomify.com/compliance/schema-crosswalk/)), but not OSCAL↔CycloneDX. An SLO that wants lossless round-trip between the two must author its own crosswalk — and should assume it will be lossy at the `claims.evidence` vs `by-component.statements.remarks` boundary.

## Architecture Options

### Option A — "CycloneDX-native with Rust escape hatch"
**Manifest:** CycloneDX 1.7 BOM + `declarations` authored as hand-rolled JSON (or a thin new Rust crate `sldo-cdx-declarations`) because `cyclonedx-bom` doesn't yet speak 1.6+. Primitive-level claims (e.g. Argon2id parameters) encoded under `properties` with a vendored SLO property-taxonomy namespace, validated against an internal schema at emit time.
**Threat model:** `/slo-threat-model` shells out to **SecOpsTM v1.1.0** in a vendored Python venv; post-processes SecOpsTM output into **OTM 0.2.0** JSON and validates against the OTM schema before writing.
**Variant analysis:** Semgrep CE + ast-grep 0.42 both emit SARIF; results merged into a CycloneDX VDR sidecar.
**Filing:** `gh issue create` with a client-side ≤40/hr guard; SLO-owned intake repo as the fallback channel.
**Trade-offs:** Pros — pragmatic, schema-agnostic, ships this quarter. Cons — introduces a Python dependency (SecOpsTM); property-taxonomy is a local convention until Upstream CycloneDX Property Taxonomy adopts it.
**Best fit:** SLO today.

### Option B — "OSCAL-anchored, compliance-defensible"
**Manifest:** OSCAL Component Definition v1.1.3 per library, referencing NIST 800-53 + ASVS 5.0 + CIS v8. Capability claims live in `implemented-requirements.statements.by-component`; primitives expressed via `props` with an SLO-controlled vocabulary. Round-trip CycloneDX declarations generated as a *derived* view, not the source of truth.
**Threat model:** SecOpsTM → OTM JSON **plus** an OSCAL `assessment-plan` that links threats to the implemented-requirements records.
**Variant analysis:** Semgrep + CodeQL emitting SARIF; CodeQL runs nightly.
**Filing:** Prefer PR-to-fork; `gh issue create` as fallback.
**Trade-offs:** Pros — audit-ready; strongest fit if SLO users will ever chase FedRAMP/SOC 2 evidence collection downstream. Cons — heavier authoring cost per library; no Rust OSCAL crate (pure JSON authoring or Python `compliance-trestle` subprocess); duplicates part of the capability claim in two formats.
**Best fit:** SLO users in regulated or gov-adjacent SaaS.

### Option C — "Rust-pure, schema-emitting core"
**Manifest:** A small Rust crate `sldo-capmanifest` defines an internal `CapabilityClaim` struct and emits **either** CycloneDX 1.6 declarations **or** OSCAL component-definition JSON from the same source. Primitive-level parameters (Argon2id iters, etc.) are first-class fields in the internal struct; lossy when serialized into either wire format, by design.
**Threat model:** Rust service calls Claude directly with a deterministic prompt, validates against an internal schema (superset of OTM 0.2.0), rejects malformed output. No SecOpsTM / pytm dependency.
**Variant analysis:** **ast-grep 0.42** as the primary engine (Rust-native, fast, MIT, SARIF); Semgrep optional.
**Filing:** SLO-owned intake repo is the default; `gh issue create` against third-party repos requires explicit user confirmation.
**Trade-offs:** Pros — no Python, single-language stack, zero upstream rate-limit pressure by default, primitive-level claims are first-class. Cons — reinvents part of SecOpsTM's rule engine; LLM-only threat-model path inherits categorization instability unless the prompt+validator combination is tested; ast-grep has a smaller rule ecosystem than Semgrep.
**Best fit:** If keeping the Rust purity of the SLO toolchain matters more than reusing OWASP's engines.

### Orthogonal decision: default framework columns

Independent of A/B/C: render **SOC 2 + OWASP ASVS 5.0** by default in the threat-model compliance column; make **GDPR, HIPAA, PCI DSS, NIST 800-53 moderate, ISO 27001** opt-in via a `compliance:` front-matter list in the runbook. GDPR should be a *section* in addition to a column, because it encodes principles + data-processing obligations rather than a discrete control catalog — a column alone misrepresents it.

## Unanswered Questions

1. **Does the CycloneDX Property Taxonomy already define primitive-level crypto namespaces** (e.g. `cdx:crypto:argon2id:iterations`)? The Taxonomy is a public registry; if it does, Option A's "vendored namespace" becomes a Taxonomy contribution instead. (Medium priority — cheap to check via the Taxonomy repo.)
2. **Is `compliance-trestle`** (IBM's OSCAL Python toolkit) mature enough to be the SLO subprocess backend for Option B? Not evaluated in any round. (Medium priority — blocks Option B from being a concrete recommendation.)
3. **SecOpsTM stability on non-Linux hosts.** v1.1.0 is real, but macOS + Windows support, ChromaDB install footprint, and Ollama fallback behaviour aren't characterised. (Medium priority — blocks SecOpsTM integration on user machines.)
4. **Empirical GitHub content-creation point cost.** Still undisclosed. An experiment firing N dummy issues against a disposable repo is the only way to know whether one SLO session can safely file ~50 capability-gap issues before throttling. (Low priority for now — a 40/hr client-side cap obviates the need.)
5. **Org-level template validation enforcement.** Confirmed unreliable in community #43859; not confirmed whether `kerberosmansour/hulumi` specifically honours `validations: required: true`. (Low priority — easy to test once Phase 1 ships.)
6. **CycloneDX Property Taxonomy extension process.** If SLO's internal primitive namespace proves useful, the upstream contribution path is undocumented in the material gathered. (Low priority, post-MVP.)
7. **Does any existing tool already emit CycloneDX 1.6 declarations from Rust?** The Node (`@cyclonedx/cyclonedx-npm`) and Python (`cyclonedx-python-lib`) ecosystems are likely ahead; a one-shot check there could unblock a "call out to Node" shortcut. (Low priority — only matters if hand-rolling JSON becomes painful.)
