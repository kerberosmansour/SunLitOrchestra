# Synthesis — slo-security-embedding

This document translates the dossier's external findings into design constraints for `/slo-architect`. Every paragraph ends with the "the design must handle X because Y" shape so downstream skills can cite each constraint individually.

## Threat-model engine must emit into a schema, not a free-form document

OWASP SecOpsTM v1.1.0 (2025-08-26) is the closest agent-composable threat-modelling engine that ships today — it fuses pytm's deterministic STRIDE rule engine with a component-level LLM and a RAG pipeline, and emits mappings to MITRE ATT&CK, CAPEC, CVE, D3FEND, CIS, and NIST 800-53. STRIDE-GPT and ThreatCompute (CCSW 2025) both demonstrate the failure mode that kills the pure-LLM alternative: categorization is unstable across runs, so the same system modelled twice produces different threats. **The design must handle threat-model output validation at emit time because LLM-driven STRIDE is documented as unstable across runs (Pure Storage 2025; FuzzingLabs 2025; ThreatCompute CCSW 2025).** Concretely: `/slo-threat-model` writes OTM 0.2.0 JSON plus a human-readable Markdown companion, and the JSON is validated against IriusRisk's otm_schema.json before either file is written to disk.

## Capability manifest format: CycloneDX 1.7 declarations — but there is no Rust emitter

CycloneDX 1.6 (April 2024) introduced the Definitions / Declarations surface purpose-built for expressing "component X conforms to standard Y"; CycloneDX 1.7 (October 2025) is now codified as ECMA-424 2nd edition (December 2025); OWASP ASVS, MASVS, SCVS, and SAMM already ship in CycloneDX format. The load-bearing constraint is downstream: `cyclonedx-bom` 0.8.1 (2026-03-19) still tracks spec **1.5** and has no 1.6+ `declarations` support, and `cargo-cyclonedx` 0.5.9 emits SBOMs but not declarations. **The design must handle the Rust CycloneDX tooling gap because no Rust crate today emits CycloneDX 1.6+ declarations (cyclonedx-rust-cargo releases page + CHANGELOG).** Concretely: Hulumi and SunLitSecurityLibraries will either hand-roll declarations JSON against the 1.6 schema, shell out to `@cyclonedx/cyclonedx-npm` or `cyclonedx-python-lib`, or upstream 1.6 support to the Rust crate — and the architect must pick one before `/slo-plan` writes milestones against it.

## CycloneDX declarations are lossy at the primitive level — SLO owns the vocabulary

CycloneDX declarations schema-enforce catalog/standard-level claims ("component X conforms to ASVS 5.0 V2.4") but do **not** schema-enforce parametric primitive claims ("Argon2id with ≥3 iterations, ≥64 MiB memory, ≥4 parallelism"). Parametric claims must ride in `properties`, a `cryptographic-asset` component, or `externalReferences`. **The design must handle a vendored SLO property-taxonomy namespace because CycloneDX 1.7 does not schema-enforce primitive-level parametric claims (CycloneDX spec + OWASP CBOM docs).** Concretely: `/slo-architect` defines an internal SLO namespace (e.g., `cdx:sunlit:crypto:argon2id:iterations`) with an emit-time validator; if the public CycloneDX Property Taxonomy later exposes equivalent names, SLO migrates and deprecates its vendored terms.

## Default compliance columns: SOC 2 + OWASP ASVS 5.0, everything else opt-in

The evidence is asymmetric. Commercial-vendor catalogs (Vanta 15k+ customers / 35+ frameworks; Drata 8k+ / 26+) converge on SOC 2, ISO 27001, HIPAA, PCI DSS, and GDPR as the common core for SaaS teams, and SOC 2 ↔ ISO 27001 overlap roughly 70–80%. No survey (Tidelift 2024; Snyk 2024; Stack Overflow; JetBrains) reports compliance-framework adoption for OSS maintainers specifically — this is a structural evidence gap. **The design must handle compliance-framework defaults via a two-tier approach because no OSS-specific adoption data exists (Tidelift 2024; Snyk 2024; all cited surveys absent framework-level numbers).** Concretely: the threat model renders SOC 2 + OWASP ASVS 5.0.0 as default columns; GDPR, HIPAA, PCI DSS 4.0, NIST 800-53 moderate, and ISO 27001 are opt-in via a `compliance:` front-matter list in the runbook; GDPR additionally gets its own section because it encodes principles, not discrete control IDs.

## AI-adjacent runbooks pull in a second framework triad

Whenever an SLO target involves an AI/LLM component, the standard citation triad surfaced repeatedly across 2025–2026 vendor and independent sources: **MITRE ATLAS + OWASP LLM Top 10 + NIST AI RMF**. ATLAS v5.1.0 (November 2025; agentic updates October 2025 and February 2026) now includes `AML.T0058 AI Agent Context Poisoning`, Memory Manipulation, Thread Injection, and Modify AI Agent Configuration techniques specifically for AI agent architectures. **The design must handle AI-specific threat-model framing because the standard triad is uniformly MITRE ATLAS + OWASP LLM Top 10 + NIST AI RMF (Giskard 2026; AWS Security Blog; Practical DevSecOps 2026; Straiker 2025; William Ogou 2025).** Concretely: the runbook's `ai_component: true` flag (set during `/slo-ideate` or `/slo-architect`) triggers the second triad on top of SOC 2 + ASVS.

## Variant analysis: Semgrep CE + ast-grep interactive, CodeQL nightly

Semgrep CE v1.160.0 (2026-04-16) hits ~10 s median scan, ~150 MB RAM, 30+ languages, and emits SARIF natively — inside the <60 s interactive budget. ast-grep 0.42.0 (2026-03-16) is Rust-native MIT with native SARIF output since 0.40.0 (November 2025), pairs for structural rules Semgrep handles less cleanly, and has no cloud tether. CodeQL's 400+ OSS CVE record via variant analysis is the strongest evidence for keeping it in the pipeline — but scans run minutes to 30+ minutes and require a buildable environment, which puts it outside the interactive budget. Trail of Bits shipped `mrva` (2025-12-11) for terminal-first multi-repo CodeQL variant analysis, directly addressing Google PSC's monorepo-assumed framing. **The design must handle a two-tier variant-analysis pipeline because no single tool meets both the interactive (<60 s) and the deep (taint + cross-function) bars (Semgrep vs CodeQL comparisons, Konvu / AppSec Santa / Doyensec 2026; ast-grep CHANGELOG; Trail of Bits blog 2025-12-11).** Concretely: `/slo-security-test` (Phase 3) invokes Semgrep + ast-grep on pre-commit / per-PR paths, and runs CodeQL (or `mrva` for cross-repo) on a nightly/weekly cadence.

## Semgrep's December 2024 relicense reshaped the OSS hedge

Semgrep's December 2024 relicense moved several previously-OSS features behind a commercial license; in January 2025 a ~10-vendor coalition forked Semgrep CE into Opengrep (LGPL-2.1). Core engine and community rules remain OSS, but pro-only rules (taint improvements for variadic functions, cross-file analysis for additional languages) are gated. **The design must handle the Semgrep licensing split because Semgrep's relicensing has already caused an industry fork (AppSec Santa 2026; Semgrep release notes post-Dec 2024).** Concretely: SLO pins Semgrep CE as the default, documents Opengrep as the supported hedge if gating becomes a blocker, and does not depend on any pro-only rule class for its Phase 3 variant-analysis baseline.

## Cross-repo filing: SLO-owned intake repo by default

GitHub's content-creation secondary rate limit (80/min, 500/hr, shared across issues/comments/PRs/commits) is ~3–4 orders of magnitude above "a few capability-gap issues per week," so primary volume is not the binding constraint. The binding constraints are (a) attribution — `gh issue create` authenticates as the local user's identity, and no service-account path exists without a separate GitHub App install; (b) enforcement unreliability — `validations: required: true` on issue-template fields is documented as unreliable in org-owned repos (community discussion #43859; checkbox/dropdown inconsistency #45084), so the REST endpoint bypasses the web form and an agent cannot trust template gates as a guardrail; (c) per-endpoint point cost is deliberately undocumented, so an agent session cannot predict its own throttle headroom. **The design must handle capability-gap filing via an SLO-owned intake repo as the default, with `gh issue create` against third-party repos only on explicit user confirmation, because cross-repo attribution is ambiguous and template validation is unreliable (GitHub Docs rate-limits, issue-form syntax, community discussions #32120 / #43859 / #45084; cli/cli #9380 / #10500).** Concretely: `/slo-sec-libs` (Phase 4) files into `kerberosmansour/slo-security-intake` or equivalent by default, and gates `gh issue create` against `kerberosmansour/hulumi` or `kerberosmansour/SunLitSecurityLibraries` behind an explicit user-provided `--file-upstream` flag with a client-side ≤40 issues/hr cap.

## Architectural recommendation: Option A (CycloneDX-native with Rust escape hatch) with Python SecOpsTM subprocess

The three architecture options in the research (A: CycloneDX-native + SecOpsTM Python subprocess + Semgrep + ast-grep; B: OSCAL-anchored + compliance-trestle; C: Rust-pure + ast-grep primary) trade differently against SLO's constraints. Option B's OSCAL authoring cost and the absence of a mature Rust OSCAL writer punishes the solo-operator wedge. Option C's LLM-only threat-model path inherits the documented categorization-instability risk without a deterministic validation gate in reach. Option A ships fastest, is schema-agnostic enough to migrate into B or C later, and its only hard dependency — SecOpsTM — is a tagged-release OWASP project actively maintained. **The design must handle the "ship working artifacts this quarter" constraint by adopting Option A because SLO is a Rust-first solo-operator toolchain, and both OSCAL depth (B) and Rust-purity (C) materially slip the wedge (dossier Option tables; SecOpsTM releases page).** Concretely: Phase 1's `/slo-architect` update emits `SECURITY.md` + `docs/slo/design/threat-model.md` as Markdown+OTM; Phase 2's `/slo-threat-model` shells out to SecOpsTM; Phase 3's `/slo-security-test` invokes Semgrep + ast-grep; Phase 4's `/slo-sec-libs` consumes CycloneDX declarations and files capability gaps via the SLO-owned intake repo.

## Residual risks carried into /slo-architect

Three findings must be surfaced by `/slo-architect` as explicit residual risks in `docs/slo/design/stack-decision.md`:

- **SecOpsTM cross-platform maturity on macOS arm64 / Windows** is not characterized in any research round, so its viability as a user-machine dependency needs a spike install before `/slo-plan` commits Phase 2 milestones to it (SecOpsTM releases page; no cross-platform data surfaced).
- **CycloneDX Property Taxonomy crypto-primitive namespaces** — whether `cdx:crypto:argon2id:iterations` exists in the public Taxonomy is unresolved; SLO starts with a vendored namespace and migrates if the upstream vocabulary covers it (CycloneDX Spec repo).
- **GitHub content-creation per-endpoint point cost** is deliberately undocumented, so any `/slo-sec-libs` rate-guard is defensive not guaranteed — the architect must pick a cap and document the SLO-intake fallback as the production path (GitHub REST rate-limit docs; octokit/plugin-throttling.js #108).
