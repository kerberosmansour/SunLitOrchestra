## Deepened Findings

### Q2 — Compliance framework default set (newly answered)

**High confidence:**
- **Vanta** (the market leader at **15,000+ customers**) and **Drata** (**8,000+ customers**) both treat the same core framework cluster as first-class: **SOC 2, ISO 27001, HIPAA, PCI DSS, GDPR**. Vanta covers 35+ frameworks total; Drata 26+. The *common* five are the commercial consensus baseline for SaaS/web-service teams ([Vanta vs Drata 2026 — ztekcyber](https://www.ztekcyber.com/resources/vanta-vs-drata-2026-comparison), [Drata blog — Secureframe vs Vanta vs Drata](https://drata.com/blog/secureframe-vs-vanta-vs-drata)).
- **Framework-selection heuristic** widely cited by compliance automation vendors in 2025–2026: SOC 2 first for US SaaS → enterprise; ISO 27001 first for EU/global; HIPAA if PHI is touched; PCI DSS if payments are in scope. SOC 2 + ISO 27001 overlap ~70–80% — economical to pair ([episki comparison](https://episki.com/now/compliance-framework-comparison), [Sprinto top-10 compliance standards](https://sprinto.com/blog/compliance-standards/)).
- Newer frameworks trending into both platforms' 2026 catalogs: **ISO 42001** (AI mgmt), **DORA** (EU financial), **NIS2** (EU critical infra), **FedRAMP** ([Drata frameworks list](https://drata.com/blog/secureframe-vs-vanta-vs-drata)).

**Medium confidence (inference):**
- An OSS skill-pack threat-model file that ships with **SOC 2, ISO 27001, HIPAA, PCI DSS, GDPR, NIST 800-53 moderate, OWASP ASVS 5.0** as selectable columns (not all rendered by default) would cover ≥80% of SLO's plausible users without bloat. The prior-round baseline (NIST 800-53 + IEC 62443 + SOC 2) under-serves EU users (no GDPR) and handwaves ASVS.

**Low confidence / unsourced:**
- No survey I found reports *per-framework adoption percentages among OSS-tool users* specifically — the closest numeric proxy is Snyk 2024's 62% SBOM / 50% pipeline-security adoption, which is about practices, not frameworks ([Snyk 2024 OSS Security blog](https://snyk.io/blog/2024-open-source-security-report-slowing-progress-and-new-challenges-for/)).

### Q5 — `gh` CLI agent-filing ergonomics (newly answered)

**High confidence (GitHub Docs):**
- **Secondary rate limit — content creation: max 80/minute, max 500/hour.** Explicitly covers issues, PRs, comments, across web UI + REST + GraphQL. Some endpoints are tighter and the exact point cost is *not publicly disclosed* ([GitHub REST rate limits](https://docs.github.com/en/rest/using-the-rest-api/rate-limits-for-the-rest-api)).
- **Primary rate limits:** 5,000 req/hr for an authenticated user token; 1,000 req/hr for `GITHUB_TOKEN` (per repo); 15,000/hr for GHEC; 2,000 OAuth token requests/hr ([same source]).
- **No way to bypass or raise secondary limits** — only wait for reset; continued requests risk banning the integration.
- **Issue forms (`.github/ISSUE_TEMPLATE/*.yml`)** support `validations: required: true`. BUT required-field enforcement is known to be **unreliable in organization-owned repos** per GitHub community discussion #43859, and checkbox/dropdown `required` validation is inconsistent (discussion #45084). An agent cannot rely on template enforcement as a guardrail — it must produce a fully valid body itself ([Syntax for issue forms](https://docs.github.com/en/communities/using-templates-to-encourage-useful-issues-and-pull-requests/syntax-for-issue-forms), [community #43859](https://github.com/orgs/community/discussions/43859)).
- **Attribution:** `gh issue create` is authored by the authenticated user's identity (there is no impersonation flag). For `kerberosmansour/hulumi` to accept a `/slo-sec-libs`-filed issue, the user's normal GitHub identity will be the author — which is fine for a single-operator repo but becomes indistinguishable noise if many SLO users start filing against the same upstream. Unsourced inference.

**Medium confidence (inferred from gh issue history, not explicit docs):**
- Cross-repo issue creation uses the same content-creation pool as in-repo creation; there is **no per-target-repo separate budget.** A single agent-driven session could plausibly burn through the 500/hr budget filing capability-gap issues across many libraries.

### Cross-cutting clarifications

- **pytm's two-year release gap** degrades its candidacy as a live backend for `/slo-threat-model`. OWASP **SecOpsTM** (which wraps pytm + LLM + RAG) is the more actively curated successor but still carries pytm's weaknesses (no direct NIST 800-53 control mappings) ([OWASP SecOpsTM](https://owasp.org/www-project-secopstm/)).
- **OSCAL Component Definition** is unambiguously the most mature "component → controls" manifest format; it ships reference XML/JSON/YAML examples and is cross-walked with SPDX and CycloneDX per NIST CSWP 53 ipd. Confidence: high.
- **CycloneDX Definitions (1.6+)** is the lighter-weight alternative: if the goal is "Hulumi/SunLitSecureLibraries components declare which ASVS 5.0 controls they implement," a CycloneDX BOM with embedded Definitions is the shortest path because an SBOM is likely already produced. OSCAL is richer but heavier. Medium-high confidence.

## Library Evaluations

### Capability-manifest formats

**OSCAL Component Definition v1.1.3 (NIST)**
- License: Public domain (NIST work). Maintenance: industry-owned via OSCAL Foundation (formed Jan 2025). JSON/XML/YAML.
- Pros: Purpose-built for "component implements these controls." First-class control-catalog linking (NIST 800-53, CIS, custom). Reference examples published. Explicit alignment with CycloneDX + SPDX.
- Cons: Verbose; each component-definition file carries heavy schema overhead. Limited tooling for non-US-gov projects. No mainstream Rust crate.
- Fit: Strong if the goal is defensible, auditable control provenance against NIST catalogs. Overkill for purely OSS capability advertising.

**CycloneDX 1.7 + Definitions (OWASP / ECMA-424)**
- License: Apache 2.0 / ECMA. Maintenance: very active.
- Pros: If Hulumi already emits an SBOM, embedding *Definitions* referencing ASVS/SCVS/SAMM is a drop-in. Rust tooling exists (`cyclonedx-rust-cargo`). Composes with VEX/VDR for the capability-gap flow.
- Cons: Definitions surface is newer and less well-documented than component-definition OSCAL. No native "control → test" linkage.
- Fit: Pragmatic default for SLO — lowest lift to add, agent-parsable, OSS-friendly.

**OTM 0.2.0 (IriusRisk)**
- License: Apache 2.0. Maintenance: schema effectively frozen at 0.2.0; adoption is niche.
- Pros: JSON/YAML; Go lib exists (`threatcl/go-otm`); StartLeft converter ecosystem.
- Cons: Primarily describes a threat model, not a component capability manifest. Schema stagnant.
- Fit: Consider as the **threat-model interchange** emitted by `/slo-threat-model`, not as the capability manifest.

**TMBOM (emerging CycloneDX x pytm x Threat Dragon)**
- Status: in development; not yet a stable spec.
- Fit: Watch, don't bet on it.

### Threat-modelling engines

**OWASP SecOpsTM**
- License: OSS via OWASP. Maintenance: active.
- Pros: Already fuses pytm rule engine + component-level LLM + RAG; produces STRIDE with CAPEC/CVE/D3FEND/CIS/NIST 800-53 mappings; supports local Ollama and commercial LLMs.
- Cons: Python stack (SLO is Rust-first). Python subprocess overhead. Tagged versions not surfaced.
- Fit: Closest composable precedent for `/slo-threat-model`. Consider wrapping, not replacing.

**pytm v1.3.1 (OWASP)**
- License: MIT. Maintenance: stalled — no release in ~24 months.
- Pros: Canonical Pythonic threat-model-as-code; stable schema.
- Cons: Stale; no native NIST 800-53 mappings; SecOpsTM supersedes the output stack.
- Fit: Use only via SecOpsTM wrapper.

**STRIDE-GPT (mrwadams)**
- License: MIT. Maintenance: active; recent OWASP LLM Top 10 additions.
- Pros: Known working pattern for LLM-emitted STRIDE + attack trees.
- Cons: Output stability issues noted in 2025 comparisons (inconsistent categorization); no formal schema.
- Fit: Reference for prompting patterns; not an integration target.

**OWASP Threat Dragon**
- License: Apache 2.0. Maintenance: active OWASP Lab.
- Pros: DFD editor, rule engine for STRIDE/LINDDUN/CIA/DIE/PLOT4ai; LLM extensions exist (threat-dragon-ai-tool, threat-dragon-llm).
- Cons: GUI-first; agent composition requires using the underlying JSON directly.
- Fit: Input/output format worth adopting (its JSON is the de facto DFD input for several LLM tools).

### Variant-analysis tooling

**Semgrep CE (LGPL core; commercial rules tier)**
- License: LGPL-2.1 core; Semgrep Pro is proprietary. Maintenance: very active.
- Pros: ~10s scans/~150MB RAM; native `--sarif`; 30+ languages; strong polyglot/IaC.
- Cons: Dec 2024 relicensing moved features behind commercial tier; community forked to **Opengrep** (Jan 2025).
- Fit: Strong default for `/slo-security-test` given <60s budget.

**Opengrep (community fork, LGPL-2.1)**
- License: LGPL-2.1. Maintenance: ~10-vendor coalition since Jan 2025.
- Pros: Community-backed; avoids the Dec 2024 commercial gating.
- Cons: Newer fork; rule ecosystem still tracking Semgrep CE.
- Fit: Hedge candidate if licensing or feature gating becomes a concern.

**CodeQL**
- License: proprietary; free for OSS via GitHub Advanced Security.
- Pros: Deeper data-flow/taint; 400+ OSS CVEs via variant analysis.
- Cons: Minutes to 30+ min scans / ~450MB DB — exceeds <60s budget; free tier constrained to public OSS repos.
- Fit: Secondary — nightly/scheduled, not interactive.

**ast-grep 0.42.0 (Rust, MIT)**
- License: MIT. Maintenance: very active (monthly releases).
- Pros: Rust-native (aligns with SLO toolchain); tree-sitter based; **SARIF output since 0.40.0 (Nov 2025)**; fast.
- Cons: Less semantic than Semgrep (no taint/equivalences); rule ecosystem much smaller.
- Fit: Excellent lightweight Rust-side grepper; pair with Semgrep rather than replace it.

### Agent-filing channels (cross-repo)

**`gh issue create` (direct)**
- Auth: user's GitHub token (`repo` scope or issues write). Limits: 500 content-creating req/hr, 80/min (secondary).
- Pros: Low friction; no fork overhead.
- Cons: Authorship ambiguity; limited template enforcement; rate-limit exhaustion if many gaps filed at once.

**`gh pr create` (against fork with capability added)**
- Auth: same token; also needs write to the fork.
- Pros: Delivers a concrete contribution, not just a request — much higher upstream acceptance rate.
- Cons: Requires the agent to actually implement the capability — much larger scope than issue-filing.

**External intake (RFC repo, email, Slack webhook)**
- Auth: varies; typically a PAT or a webhook secret for a separate intake repo owned by SLO itself.
- Pros: Decouples cross-repo load; SLO controls the triage queue; no upstream rate-limit pressure.
- Cons: Extra hop; upstream maintainers must opt-in.

## Architecture Options

### Option A — "CycloneDX-native, SecOpsTM-adjacent"
Capability manifest: **CycloneDX 1.7 BOM with Definitions** referencing ASVS 5.0 controls, emitted by Hulumi/SunLitSecureLibraries. Threat-model emission: `/slo-threat-model` shells out to a vendored **SecOpsTM** Python process (or reimplements its rule core in Rust later). Interchange: **OTM 0.2.0** JSON for the generated threat model. Variant analysis: **Semgrep CE** (+ **ast-grep** as Rust-side pre-filter) writing SARIF. Agent filing: `gh issue create` with a strict issue-form template, plus a per-session rate guard (≤40 issues/hr to stay well under the 500/hr cap).

Trade-offs:
- Pros: Lowest lift; every component is agent-parsable; OSS-native.
- Cons: Python dependency for the threat-model engine; CycloneDX Definitions surface is newer than OSCAL.
- Best fit: SLO today — Rust-first, solo-operator, wants working artifacts this quarter over a fully federated future.

### Option B — "OSCAL-anchored, compliance-first"
Capability manifest: **OSCAL Component Definition v1.1.3** per library, referencing NIST 800-53 + ASVS 5.0 + CIS v8 catalogs. Threat model: `/slo-threat-model` emits **OTM 0.2.0** JSON + an OSCAL **assessment-plan** linking findings to controls. Variant analysis: **Semgrep + CodeQL** both emit SARIF, merged into a CycloneDX VDR. Agent filing: prefers **PR-to-fork** when the capability is small and implementable; falls back to `gh issue create` only when the capability requires upstream design.

Trade-offs:
- Pros: Audit-defensible; lines up cleanly with FedRAMP/SOC 2/ISO 27001 evidence collection downstream.
- Cons: Much heavier schema overhead; CodeQL blows the <60s interactive budget; higher initial authoring cost for each SunLitSecureLibrary component.
- Best fit: Teams that already need compliance evidence (regulated SaaS, gov-adjacent OSS) and are willing to pay the upfront cost.

### Option C — "Thin Rust-native core, schema-agnostic"
Capability manifest: a small **Rust crate** (`sldo-capmanifest`) that emits either CycloneDX Definitions or OSCAL component-definition JSON from a single internal struct. Threat model: `/slo-threat-model` uses Claude directly with a deterministic prompt + a Rust validator that rejects output not matching an internal schema (superset of OTM 0.2.0). Variant analysis: **ast-grep** (Rust, MIT) as primary; Semgrep optional. Agent filing: an SLO-owned **intake repo** is the primary channel; `gh issue create` against third-party repos is a fallback guarded by an explicit user prompt + rate-limit check.

Trade-offs:
- Pros: Single-language stack (matches SLO conventions); no Python dependency; rate-limit risk to upstream repos is minimized by default; capability-manifest emission flexibility via one struct.
- Cons: Reinvents pieces of SecOpsTM's rule engine; the LLM-only threat-model path has known output-stability issues (Pure Storage, FuzzingLabs 2025 analyses).
- Best fit: If SLO wants to stay Rust-pure and avoid pulling Python into the skill-pack runtime at any cost.

## Unanswered Questions

1. **Per-framework adoption among OSS-tool users specifically.** None of the surveys I found (Snyk, Stack Overflow, JetBrains) break out compliance-framework adoption among OSS maintainers vs. enterprises. Without this, Q2's "default set" recommendation rests on vendor-catalog inference, not direct evidence. Worth one more targeted search into the OpenSSF / Linux Foundation / Tidelift 2025 maintainer surveys.
2. **CycloneDX Definitions expressivity for "Argon2id password hashing" level granularity.** The 1.6 release notes describe Definitions at catalog/standard level; whether it cleanly expresses a concrete primitive-level claim ("component X implements PBKDF2 with min iters Y") needs a schema walkthrough, not a blog.
3. **OSCAL ↔ CycloneDX crosswalk maturity.** NIST CSWP 53 ipd asserts alignment but doesn't publish a concrete mapping table. Is there a lossless round-trip from a CycloneDX Definitions block to an OSCAL `implemented-requirements` block? Unknown from current evidence.
4. **SecOpsTM tagged release / semver.** Two search rounds have not surfaced a pinned version. Needed before making it a dependency.
5. **GitHub content-creation limit point-costs.** GitHub explicitly withholds the per-endpoint point cost; an empirical test (file N issues and observe 429 behavior) is the only way to know how many capability-gap issues a single SLO session can safely file.
6. **Enforced issue-template validation in org-owned repos.** Community discussion #43859 suggests `required: true` is not honored in some org configurations — needs confirmation against the specific repos SLO will file into (e.g. `kerberosmansour/hulumi`).
7. **Rust-native OSCAL or CycloneDX Definitions writer.** Whether a mature Rust crate exists for emitting Definitions/component-definition JSON (vs. hand-rolling). Not evaluated in either search round.
8. **GDPR / NIS2 column semantics for a threat-model table.** Unlike SOC 2 or ASVS (control-IDs), GDPR maps to *principles* and data-processing-record obligations — whether it should be a column at all vs. a distinct section needs a design call.
