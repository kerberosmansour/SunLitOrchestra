## System description

~~~text
The skill pack adds two Claude Code skills (/slo-rulegen and /slo-ruleverify) plus one Cargo workspace crate (xtasks/sast-verify) plus a sibling references/sast/ scaffolding directory to the existing SunLitOrchestrate repo. The skills drive Claude Code (subprocess `claude`) through generation of Semgrep rules + paired vulnerable/clean code fixtures for Rust. The verifier crate wraps `semgrep --validate` and `semgrep --test` plus a YAML-parse-and-count check for variation coverage and a known-clean-subset false-positive check. Generation happens in two modes: bootstrap (the top-10 Rust CWE map seeds an empty pack) and extend (a Claude-found bug + fix-diff produces 3-5 variation rules). The pack is consumed by Semgrep CI on the user's project, which can be the SLO repo itself or any third-party Rust app; consumer-side wiring lands in M3 via `cargo-audit` JSON/SARIF as the trigger plus `pre-commit` (or its Rust drop-in `prek`) for local hooks.
~~~

## Top risks (from idea doc)

~~~text
- Breach: Generated rules cover the exact pattern of the original bug but not its credible variations. An attacker reads the rule diff in the public repo, reverse-engineers what classes have been fixed, crafts a variation the rule misses. Concrete scenario: original bug is .unwrap() on serde_json::from_str(http_body) in an axum handler; the auto-generated rule catches that exact sink shape but misses .expect() on serde_json::from_value() three handlers down, or body.get("field").unwrap() after a successful parse. Attacker ships a malformed payload that hits the unflagged sink, the service panics, public endpoint goes down.

- Compliance fine: The skill writes deliberately vulnerable code into the repo (the snippet/corpus step). If those snippets default to tests/corpus/ or examples/vulnerable/ in a public GitHub repo, the project has now published working exploit code for the CWE classes its own pack flags. For SOC 2 / ISO 27001 audits, "we maintain a public, version-controlled corpus of working vulnerability patterns" is a finding. For crypto-adjacent snippets, US EAR 5D002 export-control review is theoretically applicable to "intrusion software" though enforcement on snippet collections is rare. Real near-term risk is GitHub Acceptable Use Policy on "active malware or exploits".

- Prolonged outage / silent-wrongness: Generated rules ship to CI with YAML syntax errors or over-greedy patterns. Every PR's Semgrep run either fails outright or returns 100s of false positives across files unrelated to the original bug. Devs do the predictable thing: add # nosemgrep to the noisy lines, or comment out the Semgrep step in CI ("temporarily disabled while we debug"), or set the rule severity to INFO so it stops blocking merges. The pack is now decorative; the bug class returns silently next sprint; the milestone-slip pain repeats with worse signal-to-noise than before.
~~~

| Surface | Attacker | Attack step | Desired outcome (attacker) | Control | Threat-model row id |
|---|---|---|---|---|---|
