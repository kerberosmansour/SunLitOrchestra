# Stack Decision — sast-rulegen-skill-pack

## Chosen stack

- **Language / runtime:** Rust 2021 (workspace member, no MSRV pin — matches existing crates).
- **Skill format:** Markdown `SKILL.md` files under `skills/slo-rulegen/` and `skills/slo-ruleverify/`, installed as symlinks into `~/.claude/skills/` by the existing `sldo-install` binary.
- **Verifier crate:** new workspace member at `xtasks/sast-verify/` with a single binary (`sast-verify`) using subcommands (`validate`, `test`, `check-coverage`, `check-clean`, `gate`). Cargo alias `xtask` declared in a new repo-root `.cargo/config.toml` per matklad/cargo-xtask convention: `[alias] xtask = "run --package sast-verify --"`.
- **Dependencies (verifier crate, all already pinned at workspace level):** `clap 4` (derive), `anyhow 1`, `thiserror 2`, `serde 1`, `serde_yaml` (NEW — to be added at workspace level if not present; otherwise inline in the verifier's `Cargo.toml`), `colored 2`, `chrono 0.4`. No new HTTP client; the verifier never reaches the network.
- **External CLIs invoked by the verifier:** `semgrep` (via `std::process::Command` with explicit args, `Stdio::piped()` and drain-on-separate-thread per the existing `sldo-common::copilot` pattern). Version detection via `semgrep --version`; minimum version pinned in `references/sast/MIN-SEMGREP-VERSION.md` after the M1 smoke-test.
- **Shared scaffolding:** new `references/sast/` directory at repo root (sibling of `references/biz/`), holding `cwe-map-rust.md`, `variations/cwe-XXX.md` (one per top-10 CWE), `semgrep-rust-syntax.md`, `manifest-schema.md`, `AUTHORING.md` (re-authoring policy citing Trail of Bits AGPL constraint). Same installer-bypass pattern as `references/biz/` — `sldo-install`'s `discover_skills()` walker requires `skills/<name>/SKILL.md`, so a sibling `references/sast/` has zero installer interaction by construction.
- **Test command baseline addendum:** `cargo test -p sldo-common -p sldo-plan -p sldo-run -p sldo-research -p sldo-install -p sast-verify`. The new `-p sast-verify` flag MUST be appended to the explicit list in [CLAUDE.md](../../CLAUDE.md) at M1; `--workspace` remains unusable until `sldo-tauri` is unparked.
- **Skill toolflags:** new entries in `sldo-common::toolflags` — `rulegen_allow_flags()` / `rulegen_deny_flags()` and `ruleverify_allow_flags()` / `ruleverify_deny_flags()`. Both DENY `WebFetch` and `WebSearch` (rule-gen does not need external fetching; the CWE map is pre-baked in `references/sast/`). Both ALLOW `Bash` (for the `cargo xtask sast-verify` shell-out). Both ALLOW `Read` / `Write` / `Edit` scoped to `.semgrep/`, `references/sast/`, and the skill's own files.

## Reason

The synthesis file is unambiguous on three load-bearing constraints:

1. **"The corpus convention is settled"** (synthesis.md) — paired `<rule-id>.yaml` + `<rule-id>.rs` co-located with `// ruleid:` / `// ok:` annotations, run by canonical `semgrep --validate` + `semgrep --test`. **The verifier therefore wraps existing tooling, not re-implements it.** The xtask's body is small (~50–150 LOC) but its existence as the single deterministic entry point enforces the variation-coverage minimum that the idea-doc breach mitigation depends on. Markdown-only skills cannot enforce that minimum because prompt-policed Bash sequences are skippable.
2. **"Compose with Clippy, do not replace"** (synthesis.md) — the verifier consumes Clippy as a coarse-fence findings stream and Semgrep as the contextual layer. Re-implementing Clippy's sink detection in Rust would duplicate work already done in rustc's lint pipeline.
3. **"Re-author rules independently of Trail of Bits"** (synthesis.md) — Trail of Bits' `semgrep-rules` is AGPL-3.0; the wedge's intended distribution license is unspecified but explicitly NOT AGPL. The skill copies *structural shapes* (variation enumeration via `pattern-either`, `pattern-not-inside #[cfg(test)] mod tests` exclusions) from precedent into `references/sast/AUTHORING.md` as written guidance, never copies rule text.

The Rust workspace + Markdown skill pack is the existing repo shape; the synthesis surfaces no incompatibility. Adding a new workspace member (the xtask) is the cheapest deterministic-gate point per matklad's xtask precedent, used by Cargo, rust-analyzer, Tokio, ripgrep, clap, wasmtime, bevy, OpenVMM. Single-binary-with-subcommands is the dominant convention.

## Rejected alternatives

- **Approach α — pure prompt skill pack, no verifier crate.** Rejected because the variation-coverage minimum (idea-doc breach mitigation) cannot be enforced by skill discipline alone. Markdown skills are prompts; prompts are skippable; the minimum was load-bearing.
- **Approach β-shell — 50-line shell script in `xtasks/sast-verify.sh`.** Rejected on the user's explicit course-correction (2026-04-25 "xtasks/ Cargo task") and for repo consistency (existing CLIs are all Cargo crates, not shell scripts; new code should match).
- **Approach γ — first-party Rust crate `crates/sldo-sast/` parallel to `sldo-research`.** Rejected explicitly in the idea-doc and re-confirmed in synthesis: most of the actual work is LLM-judgment (writing credible Rust variations of a CWE class, mapping a bug-fix-diff to the variation-enumeration template). Rust adds rigidity to deterministic parts but does not help the squishy parts; CWE-taxonomy ingestion would become its own ongoing tax (RustSec advisories add up daily). Coupling the bug-found → rule-extended loop to a `cargo build` cycle directly opposes the user's "fix the regression now, not after a release tag" pain.
- **Approach δ — skill-only + upstream Semgrep `--test` runner in CI, no verifier crate.** Synthesis recommends this as v1 default, but the user's 2026-04-25 course-correction committed to `xtasks/` Cargo task for repo consistency and to enforce variation-coverage as a hard gate. The xtask shape adopts δ's *content* (canonical `semgrep --validate` + `semgrep --test`) inside β's *form* (Cargo crate with deterministic exit codes). Reserve δ as a fall-back if the xtask scope-creeps past 200 LOC.
- **TypeScript-first wedge instead of Rust-first.** Rejected because the user's pain anchor is Rust panic-DoS and webapp injection in Rust apps; shifting to TypeScript would lose the proof point. TypeScript is Runbook B, not the wedge.

## Non-negotiables (downstream cannot change these without migration)

These are the interfaces and conventions M1-M3 must preserve. Changing any of them in a later milestone requires explicit `/slo-architect` re-run, not a quiet edit.

1. **Cargo workspace shape.** New crate at `xtasks/sast-verify/` registered as a workspace member in repo-root `Cargo.toml`. Cargo alias `xtask = "run --package sast-verify --"` in `.cargo/config.toml`. Single binary; subcommands via `clap 4` derive.
2. **Verifier subcommand contract** (per [interfaces](sast-rulegen-skill-pack-interfaces.md)): `validate`, `test`, `check-coverage`, `check-clean`, `gate` — each with a stable exit-code shape. The `gate` subcommand is the entry point `/slo-rulegen` and `/slo-ruleverify` shell out to; its passing is the contract for rule-write authorisation.
3. **Rule pack on-disk layout.** Generated rules land at `<base>/<lang>/<rule-id>.yaml` paired with `<base>/<lang>/<rule-id>.rs` (Semgrep upstream convention; portable to any consumer's `semgrep ci`). The `<base>` defaults to `.semgrep/` for v1, with `--target-dir` flag override for the rule-pack-repo case.
4. **Skill toolflag denials.** `rulegen_*` and `ruleverify_*` flags MUST DENY `WebFetch` and `WebSearch`. The CWE map is pre-baked in `references/sast/cwe-map-rust.md`; the skills do not need network access. Removing this denial requires re-running the threat model.
5. **Two-tier corpus convention.** Snippets generated *inside the rule-pack repo* (this repo or a future SAST.GEN-shaped repo) are tracked-and-labelled per upstream Semgrep convention. Snippets generated *inside a user's application repo* by extend-mode are written to `.gitignore`'d paths by default. The skill detects which tier applies via repo metadata heuristics defined in `references/sast/manifest-schema.md`.
6. **CWE-755, not CWE-248**, as the canonical tag for the panic-DoS class. Re-tagging requires user-approved evidence change; not a per-milestone edit.
7. **`semgrep --validate` runs before `semgrep --test`** in any verifier sequence (Semgrep issue #10319 — `--test` returns 0 on invalid rule itself). The xtask `gate` subcommand encodes this order.
8. **Variation enumeration is a single-rule `pattern-either` with N arms paired with a single fixture file with N `// ruleid:` annotations** (Trail of Bits precedent). N is per-CWE in `references/sast/variations/cwe-XXX.md`. Splitting one logical rule into N files is forbidden by the manifest schema.
9. **Re-authoring policy.** No rule YAML is copied from Trail of Bits' AGPL repo; structural shapes are documented in `references/sast/AUTHORING.md` as written guidance and rules are authored fresh.
