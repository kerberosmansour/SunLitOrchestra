# SAST rule-generation skill pack for Claude Code

## The pain

Sherif was vibe-coding a Rust app a few days ago and watched it crash unexpectedly mid-run, repeatedly, across a session. The crashes were the class of bug Rust's "safe by default" claim is supposed to make impossible — `.unwrap()` on a `None`, out-of-bounds slice indexing, plus webapp-shaped issues like missing input sanitisation and missing outbound output-encoding for XSS — but Rust's definition of "safe" is "no memory corruption / no RCE," not "no crash," and the difference cost him hours per incident. Even the AI pair-programming with him could not recognise the patterns, because Semgrep's default Rust ruleset is tuned for memory-safety/`unsafe` review and does not flag panic-on-attacker-input as a category. He landed each fix hastily, was not proud of any of them, knew the fixes did not generalise to the class, and had a milestone slip in the same week. He pays for this gap three ways today: (1) every recurrence of the same class is debugged from zero because there is no rule that fired the second the variation appeared; (2) he fixes the one site he found, not the seven other shapes of the same sink — the next variation crashes him again next week; (3) when Claude Code itself finds a critical bug during `/slo-execute` or `/slo-verify`, there is no skill that takes the bug + the fix and compounds them into a *rule pack that prevents the regression and its variations from ever landing again*. Today the regression-prevention loop is "I will remember to grep for this," which is the loop that the milestone slip already proved does not work.

## Five capabilities the user described without realising

- **Bug-found → rule-pack-extended is the recurring loop, not bootstrap-pack-from-scratch.** The trigger is a Claude-found critical bug during normal `/slo-execute` or `/slo-verify` work, not a scheduled audit. The output is *appended* to the project's existing Semgrep pack, not a fresh pack regenerated per run. The wedge is bootstrap-the-pack-from-the-top-10-CWEs (one-time, for empty repos), but the long-term value is the per-bug expansion loop — without it, the pack is decorative.
- **Variation enumeration is a first-class step, not a side-effect of LLM creativity.** Catching the exact pattern of the original bug is not enough; for `.unwrap()` on `serde_json::from_str(body)` you also need rules covering `.expect()`, `.unwrap_or_else(panic)`, `[]` indexing on the parsed value, `slice::get(_).unwrap()`, `?` propagation into a context that does not handle the error, and FFI re-entries that re-introduce the same shape. The skill writes a rule per variation class, driven by a per-CWE variation template baked into `references/sast/`, not improvised per bug.
- **Corpus-driven verification is a hard gate on rule output, not advisory.** Every rule the skill writes ships with a paired (bad_snippet, good_snippet) pair. Before the rule lands in the project's `.semgrep.yml`, the verifier proves the rule fires on the bad snippet AND is silent on the good snippet. If verification fails, the rule is rejected, not warned-about. This is why `/slo-ruleverify` is priority #2 and not #5 — without the gate, the skill ships syntax-broken or false-positive-noisy rules that get muted in CI and the regression returns silently.
- **CI and dev test-env are part of the same rollout, not a follow-on skill.** The user named both explicitly. If the rule only fires when CI fails on a PR, the dev fixes the symptom and pushes again; if the rule fires locally as they type (pre-commit hook, IDE-side Semgrep, on-save), they internalise the class. The skill must wire both endpoints in one go, or it has only delivered half the value.
- **Two distinct generation modes — bootstrap vs extend — share scaffolding but ship separately.** Bootstrap mode (Rust + Semgrep + top-10 CWEs from RustSec advisory-db + OSV.dev) seeds an empty pack from a known taxonomy; extend mode (1 bug → 3-5 variation rules) compounds the pack per Claude-found incident. Both modes consume the same `references/sast/` scaffolding (CWE map, variation templates, Semgrep rule-syntax cheat-sheet, snippet corpus schema). The wedge ships bootstrap; extend is M2 of the runbook with a clear input contract.

## Top risks

- **Breach**: Generated rules cover the exact pattern of the original bug but not its credible variations. An attacker reads the rule diff in the public repo (Semgrep packs live at `.semgrep.yml` or `.semgrep/`, in-tree and visible in commit history), reverse-engineers Sherif's mental model of "what classes have been fixed," and crafts a variation the rule misses. Concrete scenario: original bug is `.unwrap()` on `serde_json::from_str(http_body)` in an axum handler; the auto-generated rule catches that exact sink shape but misses `.expect()` on `serde_json::from_value()` three handlers down, or `body.get("field").unwrap()` after a successful parse. Attacker ships a malformed payload that hits the unflagged sink, the service panics, public endpoint goes down. Mitigation: variation enumeration is mandatory and templated per CWE, with a minimum-coverage assertion (e.g. CWE-248 panic-DoS rules must cover at minimum the 7 sink shapes documented in `references/sast/variations/cwe-248.md`); the skill refuses to write a rule for a CWE class without first generating ≥ N variation rules and proving each fires on its paired bad snippet.

- **Compliance fine**: The skill writes deliberately vulnerable code into the repo (the snippet/corpus step). If those snippets default to `tests/corpus/` or `examples/vulnerable/` in a public GitHub repo, the project has now published working exploit code for the CWE classes its own pack flags. For SOC 2 / ISO 27001 audits, "we maintain a public, version-controlled corpus of working vulnerability patterns" is a finding (control deficiency around secure coding / training-material handling); for crypto-adjacent snippets (panic on a misformed signature, integer-overflow in a length-mac calc), US EAR 5D002 export-control review is theoretically applicable to "intrusion software" though enforcement on snippet collections is rare. Real near-term risk is GitHub Acceptable Use Policy on "active malware or exploits" — semgrep-rules itself ships labelled snippets, but the precedent on whether tagging is sufficient is informal. Mitigation: snippet output dir defaults to a path that is `.gitignore`'d by default; the skill warns at write-time if the parent dir is git-tracked AND `git remote get-url origin` returns a public URL; every snippet file has a header comment marking it test-only and citing the CWE; an opt-in flag is required to commit the corpus to a public branch.

- **Prolonged outage / silent-wrongness**: Generated rules ship to CI with YAML syntax errors or over-greedy patterns. Every PR's Semgrep run either fails outright or returns 100s of false positives across files unrelated to the original bug. Devs do the predictable thing: add `# nosemgrep` to the noisy lines, or comment out the Semgrep step in CI ("temporarily disabled while we debug"), or set the rule severity to INFO so it stops blocking merges. The pack is now decorative; the bug class returns silently next sprint; the milestone-slip pain repeats with worse signal-to-noise than before. Detection lag: weeks until the next regression. Mitigation: `/slo-ruleverify` runs as a hard gate on `/slo-rulegen` output before any rule is written to the project pack — no rule lands without (a) `semgrep --validate` passing on its YAML, (b) the rule firing on its paired bad snippet, (c) the rule remaining silent on its paired good snippet, and (d) the rule producing zero false positives on a designated "known-clean" subset of the host repo (default: the crate's own `src/` after the bug's fix is applied). If any of these fail, the rule is rejected with a structured reason for the user to course-correct, not auto-merged with a warning.

## Approach A — pure prompt skill pack (conservative)

- **Effort**: ~1 week for `/slo-rulegen` v1 + ~0.5 week for `/slo-ruleverify` v1.
- **Wedge**: `/slo-rulegen` ships as `SKILL.md` files only — no new Rust crate, no backend. Skills drive Claude to read a pre-baked `references/sast/cwe-map-rust.md`, generate 3-5 variation snippets per CWE class, write a Semgrep rule per variation, shell out to `semgrep --validate` and `semgrep <rule> <snippet>` via Bash to check syntax + fire/silent behaviour, write the rule pack to `.semgrep/rust/`. `/slo-ruleverify` is a thin SKILL.md that re-runs the same Bash checks against an existing pack on demand.
- **Risks**: rule quality is purely LLM-judgment with no deterministic backstop beyond `semgrep --validate`; non-determinism across runs (re-running `/slo-rulegen` on the same bug yields a different rule set, making PR-review of the rule pack itself frustrating); CWE-map drift if the map is re-derived per run instead of cached as a checked-in reference; no shared scaffolding so the second language (TypeScript) re-implements every primitive — variation templates, corpus layout, manifest schema; the variation-coverage minimum (the breach mitigation above) becomes prompt-policed rather than gate-enforced, which is the same shape as the prolonged-outage risk.

## Approach B — skill pack + Rust backend crate `crates/sldo-sast/` (cloud / SaaS)

- **Effort**: ~3-4 weeks. New crate is the bulk of the work.
- **Wedge**: a new crate under `crates/sldo-sast/` mirrors the `sldo-research` precedent. Crate owns CWE-taxonomy ingestion (RustSec advisory-db git pull + parse, OSV.dev API client, MITRE CWE database mirror), snippet corpus management with deterministic IDs and per-CWE templates, Semgrep rule syntax validation, the corpus-test runner (rule-fires-on-bad / rule-silent-on-good as a structured assertion). Skills are thin wrappers shelling out to the binary, exact parallel of `/slo-research`.
- **Risks**: most of the actual work is LLM-judgment (writing credible Rust variations of a CWE class, mapping a bug-fix-diff to the variation-enumeration template) — Rust adds rigidity to the deterministic parts (validate / fire / silent) but does not help the squishy parts; CWE-taxonomy ingestion becomes its own ongoing tax (RustSec advisories add up daily, OSV adds Rust ecosystem entries weekly); echoes the biz-skill-pack Approach C critique that prose-shaped work should not gate on `cargo build` cycles — when Sherif hits a new bug at 11pm he wants the rule extended, not a release tag; if the crate ever lags behind a Semgrep version bump that changes pattern syntax, the whole pack is broken until a crate release ships.

## Approach C — hybrid skill pack + tiny verifier (local / desktop)

- **Effort**: ~1.5 weeks total. `/slo-rulegen` v1 in week 1 (skills + `references/sast/`), tiny `sast-verify` xtask + `/slo-ruleverify` v1 in 3-4 days.
- **Wedge**: skills do all LLM-heavy work — CWE map authoring (as a baked, checked-in `references/sast/cwe-map-rust.md` produced ONCE during runbook M0 and updated only when a new CVE class emerges), snippet generation, rule drafting. A small verifier ships as an `xtask` — `cargo xtask sast-verify` per the existing `xtasks/` convention in this repo, with a single `xtasks/sast-verify/` Cargo crate added to the workspace — that does ONE deterministic loop: given `(rule.yaml, bad_snippet.rs, good_snippet.rs)`, call `semgrep --validate`, run `semgrep <rule> <bad>`, run `semgrep <rule> <good>`, assert fire-on-bad and silent-on-good with structured exit codes. The xtask pattern is preferred over a shell script for repo consistency (matches `sldo-research`, `sldo-install`, etc.), gets `cargo test` for free, and keeps the verifier in the workspace test baseline. Pre-built `references/sast/` scaffolding holds the Semgrep rule-syntax cheat-sheet, the Rust-CWE corpus snippet templates, the rule-pack manifest schema, and the per-CWE variation enumeration files (`variations/cwe-248.md`, `variations/cwe-190.md`, etc.) that drive minimum-coverage. Same shared-scaffolding pattern that the biz-skill-pack idea doc landed on.
- **Risks**: the "tiny xtask" tends to grow — easy to scope-creep into a full `/slo-secscan` runner, blurring the skill boundary; if `references/sast/` is too thin (especially the variation-templates dir), rule quality reverts to Approach A's drift profile and the breach mitigation collapses; CI-injection scope-creep — week 1 should NOT also try to wire the GitHub Action and the pre-commit hook, that is M3 of Runbook A; the references-as-checked-in-CWE-map pattern means the second language (TypeScript, Runbook B) needs its own `cwe-map-typescript.md` written by hand, which is the price for not building the ingestion pipeline of Approach B; adding a Cargo crate to the workspace means the parked `sldo-tauri` constraint (no `--workspace` test baseline) still applies — `xtasks/sast-verify/` must be added to the explicit `cargo test -p` baseline list in CLAUDE.md, not assumed via `--workspace`.

## Recommendation

Approach C, split across two runbooks for a cleaner cut. **Runbook A — Rust + Semgrep, end-to-end (3 milestones).** M1 ships the wedge: `/slo-rulegen` v1 (Rust + Semgrep + top-10 CWEs bootstrap), `/slo-ruleverify` v1 (the corpus-driven fire-on-bad / silent-on-good `xtasks/sast-verify` Cargo task), and `references/sast/` shared scaffolding (CWE map, variation templates, Semgrep rule-syntax cheat-sheet, rule-pack manifest schema). Verification is a *hard gate* on `/slo-rulegen` output in the same milestone — not a follow-on — because the prolonged-outage risk kills the pack if rules can land unverified. M2 ships extend-mode: `/slo-rulegen --extend` takes `(bug_summary, fix_diff, file_paths)` from a Claude-found bug, outputs `(variation_snippets[3-5], rules[3-5], corpus_pairs[3-5])`, and appends to the project's existing Semgrep pack only after each of the 3-5 rules passes the verifier gate. M3 wires CI + dev-env: pre-commit hook (or Rust-native equivalent if Open question 5 surfaces one) for local fire-on-type, GitHub Action for PR-blocking. **Runbook B — TypeScript replication, separate runbook for cleaner cut.** Reuses the `references/sast/` scaffolding from Runbook A without modifying any skill code, validating that the architecture is genuinely language-agnostic — the skills consume a `references/sast/cwe-map-<lang>.md` and `references/sast/variations/<lang>/cwe-XXX.md` set, so adding TypeScript is "write the references files, run the same skills." If Runbook B requires skill-code changes, the architecture failed and Approach C needs revisiting. Approach A is rejected because the prolonged-outage risk requires a deterministic verifier gate that prompt-only cannot guarantee. Approach B is rejected because `cargo build` between bug-found and rule-extended breaks the recurring loop the user described — the user's pain is that he wants the rule pack extended *now*, not after a crate release.

## Open questions for /slo-research

1. **CWE susceptibility ranking for Rust** — what are the top 10 CWE classes that idiomatic safe-Rust + the popular ecosystem crates (axum, tokio, serde, sqlx, reqwest) are actually most susceptible to in production, ranked by frequency-in-the-wild not theoretical risk? Authoritative sources: RustSec advisory-db at https://github.com/RustSec/advisory-db (categorise the last 24 months of advisories by CWE), OSV.dev API at https://osv.dev/#use-the-api (Rust ecosystem advisories with CWE mapping), public Rust audit reports (Trail of Bits, NCC Group, cure53) for "what bugs they actually find vs. what the threat model said they'd find," and Rust-Sec / cargo-audit issue tracker for false-negative discussions. Output: a ranked list of CWE IDs with frequency-in-the-wild, plus a one-paragraph "why Rust is susceptible to this class despite the safety claim" per CWE.

2. **Variation enumeration per CWE for Rust** — for the top-3 CWE classes (likely panic-DoS / integer-overflow-in-capacity-or-length / unsafe-FFI-misuse), what are all the credible Rust idioms that express the same vulnerability class? Need 5-10 distinct variation patterns per class with a representative bad-snippet + good-snippet pair each. Sources: Clippy lints (`clippy::unwrap_used`, `clippy::expect_used`, `clippy::panic`, `clippy::indexing_slicing`, `clippy::arithmetic_side_effects`), the Rustonomicon's unsafe-code chapter, real-world fixed bugs from RustSec advisories with the fix-diffs as ground truth, and the panic-strategy section of the Rust Book. This output drives `references/sast/variations/cwe-XXX.md` and is the breach-mitigation enforcement file.

3. **Semgrep rule-syntax constraints for Rust** — what does Semgrep's Rust support actually cover in 2026? Specifically: does `pattern-either` work the same on Rust as on Python; does taint mode have Rust source/sink declarations; does metavariable type filtering work on Rust generics; does `pattern-inside` work for `unsafe { ... }` blocks; what is the false-positive rate on macro-heavy code (every axum handler is macro-expanded). Sources: the existing `rust/` rules in the public semgrep-rules registry, semgrep/semgrep-rules issue tracker filtered to "language:rust," the Semgrep "Languages" page, and any 2025-2026 conference talks on Semgrep's Rust frontend. Output drives the `references/sast/semgrep-rust-syntax.md` cheat-sheet that the skills consume.

4. **Corpus-and-pack layout convention** — how do other rule-pack maintainers (semgrep/semgrep-rules itself, Trail of Bits' semgrep-rules, the deepsource ruleset, returntocorp's pro packs) physically lay out their bad-snippet + good-snippet corpus? Per-rule `.test.yaml` co-located with the rule? A single `tests/` dir with naming convention? Inline in the rule via `metadata: examples`? Convention matters because `/slo-ruleverify` reads the corpus and `/slo-secscan` runs the rules — we should match the most common convention so the generated pack is portable to other Semgrep CI setups (`semgrep ci`, `semgrep --config <pack>`).

5. **CI + dev-env wiring options for Rust projects in 2026** — what are the canonical local-dev entry points for Semgrep on a Rust project (pre-commit hook via `pre-commit`, rust-analyzer plugin, `cargo` subcommand wrapper, on-save VS Code extension), and what are the canonical CI entry points (GitHub Action `returntocorp/semgrep-action`, GitLab CI template, Bazel rules)? Need this for M3 of the runbook (`/slo-secscan` + CI injection). Specifically: has a Rust-native equivalent to `pre-commit` emerged that handles `cargo`-shaped repos better, or is `pre-commit` still the canonical entry point?

6. **Publication-risk precedent on vulnerable-snippet corpora** — does shipping a public corpus of "deliberately vulnerable Rust snippets" trigger any practical export-control or platform-policy issue? GitHub's Acceptable Use Policy on "active malware or exploits" is the immediate concern. Need a precedent check: does semgrep/semgrep-rules ship snippets, how are they labelled, has any public rule-pack repo had a takedown notice? Output decides whether the corpus output dir defaults to `.gitignore`'d (most conservative), test-only-marked-but-tracked (middle), or fully public (most permissive). The compliance-fine risk above currently anchors the default at `.gitignore`'d.
