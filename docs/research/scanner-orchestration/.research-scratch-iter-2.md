## Deepened Findings

### Q1 — Semgrep CI integration (consolidated, high confidence)

- **`semgrep ci` auto-detects baseline on GitHub Actions.** The `SEMGREP_BASELINE_REF` / `--baseline-ref` knob is reserved for "other" CI providers; on GitHub Actions and GitLab, `semgrep ci` derives the merge base from the PR event payload. **Operational implication:** the v1 wedge does not need to compute or pass a baseline ref on GitHub — the checkout fetch depth is the actual lever. Source: [Add Semgrep to CI/CD](https://semgrep.dev/docs/deployment/add-semgrep-to-ci), [CI configuration reference](https://semgrep.dev/docs/semgrep-ci/configuration-reference/).
- **Fetch-depth is the real footgun for diff-aware scans.** `actions/checkout@v4` defaults to `fetch-depth: 1`; `semgrep ci` then can't reach the merge base and emits "failed to run a git command." Mitigations: `fetch-depth: 0` on `actions/checkout`, or set `SEMGREP_GHA_MIN_FETCH_DEPTH`. Source: [Semgrep KB — git command errors](https://semgrep.dev/docs/kb/semgrep-ci/git-command-errors), [CI environment variables](https://semgrep.dev/docs/semgrep-ci/ci-environment-variables).
- **Recommended adoption pattern (split-job):** scheduled full-scan job on `main` for trend tracking + a `pull_request` diff-aware job that gates merges. Both can run from one workflow with conditional steps. Source: [Add Semgrep to CI/CD](https://semgrep.dev/docs/deployment/add-semgrep-to-ci), [Trail of Bits — Continuous integration](https://appsec.guide/docs/static-analysis/semgrep/continuous-integration/).
- **OSS-only PR-comment path:** SARIF upload to GitHub Code Scanning via `github/codeql-action/upload-sarif`. Findings appear in the repo's Security tab and as PR review comments; no Semgrep AppSec Platform required. Source: [0xdbe — How To Enable Code Scanning With Semgrep](https://0xdbe.github.io/GitHub-HowToEnableCodeScanningWithSemgrep/).
- **Severity gating mechanism.** Severity-based blocking is a config-level concern, not a CLI flag in the `semgrep ci` invocation. Set rule-level `severity: ERROR` in the rule (or filter the pack) and let `semgrep ci` exit non-zero on blocking-severity findings. Source: [Findings in CI](https://semgrep.dev/docs/semgrep-ci/findings-ci) — confidence: medium (page references behavior; specific flag semantics not deep-read).

### Q2 — CWE → rule-pack mapping (verdict: REGISTRY METADATA IS CANONICAL)

Strong primary-source confirmation this round.

- **Schema is structured and required for security rules.** Every Semgrep registry rule with `category: security` must populate: `cwe` (array of strings), `owasp` (array, multi-year), `confidence` (HIGH/MEDIUM/LOW), `subcategory` (`vuln` / `audit` / `secure default`), `likelihood`, `impact`, `vulnerability_class`. All registry rules also require `technology`, `category`, `references`. Source: [Contribute rules to the Semgrep Registry](https://semgrep.dev/docs/contributing/contributing-to-semgrep-rules-repository).
- **Concrete YAML proof from a live registry rule** (`semgrep-rules/python/django/security/injection/sql/sql-injection-using-raw.yaml`):
  ```yaml
  metadata:
    cwe:
    - "CWE-89: Improper Neutralization of Special Elements used in an SQL Command ('SQL Injection')"
    owasp: [A01:2017 - Injection, A03:2021 - Injection, A05:2025 - Injection]
    technology: [django]
    cwe2022-top25: true
    cwe2021-top25: true
    subcategory: [vuln]
    likelihood: MEDIUM
    impact: HIGH
    confidence: MEDIUM
  ```
  Note: this older rule is missing `vulnerability_class` — evidence that **legacy rules predating the schema tightening exist** and the skill must tolerate field absence. Source: live `semgrep/semgrep-rules` raw fetch (verified 2026-04-26).
- **Verdict for the wedge:** the skill should query the Semgrep registry / `semgrep-rules` repo directly via `metadata.cwe` substring match (`CWE-NN:`) and `metadata.technology` intersect — **no need to author and maintain a CWE → pack table by hand.** Confidence: high for this primary path; medium on completeness — coverage gaps per language exist but can be discovered empirically by scanning the registry.
- **Pre-built pack shortcuts that already do CWE filtering:** `p/owasp-top-ten`, `p/cwe-top-25` are curated by Semgrep and re-mapped to OWASP 2025 in early 2026. For "I want CWE-77/78/89 on a Python+Django repo" the skill can either pick `p/cwe-top-25` wholesale OR programmatically subset `semgrep-rules` by `metadata.cwe ∋ "CWE-89:"` AND `metadata.technology ∋ "django"`. Source: [cwe-top-25 ruleset](https://semgrep.dev/p/cwe-top-25), [OWASP Top 10 2025: What's New](https://semgrep.dev/blog/2026/owasp-top-10-2025-whats-new/).
- **Caveat — SARIF/CWE round-trip historically buggy.** Issue [semgrep/semgrep #4673](https://github.com/semgrep/semgrep/issues/4673) documented `cwe`/`owasp` as arrays breaking SARIF export. Verify on current Semgrep ≥ 1.161 before relying on multi-CWE rules in SARIF uploads.

### Q3 — PCI DSS 6.3.2 / SOC 2 CC7.1 evidence (partial; primary-source still required)

What two rounds of search **did** establish, citing public sources only:

- **PCI DSS 6.3.2 expects two artifacts:** (a) evidence the review happened (tool output OR peer-review record), and (b) evidence each finding was either remediated or risk-accepted. Quote: *"PCI DSS assessors will expect to see evidence that you review scan results and remediate or formally accept the risk of each finding."* Source: [securityreview.ai — PCI DSS 4.0 CI/CD risk](https://www.securityreview.ai/blog/is-your-code-or-your-pipeline-the-bigger-pci-dss-4-0-risk). **No public source fixed format (SARIF vs PDF) or retention window.** Confidence: medium on the two-artifact claim, low on format/retention specifics.
- **PenTest Partners' v4.0 evidence checklist** lists *"Code review records: Evidence of peer reviews or automated scans of application code"* under R6 — confirms tool output is acceptable evidence in principle. Source: [Pen Test Partners — PCI DSS v4.0 evidence checklist](https://www.pentestpartners.com/security-blog/pci-dss-v4-0-evidence-and-documentation-requirements-checklist/).
- **NIST SSDF SP 800-218 PW.7 / PW.8** are the canonical secure-coding practice that maps to PCI 6.3.2 + SOC 2 CC7.1. Practice description: PW.7 = review/analyze human-readable code; PW.8 = test executable code. Implementation examples include "automated static analysis tools in the CI pipeline." Retention is referenced abstractly: NIST 800-218 calls for "establishing and enforcing security and retention policies for artifact data" without prescribing a specific window. Source: [NIST SP 800-218](https://csrc.nist.gov/pubs/sp/800/218/final), [aikido — NIST SSDF explained](https://www.aikido.dev/learn/compliance/compliance-frameworks/nist-ssdf). Confidence: medium — the PDF didn't fetch as text; the secondary summaries align consistently across sources.
- **What the wedge should capture per scan run** (inferred from the above, flagged as **inference, not direct citation**):
  | Datum | Why | Confidence |
  |---|---|---|
  | Scan timestamp + git SHA scanned | Proves "before release" timing for 6.3.2 | High |
  | Rule pack identifier(s) + Semgrep version | Reproducibility / dispute resolution | High |
  | SARIF output (full, not just summary) | Industry-standard format, GitHub-native | High (format choice); auditor acceptance is medium |
  | Findings with status (`open`, `fixed`, `risk_accepted`) | "Addressed" criterion for 6.3.2 | High |
  | CWE list claimed-as-covered + CWE list actually-tagged-by-rules-run | Defends against "mapped-but-not-scanned" audit gotcha | Inference |

- **"Mapped-but-not-scanned" audit-failure pattern is intuitive but unsourced.** Two rounds of searches did not surface a published QSA write-up of an audit failing on this exact pattern. Treat the risk as plausible, not documented.
- **Primary-source read is still required** for any production claim: the PCI DSS 4.0.1 Requirements and Testing Procedures PDF (PCI Council members-only) and AICPA TSC 2017 + 2022 revisions. Search did not surface a publicly-quotable testing procedure 6.3.2.a/.b text.

### Q4 — `pull_request_target` posture (high confidence; one new datum)

- **Material 2025 GitHub change confirmed.** From 2025-12-08, `pull_request_target` workflow file + checkout commit are forced to come from the **default branch**, regardless of the PR's base branch. `GITHUB_REF` resolves to the default branch and `GITHUB_SHA` to its latest commit. This kills the "outdated workflow on a stale base branch is still exploitable" attack class. Source: [GitHub Changelog 2025-11-07](https://github.blog/changelog/2025-11-07-actions-pull_request_target-and-environment-branch-protections-changes/).
- **Implication for the v1 wedge:** the skill's hard ban on `pull_request_target` is correct; even with the 2025-12-08 mitigation, `pull_request_target` still confers full secret access on PR runs and the secure-by-default posture is `on: pull_request` with no checkout of untrusted code into a privileged context. Confidence: high.
- **Two-workflow split is the canonical hardening pattern when comments need elevated permissions.** Workflow A (`on: pull_request`) does the analysis with no secrets and uploads SARIF as an artifact; Workflow B (`on: workflow_run` after Workflow A succeeds) reads the artifact and posts comments with `pull-requests: write`. Source: [GitHub Security Lab — Preventing pwn requests](https://securitylab.github.com/resources/github-actions-preventing-pwn-requests/).
- **Minimum-permissions block for the SAST workflow** (cited rationale per line):
  - top-level `permissions: {}` — least-privilege default ([Wiz](https://www.wiz.io/blog/github-actions-security-guide))
  - per-job `contents: read` — for `actions/checkout` ([GitHub Security Lab](https://securitylab.github.com/resources/github-actions-preventing-pwn-requests/))
  - `security-events: write` only on the SARIF-upload job — required by `github/codeql-action/upload-sarif` ([0xdbe walkthrough](https://0xdbe.github.io/GitHub-HowToEnableCodeScanningWithSemgrep/))
  - actions pinned by SHA (the tj-actions lesson) ([GHSA-mrrh-fwg8-r2c3](https://github.com/advisories/ghsa-mrrh-fwg8-r2c3))
- **First-time-contributor handling.** GitHub's "Require approval for first-time contributors" repo setting is the documented gate; combined with `on: pull_request` (no fork-secret access), this is sufficient — no need for `pull_request_target`. Confidence: high.

### Q5 — Threat-model-driven scanner configuration (verdict: WEDGE IS UNOCCUPIED)

- **OTM 0.2.0 has a `cwes` field on threats** — the spec is structurally compatible with downstream CWE consumption. Required top-level fields: `otmVersion`, `project`. Threats: `id`, `name`, `risk` (required) + `description`, `categories`, `cwes` (array of strings), `tags` (optional). Mitigations: `id`, `name`, `riskReduction` (required). Source: [iriusrisk/OpenThreatModel `otm_schema.json`](https://github.com/iriusrisk/OpenThreatModel/blob/main/otm_schema.json), confirmed via direct fetch.
- **OTM is stagnant.** Latest schema release: 0.2.0 (2023-08-30). License: CC-BY-SA-4.0. No documented downstream Semgrep / SAST consumer surfaced across two search rounds. Source: same.
- **No published OTM → Semgrep config converter exists.** Adjacent prior art:
  - **Autogrep** (LambdaSec) — automates Semgrep rule *generation* from CVE patches, orthogonal to threat-model-driven *selection*. Source: [Autogrep](https://lambdasec.github.io/AutoGrep-Automated-Generation-and-Filtering-of-Semgrep-Rules-from-Vulnerability-Patches/).
  - **AppSec Untangled "Threat Modeling Handbook #5"** — manual practice, not a tool. Source: [Medium](https://medium.com/appsec-untangled/threat-modeling-handbook-5-convert-your-threat-model-into-an-automated-pentest-using-devsecops-84efcd138202).
  - **`sast-semgrep` and `claude-build-workflow-sast-semgrep` Claude skills** — wire Semgrep into a project but neither reads a threat model. Source: [claude-plugins.dev sast-semgrep](https://claude-plugins.dev/skills/@AgentSecOps/SecOpsAgentKit/sast-semgrep).
  - **SecOpsTM** (ellipse2v) — STRIDE-as-code with MITRE ATT&CK / D3FEND / CAPEC mapping; outputs reports, not Semgrep configs. Source: [github.com/ellipse2v/SecOpsTM](https://github.com/ellipse2v/SecOpsTM).
- **Why it doesn't already exist (defensible answer):** (1) threat-model formats fragmented across pytm, Threat Dragon, OTM, Threagile (CycloneDX TM-BOM is the in-flight unifier, no shipping consumers yet); (2) Semgrep registry only got dense, queryable CWE metadata in the last ~24 months; (3) the "diff the threat model → re-derive ruleset → surface diff" loop is workflow-design work, not algorithm work — the kind of integration that vendors don't ship because it's small surface area, and OSS hasn't because the constituent pieces only just arrived. Confidence: medium-high (synthesis from the negative search results plus inferred reasoning).
- **Strategic positioning:** v1 should accept *Markdown threat models* (the SLO convention — `docs/design/<slug>-threat-model.md`) and **optionally** parse OTM-0.2.0 `threats[].cwes` arrays as a structured input. This keeps the wedge focused on what SLO actually emits today while leaving a non-binding upgrade path to OTM consumption.

## Library Evaluations

### Semgrep CLI (CE) `1.161.0` (April 2026)

- **License:** LGPL-2.1 (CLI) + Semgrep Rules License for community rules.
- **Pros:** monthly cadence; SARIF output supported; diff-aware on GitHub auto-magic; rich queryable `metadata.cwe`/`technology` schema; OWASP 2025 re-mapping completed in early 2026.
- **Cons:** SARIF + array-typed `cwe`/`owasp` had historical bugs ([#4673](https://github.com/semgrep/semgrep/issues/4673)) — verify on current minor; `semgrep ci` command-line surface is large, easy to misuse for severity gating.
- **Fit:** **primary engine, no substitutes considered.** Pin a specific minor (e.g., `1.161.x`) in the emitted workflow and bump deliberately.
- **Source:** [PyPI semgrep](https://pypi.org/project/semgrep/), [release notes index](https://semgrep.dev/docs/release-notes).

### Semgrep registry packs `p/owasp-top-ten`, `p/cwe-top-25`, `p/security-audit`

- **License:** Semgrep Rules License (review whether commercial OSS-pack use is fully clean for the consuming project).
- **Pros:** curated, recently re-mapped to OWASP 2025; rules carry `cwe2022-top25`/`cwe2021-top25` boolean tags for fast filtering; 4k+ rules.
- **Cons:** "rule pack" is opaque — picking `p/cwe-top-25` includes everything, even rules unrelated to the threat model's CWE list, which conflicts with the wedge's "tuned, not generic" promise.
- **Fit:** use the curated packs as the *fallback default* when threat-model parsing yields nothing; otherwise programmatically subset by `metadata.cwe ∋ "CWE-NN:"` AND `metadata.technology ∋ stack`.

### `semgrep-rules` raw repo (`github.com/semgrep/semgrep-rules`)

- **Pros:** ground-truth source, queryable as files; lets the skill ship a deterministic rule-list rather than `--config p/foo` which is opaque to audit.
- **Cons:** ~4k YAML files; needs a small index step; legacy rules have schema gaps (e.g., missing `vulnerability_class`).
- **Fit:** the auditable path is "skill enumerates the rule files used by SHA, commits them as `.semgrep/rules/` or pins via `--config <pack>@<sha>`." This makes the "I scanned for CWE-89" claim defensible against the mapped-but-not-scanned audit failure mode flagged in Q3.

### OWASP Threat Dragon (current OWASP flagship)

- **License:** Apache-2.0.
- **Pros:** active OWASP project; produces JSON + OTM exports; threats can carry `cwe` references.
- **Cons:** out-of-the-box threat objects don't always populate CWE; UI-driven not file-first; not the format SLO emits.
- **Fit:** **secondary input** behind Markdown — accept its OTM export but don't require it.

### OWASP pytm

- **License:** GPL-3.0.
- **Pros:** code-as-threat-model, programmatic CWE export possible; active OWASP project.
- **Cons:** Python dependency; no built-in Semgrep adapter; output is reports, not OTM.
- **Fit:** out of v1 scope; reasonable v2 additional input source.

### Open Threat Model (OTM) spec `0.2.0`

- **License:** CC-BY-SA-4.0.
- **Pros:** stable since 2023-08-30; explicit `threats[].cwes` array; tool-agnostic.
- **Cons:** stagnant since 2023; no documented SAST consumer; ecosystem fragmented vs. CycloneDX TM-BOM (in flight).
- **Fit:** v1 *optional* parser path; do not depend on it.

### `actions/checkout@v4` + `github/codeql-action/upload-sarif@v3`

- **License:** MIT (both).
- **Pros:** GitHub-maintained; SARIF upload integrates with Code Scanning UI without Semgrep AppSec.
- **Cons:** must be SHA-pinned (tj-actions lesson); checkout's default `fetch-depth: 1` breaks Semgrep's diff-aware scan.
- **Fit:** template both as SHA-pinned in the emitted workflow; emit `fetch-depth: 0` in the checkout step.

### Autogrep (LambdaSec research artifact)

- **License:** unspecified in the public write-up.
- **Pros:** demonstrates LLM-driven Semgrep rule synthesis from patches.
- **Cons:** research write-up only; no packaged release; orthogonal problem (rule generation, not selection).
- **Fit:** out of scope for v1; tangentially relevant to the eventual `/slo-rulegen` integration.

## Architecture Options

### Option 1 — Pure Markdown skill, registry-query at scan time (matches idea-doc Approach A, refined)

- **Shape.** SKILL.md drives Claude to (a) parse the threat-model Markdown for `CWE-\d+` mentions, (b) detect stack from manifests, (c) **fetch the live Semgrep registry index** (`semgrep-rules` repo) and filter rules by `metadata.cwe ∋ <each CWE>` AND `metadata.technology ∋ <stack>`, (d) emit `.semgrep.yml` with explicit rule paths and `.github/workflows/sast.yml` with safe defaults, (e) on threat-model edit, re-run and emit a diff PR.
- **What's new vs. prior round.** Q2 verdict eliminates the hand-maintained CWE → pack table — registry metadata is queryable. Q4 confirms `on: pull_request` + `permissions: {}` + SHA-pinned actions is the safe template; no `pull_request_target`.
- **Trade-offs.** Pros: low effort (~1 person-week), no new Rust crate, deterministic-enough because the rule set is committed back into the repo as explicit paths/SHAs (not opaque `p/foo`). Cons: registry queries happen via Claude's prompt logic, which is less crisp than typed code; the skill must include a fallback path when the registry is unreachable.
- **Best when.** v1 wedge for solo OSS maintainers; SLO's "skills are the canonical interface" posture.

### Option 2 — Markdown skill + tiny `crates/sldo-sast` index helper

- **Shape.** Same as Option 1 but a small Rust binary (one command, ~200–400 lines) handles two deterministic operations: (a) clone/cache `semgrep-rules` and build a sqlite/JSON index of `(cwe, technology) → rule-path`, (b) given a CWE list + stack, emit the explicit rule-path list. The skill calls the binary; everything else (YAML emission, PR opening) stays in Markdown.
- **Trade-offs.** Pros: deterministic CWE → rules query (auditable, unit-testable, reproducible across Claude runs), strong defense against the Q3 "mapped-but-not-scanned" failure mode. Cons: re-introduces the `cargo install` bootstrap that the 2026-04 cleanup deliberately removed; conflicts with the "skills are canonical" direction in CLAUDE.md.
- **Best when.** Enterprise/regulated users who need reproducibility evidence; *not* the v1 wedge.

### Option 3 — Markdown skill that emits committed rule SHAs + SARIF retention manifest

- **Shape.** Option 1, plus a per-scan-run `.semgrep/manifest.json` capturing: `semgrep_version`, `rules_sha` (the commit of `semgrep-rules` from which rules were drawn), `cwes_claimed` (from threat model), `cwes_actually_covered` (computed from union of `metadata.cwe` across selected rules), `stack`, `scan_timestamp`, `git_sha`. SARIF output retained as a workflow artifact with a long retention window.
- **Trade-offs.** Pros: front-loads everything Q3 inferred-but-couldn't-fully-cite — `cwes_claimed` vs `cwes_actually_covered` divergence is computable in v1, defending against the mapped-but-not-scanned failure mode. Cons: introduces a manifest schema the user has to live with; "long retention" needs a concrete number that PCI 6.3.2 doesn't crisply specify in public sources.
- **Best when.** v1 wants to leave a clean upgrade path to a post-v1 coverage doc. Compatible with Option 1.

### Option 4 — Two-workflow split (analysis + commenter), pre-templated

- **Shape.** Skill emits two workflow files: `sast-analysis.yml` (`on: pull_request`, no secrets, runs `semgrep ci`, uploads SARIF as artifact) and `sast-comment.yml` (`on: workflow_run`, `permissions: pull-requests: write`, downloads artifact, posts a single PR comment summarizing findings).
- **Trade-offs.** Pros: matches the GitHub Security Lab canonical pattern for SAST that needs to comment on fork PRs without `pull_request_target`. Cons: more YAML for a v1 wedge; users without fork PRs (most solo maintainers) don't need the comment workflow — SARIF upload to Code Scanning already shows in PR review.
- **Best when.** Project receives meaningful fork-PR traffic and maintainers want a single bot comment per scan rather than the Code Scanning tab. **Likely v2.**
