## Deepened Findings

### Correction — PCI DSS requirement number is 6.2.3, not 6.3.2 (high confidence)

The idea doc and brief both cite **PCI DSS 6.3.2** for "code review of custom and bespoke software." That number is **stale (v3.2.1)**. In **PCI DSS v4.0.1** (current as of 2026-04), the relevant requirements are:

- **6.2.3** — *"Bespoke and custom software is reviewed prior to release into production or to customers, to identify and correct potential coding vulnerabilities."* Defined-approach testing procedure: examine software development procedures + interview personnel; reviewers must be **other than the originating author**, **knowledgeable in code-review techniques and secure coding**; manual, automated, or hybrid is acceptable; **management approval prior to release** is required.
- **6.2.3.1** — Code-change-specific sub-requirement for the same review on changes.
- **6.3.2** in v4.0.1 is a **different** requirement: "**An inventory of bespoke and custom software, and third-party software components incorporated into bespoke and custom software is maintained**" — i.e., the new SBOM-adjacent inventory mandate.

Sources: [VISTA InfoSec — PCI DSS Req 6 changes v3.2.1→v4.0](https://vistainfosec.com/blog/pci-dss-requirement-6-changes-from-v3-2-1-to-v4-0-explained/), [securityreview.ai](https://www.securityreview.ai/blog/is-your-code-or-your-pipeline-the-bigger-pci-dss-4-0-risk), [Halock — v4.0.1 software catalog mandate](https://www.halock.com/what-is-the-new-pci-dss-v4-0-1-software-catalog-mandate/), [Cybeats — 6.3.2 SBOM 2025 readiness](https://www.cybeats.com/blog/pci-dss-4-0-sboms-a-2025-readiness-guide). Confidence: **high**. Action for v1: idea doc + threat-model template should reference **6.2.3 (review)** for SAST coverage and **6.3.2 (inventory/SBOM)** separately if relevant. The v1 wedge addresses 6.2.3.

### Q1 — Semgrep CI: severity gating, flag names, baseline (high confidence)

Three corrections / consolidations to the previous round:

- **The flag is `--baseline-commit`, not `--baseline-ref`.** Env var is `SEMGREP_BASELINE_REF`. CLI doc verbatim: *"Only show results that are not found in this commit hash. Aborts run if not currently in a git directory, there are unstaged changes, or given baseline hash doesn't exist."* On GitHub Actions, `semgrep ci` derives the baseline from PR event payload automatically — explicit baseline-commit flag is for "other" CI providers. Source: [Semgrep CLI reference](https://semgrep.dev/docs/cli-reference), confirmed by webfetch.
- **`--config` is "not supported in ci mode"** per the canonical CLI reference. In practice, OSS users invoke `semgrep ci --config p/cwe-top-25` (or via `SEMGREP_RULES`) and it works because without a Semgrep AppSec login it falls back to honoring `--config`. **The wedge should set `SEMGREP_RULES` env var** rather than rely on a documented-as-unsupported flag, to avoid future breakage. Confidence: medium-high (documented-vs-observed conflict; needs an empirical sweep to lock in).
- **Severity-based blocking does NOT exist for OSS users.** Definitive doc text: *"If you do not use Semgrep AppSec Platform with Semgrep in CI or Semgrep Managed Scans (that is, you are using a stand-alone setup), all Semgrep findings are blocking findings."* Source: [Configure blocking findings | Semgrep](https://semgrep.dev/docs/semgrep-ci/configuring-blocking-and-errors-in-ci). **Implication for v1**: rule *selection* is the only severity-gating mechanism available — the skill cannot say "include these rules but only block on ERROR." If a rule is selected, every finding from it blocks. Pre-existing findings handled via baseline (auto-derived on GHA) and/or `.semgrepignore`. Confidence: **high**.

Exit-code semantics for `semgrep ci`: `0`=clean, `1`=findings present, `2`=fatal error, `3`=invalid target code, `4`=invalid pattern, `5`=unparseable YAML, `7`=missing config, `8`=invalid language, `13`=invalid API key. `--suppress-errors` only suppresses *error* exit codes, not finding exit codes.

## Library Evaluations

### Semgrep CLI (CE) `1.161.x` (April 2026)

- **License**: LGPL-2.1 (CLI) + Semgrep Rules License for community rules.
- **Pros**: monthly cadence; dense queryable `metadata.cwe`/`technology`; native GHA baseline detection; SARIF output; OSS-installable via `pip install semgrep`.
- **Cons**: no severity-based gating for stand-alone users (rule-selection IS gating); `--config` officially "unsupported in ci mode" but works in practice (docs/observed conflict); SARIF-vs-array-CWE historical bugs ([#4673](https://github.com/semgrep/semgrep/issues/4673)) — verify on installed version; CLI surface large.
- **Fit**: primary engine, no substitutes. Pin minor version in emitted workflow; bump deliberately. Use `SEMGREP_RULES` env var rather than `--config` flag in `semgrep ci` to dodge the docs-vs-behavior risk.

### `semgrep-rules` raw repo (`github.com/semgrep/semgrep-rules`)

- **Pros**: ground truth; lets the skill commit `.semgrep/rules/` with explicit pinned SHAs (auditable); enables the `cwes_actually_covered` calculation.
- **Cons**: ~4k YAMLs, needs an index step; legacy rules have schema gaps.
- **Fit**: query at scan time, emit explicit rule list, make audit defense possible.

### Semgrep curated packs `p/cwe-top-25`, `p/owasp-top-ten`, `p/security-audit`

- **Pros**: curated, recently re-mapped to OWASP 2025; small and well-known.
- **Cons**: opaque "pack" identifier conflicts with audit reproducibility; pulling all 4k+ rules conflicts with "tuned not generic" wedge promise.
- **Fit**: **fallback only** when threat-model parsing yields nothing, or when stack/CWE intersection is empty in registry.

### `actions/checkout@v4.2.2` (`11bd71901bbe5b1630ceea73d27597364c9af683`)

- **License**: MIT.
- **Pros**: GitHub-maintained; SHA-pinnable.
- **Cons**: default `fetch-depth: 1` breaks Semgrep diff-aware scans → must emit `fetch-depth: 0`.
- **Fit**: SHA-pin in template; document `fetch-depth: 0` rationale inline.

### `github/codeql-action/upload-sarif@v3`

- **License**: MIT.
- **Pros**: native Code Scanning UI integration without Semgrep AppSec Platform; surfaces findings as PR review comments automatically.
- **Cons**: requires `security-events: write` permission scoped to that job only; SHA must be re-pinned periodically.
- **Fit**: SARIF-upload step in the SAST workflow; only job with elevated permissions.

### OTM spec `0.2.0` + Threat Dragon (Apache-2.0) / pytm (GPL-3.0)

- **OTM**: stable since 2023-08-30, stagnant; `threats[].cwes` is the integration point. Optional v1 input.
- **Threat Dragon**: active; OTM export available. Secondary v1 input.
- **pytm**: defer to v2; out of v1 scope.

### CycloneDX Threat-Model BOM (TM-BOM)

- **Status**: in-flight standard, no shipping consumers as of 2026-04. Re-check in 6 months.
- **Fit**: not v1; potential v2 swap-in if it stabilizes and Threat Dragon / pytm both adopt it.

## Architecture Options

Two viable options crystallize from the consolidated findings.

### Option A — Pure Markdown skill, registry-query at scan time, deterministic rule manifest (recommended)

- **Shape.** SKILL.md drives Claude to: (1) parse `docs/design/<slug>-threat-model.md` for `CWE-\d+` mentions and any `## Stack` section, (2) detect stack from manifest files (`Cargo.toml`, `package.json`, `requirements.txt`, `go.mod`, `pom.xml`, `pyproject.toml`), (3) clone-or-cache `semgrep-rules` at a pinned SHA and filter by `metadata.cwe ∋ <each CWE>` AND `metadata.technology ∋ <stack>`, (4) emit `.semgrep/rules/` with the selected rule files committed, (5) emit `.semgrep.yml` referencing the local rule set, (6) emit `.github/workflows/sast.yml` with the safe template (single `pull_request` workflow, `permissions: {}` top + per-job overrides, SHA-pinned actions, `fetch-depth: 0`, SARIF upload), (7) write `.semgrep/manifest.json` with `cwes_claimed`, `cwes_actually_covered`, threat-model SHA, `semgrep-rules` SHA, generated-at timestamp; (8) on threat-model edit, re-derive and emit a diff PR.
- **Trade-offs.** **Pros**: matches CLAUDE.md's "skills are the canonical interface" direction; ~1 person-week; deterministic enough because the rule set is committed back into the repo as concrete files (not opaque `p/foo`); manifest defends against the inferred mapped-but-not-scanned audit pattern; no new Rust crate. **Cons**: registry filtering happens via Claude prompt logic, less crisp than typed code; `cwes_actually_covered` computation is prompt-driven (test fixtures must verify it).
- **Best when.** v1 wedge for solo OSS maintainers — this is the right answer.
- **What changed since iter 2.** The manifest construction (Option 3 in iter 2) folds *into* the recommended option, not as an alternative. The PCI numbering correction (6.2.3 not 6.3.2) goes into the manifest's frontmatter. The severity-gating-doesn't-exist finding shifts the design: the skill must prove rule-pack tightness because there's no second gate.

### Option B — Markdown skill + tiny `crates/sldo-sast-index` helper (rejected for v1)

- **Shape.** Same as A, but a small Rust binary (~200–400 LOC) handles the deterministic index: clone `semgrep-rules` at a pinned SHA, build a sqlite index of `(cwe, technology) → rule-path`, expose `query --cwe CWE-89 --tech python,django` returning a sorted rule list. Skill shells out for the lookup; everything else stays Markdown.
- **Trade-offs.** **Pros**: deterministic CWE→rules query (auditable, unit-testable, byte-stable across Claude runs), strongest defense against mapped-but-not-scanned audit pattern. **Cons**: re-introduces the `cargo install` bootstrap that the 2026-04 cleanup explicitly removed (CLAUDE.md: *"all other Rust code... was removed in the 2026-04 cleanup — the skills are the canonical interface now"*); conflicts with stated direction.
- **Best when.** A future enterprise / regulated v2 where reproducibility evidence is required. **Not v1.**

### Option C — Two-workflow split (analysis + commenter), pre-templated (deferred)

- **Shape.** Skill emits `sast-analysis.yml` (`on: pull_request`, no secrets, runs `semgrep ci`, uploads SARIF artifact) + `sast-comment.yml` (`on: workflow_run`, `permissions: pull-requests: write`, downloads artifact, posts a single PR comment).
- **Trade-offs.** **Pros**: matches GitHub Security Lab's canonical pattern for SAST that needs to comment on fork PRs without `pull_request_target`. **Cons**: SARIF-to-Code-Scanning already surfaces in PR review without comments — most solo-OSS-maintainer wedges don't need this; doubles the YAML.
- **Best when.** v2 add-on for projects with meaningful fork-PR traffic that want a single bot comment per scan rather than the Code Scanning tab.
