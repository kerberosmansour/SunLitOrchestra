# Synthesis — scanner-orchestration

What the research tells the design. Each paragraph ends with the load-bearing implication.

## Severity gating doesn't exist standalone — rule selection IS the gate

Semgrep's "Configure blocking findings" doc states verbatim: *"If you do not use Semgrep AppSec Platform with Semgrep in CI or Semgrep Managed Scans (that is, you are using a stand-alone setup), all Semgrep findings are blocking findings."* The wedge cannot promise users "include this rule but only block on ERROR" semantics — that gate doesn't exist for non-paying users. The skill must do its severity discrimination at rule-selection time, not at finding-evaluation time. **The design must handle severity gating by tightening rule selection (only include rules whose `metadata.likelihood`, `impact`, and `confidence` clear a configured threshold) rather than promising post-scan severity filtering, because Semgrep's blocking-findings doc confirms standalone users have no post-scan gate.**

## Auto-baseline on GHA — don't emit `--baseline-commit`

`semgrep ci` auto-detects diff range from the PR event payload on GitHub Actions and GitLab CI. The `SEMGREP_BASELINE_REF` / `--baseline-commit` flags are documented as for "other" CI providers. The real operational footgun is `actions/checkout`'s default `fetch-depth: 1`, which prevents `semgrep ci` from reaching the merge base and produces "failed to run a git command." **The design must handle baseline computation by relying on `semgrep ci`'s GHA auto-detection and emitting `fetch-depth: 0` on the checkout step, because Semgrep's CI environment variables doc reserves baseline-flag mechanics for non-GHA providers and the KB git-command-errors doc names fetch-depth as the canonical pitfall.**

## Registry `metadata.cwe` is canonical — query, don't hand-curate

CWE metadata is mandatory on `category: security` rules per the Semgrep registry contributing guide; a `metadata-cwe.yaml` linter enforces tagging at contribution time. The OWASP Top 10 2025 re-mapping completed early 2026 with all 4,000+ rules carrying current taxonomy. No external CWE → Semgrep mapping exists at MITRE, OWASP, or third parties — the registry IS the canonical source. A live Django SQLi rule shows the long-form CWE string convention (`"CWE-89: Improper Neutralization..."`) and demonstrates that legacy rules can be missing fields like `vulnerability_class` even though they carry `cwe`. **The design must handle CWE → rule mapping by querying registry metadata at scan time (not maintaining a hand-curated lookup table) and tolerating missing non-required schema fields, because the registry is the canonical source per the contributing guide and the live rule example proves legacy schema gaps exist.**

## `pull_request_target` ban survives the 2025-12-08 mitigation

GitHub's 2025-12-08 change forces `pull_request_target` workflow source and checkout commit to come from the default branch, killing the "stale-base-branch workflow" attack class. But `pull_request_target` still confers full secret access to PR runs — the wedge's hard ban remains correct because no SAST step the skill emits needs that. The Sysdig 2024 audit, the pytorch self-hosted runner takeover (Oct 2024), Shai Hulud v2 (Nov 2025, ~20k repos), and CVE-2025-61671 (Microsoft, CVSS 9.3) are the post-2024 demonstration set that misconfiguration is widespread and exploited at scale. **The design must handle the auto-emitted workflow's secret-exposure surface by hard-banning `pull_request_target` for SAST jobs and defaulting to `on: pull_request`, because GitHub Security Lab's "Preventing pwn requests" plus the post-2024 incident set show the trigger confers full secret access even after the 2025-12-08 mitigation.**

## SHA-pin every action — tag rewriting is the canonical failure case

`tj-actions/changed-files` (CVE-2025-30066, March 2025) is the canonical "tag rewriting" supply-chain failure: a compromised maintainer pushed a malicious commit and re-pointed `v35` and similar tags. The Shai Hulud v2 self-replicating worm (Nov 2025, ~20k repos) compounded the precedent. The skill emits a workflow file that runs untrusted-author code in a privileged context if any third-party action resolves to a tag the attacker controls. The 2025-08-15 GitHub Actions policy change supports SHA-pinning at org level; the 2026 security roadmap proposes a `dependencies:` lockfile for action SHAs but is not GA. **The design must handle action supply-chain risk by SHA-pinning every third-party action in the emitted template and establishing a refresh cadence (suggested 90 days) as SLO project policy, because CVE-2025-30066 and Shai Hulud v2 demonstrate tag-rewriting attacks at scale and the lockfile mechanism is not yet GA.**

## Compliance claim must cite PCI DSS 6.2.3 (v4.0.1), not 6.3.2

The idea doc cites **PCI DSS 6.3.2** for code review — that's v3.2.1 numbering. In current **v4.0.1**, the code-review requirement is **6.2.3**: *"Bespoke and custom software is reviewed prior to release into production or to customers, to identify and correct potential coding vulnerabilities."* The v4.0.1 6.3.2 requirement is the new bespoke / third-party software inventory mandate — different scope, out of v1. Multiple sources (VISTA InfoSec, Halock, Cybeats, KirkpatrickPrice, Pen Test Partners) corroborate the renumbering. **The design must handle compliance-claim accuracy by citing PCI DSS 6.2.3 v4.0.1 (not 6.3.2 v3.2.1) anywhere SAST coverage is mentioned — idea doc, threat-model template, future coverage-doc skill — because v4.0.1's renumbering moves code-review from 6.3.2 to 6.2.3 and 6.3.2 is now the SBOM-inventory mandate.**

## Threat-model formats are fragmented — Markdown is the v1 input

OTM 0.2.0 (released 2023-08-30, CC-BY-SA-4.0) has a `threats[].cwes` array and is the most structurally compatible threat-model format, but the spec has been stagnant since 2023 and no SAST consumer documented; StartLeft is OTM-input-only. OWASP Threat Dragon's TMF format is incompatible with pytm, Threagile, and OTM per its own wiki. pytm uses internal threat IDs (`INP01`, `CR01`) rather than CWE references. CycloneDX TM-BOM (the OWASP Threat Model Library project's successor) is in-flight with no shipping consumers as of 2026-04. SLO already emits Markdown threat models via `/slo-architect`. **The design must handle threat-model input by accepting Markdown as the v1 canonical format and treating OTM 0.2.0 `threats[].cwes` as an optional parse path (not a requirement), because all four candidate structured formats are fragmented or stagnant and SLO's existing artifact path is Markdown.**

## The wedge is genuinely unoccupied — frame as first-of-its-kind

No published OTM-or-similar → Semgrep config converter exists in OSS or commercial tooling across multiple research rounds. SecOpsTM is the closest adjacent player (STRIDE-as-code with MITRE mappings, but emits Navigator layers, not Semgrep configs). Vendor presets (Snyk Code, GHAS CodeQL, Checkmarx One, Veracode) require manual policy authoring rather than threat-model intake. The two community Claude skills wiring Semgrep (`sast-semgrep`, `claude-build-workflow-sast-semgrep`) don't read threat models. The defensible "why doesn't this already exist?" answer: threat-model formats remain fragmented; Semgrep's CWE metadata only got dense and queryable in the last ~24 months; the diff-and-rederive loop is workflow-design work that vendors don't ship and OSS hadn't built because the constituent pieces only just arrived. **The design must handle competitive positioning by framing the wedge as a first-of-its-kind workflow rather than a vendor-displacement story, because no published OTM-or-similar → Semgrep config converter surfaces in OSS or commercial tooling and the constituent registry-metadata maturity arrived only recently.**

## Audit defense via per-scan-run manifest is defensive, not regulatory

PCI DSS 6.2.3 expects evidence the review happened (tool output OR peer review) plus evidence each finding was remediated or formally risk-accepted. SOC 2 CC7.1 asks for system-operations vulnerability detection but doesn't fix a format. NIST SSDF PW.7 / PW.8 reference automated static analysis in CI without specifying retention. The "mapped-but-not-scanned CWE coverage claim as audit-failure pattern" is intuitive but unsourced after multiple search rounds. The wedge can capture per-scan-run data (timestamp, git SHA scanned, rule pack identifiers + Semgrep version, SARIF output, finding statuses, `cwes_claimed` from threat model vs `cwes_actually_covered` from union of selected rules' `metadata.cwe`, threat-model file SHA, GitHub PR review record as management-approval artifact) — and that capture is genuinely useful — but the `cwes_claimed` vs `cwes_actually_covered` split is **defensive design**, not a regulatory necessity proven by published audit-failure precedent. **The design must handle audit-defense data capture by collecting the per-scan-run manifest in v1 (timestamp, git SHA, rule pack IDs, Semgrep version, SARIF, finding statuses, CWE-claimed-vs-covered diff, threat-model SHA) and framing the CWE coverage divergence as defensive design rather than regulatory mandate, because PCI 6.2.3 / SOC 2 CC7.1 / NIST SSDF accept tool output as evidence but no published audit-failure precedent fixes the mapped-but-not-scanned pattern as a documented requirement.**

## Defer the two-workflow split to v2

The GitHub Security Lab canonical pattern for SAST that needs to comment on fork PRs is a two-workflow split: `sast-analysis.yml` (`on: pull_request`, no secrets, uploads SARIF) plus `sast-comment.yml` (`on: workflow_run`, `permissions: pull-requests: write`, downloads artifact and posts comment). The wedge's audience — solo and small-team OSS maintainers — gets PR-review feedback for free via SARIF-to-Code-Scanning without any explicit comment. The two-workflow split doubles the YAML and only earns its keep when fork-PR traffic justifies a single bot-comment summary. **The design must handle PR feedback in v1 by uploading SARIF to GitHub Code Scanning (single workflow) and deferring the two-workflow split to v2, because Code Scanning surfaces findings in PR review without an explicit comment and the wedge's audience does not need the bot-comment path.**
