## Topic Decomposition

- **Q1a — Format inventory.** For each of {CycloneDX, OSCAL, OpenSSF Scorecard, SKILL.md, MCP capability descriptors, SPDX, SARIF}, what is the JSON/YAML shape, does it natively express "control → component" mapping, and what's the current tooling support (parsers, validators) that an LLM can rely on?
- **Q1b — Control-vs-finding distinction.** Which of those formats are *forward* declarations (capabilities a component provides) vs *backward* declarations (findings, VEX statements, vulns)? SARIF and OpenSSF Scorecard are finding-shaped; OSCAL `component-definition` and CycloneDX `components[].data.classification` + `services[].authentication` lean forward — needs verification which fields map cleanly to "this lib provides Argon2id".
- **Q1c — Authoring cost.** How much YAML/JSON does a Hulumi/SunLitSecureLibraries owner have to maintain per control to keep the manifest live? Rank by (lines per control) × (parser maturity).

- **Q2a — Audience composition.** What does public data say about SLO's likely user mix (solo devs / startups / EU vs US / regulated vs unregulated) — and what compliance obligations track each bucket? Anchor candidate: OWASP ASVS as the non-regulatory baseline, plus one regulatory pair (GDPR + SOC 2) that covers most web-service shipping.
- **Q2b — Unwieldiness threshold.** At what number of columns does the threat-model compliance table become noise? Is there prior art (NIST OSCAL profile resolution, OWASP CRE crosswalks) that collapses overlapping frameworks into a minimal spanning set?

- **Q3a — LLM+TM prior art taxonomy.** Separate *inputs* (DFD, narrative, code) and *outputs* (STRIDE table, ATT&CK chain, OTM JSON) across MITRE ATLAS, Microsoft TMT 2016 export, pytm, OpenThreatModel (OTM), IriusRisk, STRIDE-GPT (2024 GitHub), academic 2024–2026 papers. Mark which emit OTM JSON (the only candidate interchange format).
- **Q3b — Reproducibility/eval signal.** Of the LLM-TM publications, which ones measure precision/recall or human-vs-LLM agreement? Raw "agents can do STRIDE" anecdotes have low evidentiary value.

- **Q4a — Agent-invokable scanner matrix.** Per candidate {Semgrep, CodeQL, Sourcegraph, ast-grep, tree-sitter, ripgrep}: install surface, license for OSS vs commercial, SARIF output support, cold-start latency on 50k LOC, rule authoring cost, false-positive reputation from peer benchmarks.
- **Q4b — Variant-analysis UX.** Given a specific finding like "SQL string concatenation in module X", which tool reaches from one example to a codebase-wide sweep fastest? ast-grep/Semgrep both claim this; verify with real command.

- **Q5a — gh CLI attribution + scope.** Does `gh issue create` across repos require `public_repo` or `repo` scope on the user's PAT? Is authoring under the user's identity acceptable to upstream maintainers (Hulumi, SunLitSecureLibraries) — or does it need a labeled "filed via SLO" footer?
- **Q5b — Rate / spam floor.** REST v3 authenticated rate limit is 5000/hr at user granularity — needs verification that issue-create counts here and not under the more aggressive abuse-detection pipeline. What's the observed ceiling (anecdotal) for cross-repo issue filing before GitHub flags it?
- **Q5c — Alternatives.** For each of {fork+PR to a template file, mailto intake, RFC repo under the owner's org, Discussion category}, compare maintainer cost and agent friction.

## Key Questions

1. **Q1** — Which *one* manifest format should Hulumi + SunLitSecureLibraries adopt so `/slo-sec-libs` has a deterministic index? (OSCAL `component-definition` is the strongest candidate on content; CycloneDX has the stronger tooling ecosystem — this is the key tradeoff to resolve.)
2. **Q2** — Is the minimal default set `{OWASP ASVS L1/L2, SOC 2 CC-series, NIST 800-53 moderate baseline}` enough to cover ≥80% of SLO users, with GDPR/HIPAA/PCI as opt-in columns? Need survey data to defend ≥80%.
3. **Q3** — Is there an existing JSON/YAML threat-model interchange format (OTM is the leading candidate) that `/slo-threat-model` should emit so downstream tools (`/slo-critique`, `/slo-verify`) parse rather than re-LLM-read it?
4. **Q4** — Shortlist two scanners for `/slo-security-test`: the hypothesis to test is *Semgrep (rules-as-code, SARIF, fast) + ast-grep (structural, polyglot, no cloud)*, with CodeQL relegated to an optional CI-only tier because it's heavier and GitHub-Actions-coupled.
5. **Q5** — Produce the decision table for ≥3 filing channels (gh issue, fork+PR, Discussions). The tightest unknown is whether cross-repo issue-create triggers abuse detection at the frequency `/slo-sec-libs` would realistically hit (order-of-magnitude: a few per week, not per minute).

## Initial Findings

### Q1 — manifest formats

- **OSCAL** (NIST): YAML/JSON, has `component-definition` and `control-implementation` models that explicitly encode "this component implements control X from catalog Y". Designed precisely for "library provides control" claims. High content fit, medium tooling maturity (Rust parsers sparse — `oscal-cli` is Java). Confidence: **high** on model fit; **medium** on Rust-side integration cost.
- **CycloneDX** 1.6 (OWASP): JSON/XML. SBOM-first, but has expanded to `services[].authentication`, `components[].cryptoProperties`, VDR/VEX. `cryptoProperties` can express "AES-256-GCM / Argon2id" at component level. Very strong tooling (cyclonedx-rust, cyclonedx-bom). Does **not** natively express "implements control CIS-X" without extensions. Confidence: **high** on tooling, **medium** on control-mapping fit.
- **SPDX 3.0** adds `Security` profile including VEX statements but is weaker on forward capability claims. Confidence: **medium** — needs verification against SPDX 3.0 spec.
- **SARIF 2.1.0** (OASIS): JSON, finding-oriented (`runs[].results`). Right format for `/slo-security-test` *output*, wrong format for capability manifests. Confidence: **high**.
- **OpenSSF Scorecard**: computed score card (checks × scores), not a capability manifest. Confidence: **high** — orthogonal to this problem.
- **SKILL.md frontmatter** (agentskills.io / Anthropic Claude skills): YAML frontmatter at top of a markdown file, currently carries `name`, `description`, `allowed-tools`. No standardized security-controls schema. Trivial to extend but no prior art Claude agents already parse. Confidence: **high** on format, **low** on reuse value for this problem.
- **MCP server capability descriptors**: describe *tools* an MCP server exposes, not security controls. Misfit. Confidence: **high**.

### Q2 — compliance defaults

- **OWASP ASVS 4.0.3** is the closest thing to a non-regulatory control catalog designed for web apps; L1 is "minimum for anything public-facing", L2 is default, L3 is high-risk. Low adoption cost (one column, ~280 requirements). Confidence: **high**.
- **SOC 2 Type II** (AICPA Trust Services Criteria, CC-series) is the *de facto* US B2B SaaS baseline; widely cited in YC/SaaS survey data. Confidence: **high** on prevalence, **needs verification** on which survey gives the best citation for 2025/2026.
- **NIST 800-53** is heavyweight; FedRAMP/government only. Probably stays because SunLitSecureLibraries already cites it, but should be optional for SLO user baseline. Confidence: **medium**.
- **IEC 62443** is industrial control systems. Niche for a generic SLO user base; likely drop or mark "opt-in, OT users only". Confidence: **high**.
- **GDPR** applies to anyone with EU users — so for any web service, at least the data-handling subset is table-stakes. A *column* in the TM may be overkill; a *privacy section* in SECURITY.md is the better home. Confidence: **medium**, needs verification against Jim Manico's SECURITY.md pattern.
- **PCI DSS 4.0**, **HIPAA**, **FedRAMP**, **ISO 27001** are opt-in (user toggles a flag; the generator adds the column). Confidence: **high**.
- **CIS Controls v8** is a reasonable implementation-level crosswalk target; cheap to include as a single column mapped from ASVS via OWASP CRE. Confidence: **medium**.
- **Candidate minimal default set**: `OWASP ASVS (L2) + SOC 2 CC + NIST 800-53 moderate` — three columns, covers web-service shipping. Regulatory extras (GDPR/HIPAA/PCI/FedRAMP) are flags the user enables. **Confidence: medium — proposal, not confirmed by survey data.**

### Q3 — LLM-agent threat-modelling prior art

- **OpenThreatModel (OTM)** by IriusRisk: JSON schema covering project, assets, components, dataflows, trustZones, threats, mitigations. Open-source, designed as an interchange. Strong candidate for `/slo-threat-model` output format. Confidence: **high** on existence, **medium** on ecosystem depth.
- **pytm** (OWASP): Python-DSL → STRIDE table + DFD via graphviz. Deterministic. Not LLM-native, but LLM can *emit* pytm Python and run it. Confidence: **high**.
- **Microsoft Threat Modeling Tool 2016**: Windows-only, emits `.tm7` XML. Legacy. Confidence: **high**.
- **MITRE ATLAS**: adversarial ML threat catalog (for AI systems *as targets*), not a generic threat-modeling framework. Relevant for SLO only when the target app is an AI system. Confidence: **high**.
- **MITRE ATT&CK**: TTP catalog; useful as the *mapping target* for the "realized threat → technique" column in the TM table, similar to how SunLitSecureLibraries cites it. Confidence: **high**.
- **STRIDE-GPT** (GitHub, 2024) and academic literature: multiple 2024–2026 papers exist on "LLM-assisted threat modelling" with mixed precision/recall results. **Needs verification** — specific papers and their empirical findings have to be pulled in the deepening pass; claims should not be made without citations.
- Confidence: **medium** that OTM JSON is the right interchange; **low** until we confirm tooling activity (last release, issues, adopters).

### Q4 — variant analysis tooling

- **Semgrep** OSS: CLI-invokable, SARIF output, rules-as-code (YAML), strong polyglot coverage (Rust, TS, Python, Go…), community rule registry. Tens of seconds on 50k LOC typical. Confidence: **high**.
- **ast-grep**: Rust, MIT, tree-sitter based, structural pattern matching with a concise YAML rule format, very fast, SARIF output landed. Great fit for a Rust skill pack. Confidence: **high**.
- **CodeQL**: heavy (build-required for compiled languages, database-extraction step), GitHub-owned, license now permits OSS + research use but commercial restrictions exist; best as a GH-Actions tier, not a dev-loop tool. Confidence: **high**.
- **Sourcegraph OSS** path is shrinking — core moved to proprietary; self-host is heavier than Semgrep. Less attractive for an OSS skill pack. Confidence: **medium**, needs verification on current OSS status.
- **tree-sitter + custom queries**: building block for ast-grep; using it raw means hand-rolling a CLI. Skip unless ast-grep is unacceptable. Confidence: **high**.
- **ripgrep**: fallback tier — fast, structured JSON output (`--json`), zero rule semantics. Best for narrow regexes the agent generates on the fly post-finding. Confidence: **high**.
- **Hypothesis**: wire `Semgrep + ast-grep` into Phase 3 (`/slo-security-test`), keep ripgrep as the tier-0 fallback the agent already knows.

### Q5 — gh CLI filing

- `gh auth login` by default requests `repo` scope (full) — too broad. Fine-grained PATs or `gh auth login --scopes "public_repo"` narrow it. Confidence: **high**.
- **REST rate limit** authenticated: 5000 req/hr per user. Issue creation is covered. For `/slo-sec-libs` filing < 10 issues/week, no rate concern. Confidence: **high**.
- **Abuse/secondary rate limits**: GitHub does rate-limit issue creation aggressively if patterns look bot-like (many issues, same body, cross-repo) — documented but thresholds undisclosed. Confidence: **medium**.
- **Issue templates**: repos can enforce `.github/ISSUE_TEMPLATE/*.yml` with required fields; `gh issue create --template <name>` honors it, but non-interactive `--title --body` invocations bypass templates unless explicitly fetched. Confidence: **high**.
- **Attribution**: issue is created under the user's GitHub identity. Not "from SLO". Upstream maintainers (Hulumi/SLS = Sherif himself today) see their own user account opening cross-repo issues — fine for dogfood, ambiguous once SLO has external users. Mitigation: footer block `Filed via /slo-sec-libs vX.Y — (command used)` is cheap and transparent. Confidence: **high**.
- **Alternatives**:
  - **Fork + PR against `capabilities.yml`** — higher friction for SLO (shell-out + branch), lower friction for maintainer (structured change, reviewable). Confidence: **high**.
  - **Discussions**: low friction for a question ("does this lib do X?"), wrong medium for a tracked gap. Confidence: **high**.
  - **RFC repo** under the owner's org: adds a directory layer; useful only if SLS/Hulumi adopt one. Confidence: **medium**.
- Proposed decision table (to be firmed up in deepening): **gh issue create** = auth=medium, friction=low, maintainer-cost=medium (noise); **fork+PR** = auth=medium, friction=high, maintainer-cost=low; **Discussions** = auth=low, friction=low, maintainer-cost=high (untriaged surface).
