## Deepened Findings

### F1 — RustSec→GHSA→CWE coverage rate (refined, sample n=8 of 2024-2026 vuln-class advisories)
- **EVIDENCE (HIGH confidence)**: 8/8 sampled GHSA aliases returned populated `database_specific.cwe_ids`. Source: `https://api.osv.dev/v1/vulns/<GHSA>` for GHSA-x9xc-63hg-vcfq, GHSA-r8w9-5wcg-vfj7, GHSA-x3f4-45xf-rjm7, GHSA-h97m-ww89-6jmq, GHSA-g433-pq76-6cmf, GHSA-2c6h-4899-wjxr, GHSA-hfr4-7c6c-48w2, GHSA-965h-392x-2mh5.
- **EVIDENCE (HIGH)**: The RUSTSEC-side OSV record carries **no** `cwe_ids` field — even when the GHSA alias has them. The previous round's two-hop join (RUSTSEC → `aliases[]` → GHSA-OSV → `cwe_ids`) is required, **not** a convenience.
- **INFERENCE (MEDIUM)**: For RustSec advisories with `informational: vulnerability` (i.e. real CVE-class, not unmaintained-crate warnings), expect ~95%+ GHSA-side CWE coverage based on this sample. Residual gap will be the GHSA-less subset (informational advisories that never produced an upstream CVE).

### F2 — Refined CWE frequency-in-the-wild ranking (8-advisory sample, 2024-2026)
Confidence: **MEDIUM** — sample size is small but skewed toward recent / high-severity advisories. Multiple-hit CWEs in this slice:
- **CWE-416 (use-after-free)** — 3 hits (cassandra-rs, mio, wasmtime). Confirms F2 promotion: this should be **top-3** in the pack scope, not absent from user's prior top-7.
- **CWE-697 (incorrect comparison)** — 2 hits (idna, hpke-rs). **Net-new vs. user's prior list** — surfaces in real web code (idna's Punycode comparison flaw, RUSTSEC-2024-0421).
- **CWE-125 (OOB read)** — 2 hits (ruzstd, scaly).
- Single hits in this slice: **CWE-787, CWE-190, CWE-295, CWE-1289, CWE-20, CWE-672**.
- **CWE-248 panic-DoS is not natively tagged**; the 2026 precedent (Trail of Bits) uses **CWE-755 (improper handling of exceptional conditions)** instead.

### F3 — Trail of Bits `panic-in-function-returning-result.yaml` is the gold-standard precedent for the wedge
- **EVIDENCE (HIGH)**: file exists at `https://raw.githubusercontent.com/trailofbits/semgrep-rules/main/rs/panic-in-function-returning-result.yaml` with paired test fixture `panic-in-function-returning-result.rs` (3,676 bytes).
- The rule enumerates **4 fn-context variations** in a single `pattern-either`: direct `Result<T1, T2>`, direct `Result<T>`, type-alias to `Result<T1, T2>`, type-alias to `Result<T>`. Plus **3 `pattern-not-inside`** exclusions for `#[cfg(test)] mod tests { ... }` blocks.
- **Implication for the wedge**: the "3-5 variation rules per bug" target is achievable **inside one Semgrep rule's `pattern-either`** — variation enumeration is a YAML structural concern, not a multi-file concern. This simplifies the corpus layout decision (one rule, one fixture, one `// ruleid:` annotation).
- Metadata uses `cwe: "CWE-755: Improper Handling of Exceptional Conditions"` (not CWE-248). Adopt CWE-755 as primary tag with `category: security`, `subcategory: [audit]`, `confidence: HIGH`, `likelihood: MEDIUM`, `impact: LOW`.

### F4 — Upstream Semgrep `rust/lang/security/` is exactly 10 rules + 10 fixtures
- **EVIDENCE (HIGH)**: tree API at `https://api.github.com/repos/semgrep/semgrep-rules/git/trees/develop?recursive=1` returned 20 blobs in `rust/`: `args-os`, `args`, `current-exe`, `insecure-hashes`, `reqwest-accept-invalid`, `reqwest-set-sensitive`, `rustls-dangerous`, `ssl-verify-none`, `temp-dir`, `unsafe-usage` — each a `.yml` + `.rs` pair in the same directory (no `tests/` subdir, no per-rule subdir).
- **CWE overlap with the wedge**: CWE-242 (unsafe-usage), CWE-295 (rustls-dangerous, ssl-verify-none, reqwest-accept-invalid), CWE-327 (insecure-hashes presumed), CWE-78/CWE-88 (args/args-os). **Zero overlap with CWE-755/CWE-248/CWE-190/CWE-416/CWE-125/CWE-787** — the wedge is genuinely net-new content, not a duplication of the upstream pack.

### F5 — `// ruleid:` on `.rs` files is confirmed by the upstream pack itself
- **EVIDENCE (HIGH)**: all 10 upstream Semgrep Rust fixtures are tracked `.rs` files using `// ruleid:` annotations — same convention semgrep-rules uses for every other language. No Rust-specific quirks; `--test` accepts `//` comments verbatim. Source: directory listing above + `https://semgrep.dev/docs/writing-rules/testing-rules`.
- Convention is `path/to/<rule-id>.yaml` + `path/to/<rule-id>.rs` co-located in the same directory; the runner discovers fixtures automatically based on `languages:` field.

### F6 — `pattern-inside: unsafe { ... }` smoke-test status
- **INCONCLUSIVE → likely supported**: upstream `unsafe-usage.yml` uses bare `pattern: unsafe { ... }` (not `pattern-inside`). No public example uses `pattern-inside: unsafe { ... }`. Documentation does not restrict it. **Smoke-test remains required** before templating in `references/sast/`. Single-rule fallback if it fails: nest `pattern-inside: fn $F(...) { ... unsafe { ... } ... }` or use a `metavariable-pattern` over the function body.

### F7 — `metavariable-type` on Rust generics — docs are silent, but the wedge does not need it
- **EVIDENCE (HIGH)**: Rust is listed as supported on `https://semgrep.dev/docs/writing-rules/experiments/metavariable-type`, but the page shows no Rust example and the linked metavariable-type docs page shows only Java.
- **EVIDENCE (HIGH)**: the Trail of Bits CWE-755 precedent does **not** use `metavariable-type` — it uses purely structural `pattern-inside fn ... -> Result<$T1, $T2> { ... }`. Result-type discrimination falls out of the structural pattern.
- **Implication**: drop `metavariable-type` from the v1 rule template requirements. If a future class needs trait-bound matching, fall back to `metavariable-pattern` + regex (matches the previous round's recommendation).

### F8 — `prek` 0.3.10 is a real, MIT-licensed alternative now in production at CPython and Apache Airflow
- **EVIDENCE (HIGH)**: latest release `0.3.10` shipped 2026-04-21; license MIT; documented as fully compatible with `.pre-commit-config.yaml`. Sources: `https://github.com/j178/prek` (release page) and project README.
- **CAVEAT**: still pre-1.0; some language hook types unsupported. Default to `pre-commit` (Python, canonical) and document `prek` as a drop-in alternative.

### F9 — GitHub AUP carves out educational vulnerability content explicitly
- **EVIDENCE (HIGH)**: `https://docs.github.com/en/site-policy/acceptable-use-policies/github-active-malware-or-exploits` — quoted policy: *"GitHub allows dual-use content and supports the posting of content that is used for research into vulnerabilities, malware, or exploits, as the publication and distribution of such content has educational value."*
- **Implication**: the conservative `.gitignore`'d-corpus default in the idea doc is over-cautious. The defensible-and-portable default is **tracked vulnerable snippets co-located with rules, README disclaimer, repo-level `SECURITY.md`** — exactly what semgrep-rules itself ships. This unlocks `semgrep ci` portability.

## Library Evaluations

### Trail of Bits `semgrep-rules` (CWE-755 panic precedent)
- **Version**: live, `main` branch, accessible at `https://github.com/trailofbits/semgrep-rules/tree/main/rs`.
- **License**: AGPL-3.0 (Trail of Bits semgrep-rules repo) — **vendoring concern**. Inspecting / mirroring patterns is fine; copy-pasting YAML wholesale into a permissively-licensed pack would inherit AGPL.
- **Pros**: only public production-grade Rust panic-DoS rule; demonstrates the variation-enumeration shape (`pattern-either` + `pattern-not-inside` test-block exclusions); validates the wedge's CWE-755 framing.
- **Cons**: AGPL forces a re-implementation rather than a vendor; only one panic-class rule (no integer-overflow, no UAF).
- **Fit**: **reference precedent only — re-author rules independently** to keep the project pack's license clean.

### Upstream `semgrep/semgrep-rules` Rust pack (10 rules)
- **Version**: continuously updated `develop` branch.
- **License**: Semgrep Rules License.
- **Pros**: defines the layout convention (paired `.yml` + `.rs` co-located); confirms `// ruleid:` annotation is the active discovery mechanism; covers crypto/TLS/process-arg classes the wedge does NOT need to re-do.
- **Cons**: no overlap with the wedge's panic/integer/memory CWE classes — the wedge is net-new content, not an extension of upstream.
- **Fit**: **layout-convention reference + complementary install** (the wedge ships ON TOP of upstream's 10 rules, not instead of).

### Trail of Bits panic rule's variation-enumeration shape
- **Status**: gold standard for "5 variations per CWE in one rule".
- **Pros**: variation enumeration lives inside `pattern-either` (one rule, not five files). `pattern-not-inside` is the canonical exclusion mechanism for `#[cfg(test)]` blocks. Lowers the corpus-layout cost: 1 rule = 1 fixture file with 5 inline `// ruleid:` annotations.
- **Cons**: `pattern-either` blocks scale with the number of credible idioms — for CWE-416 (UAF) the credible idioms span concurrency primitives, raw pointers, and `Pin` misuse; one rule may not cover all without taint mode.
- **Fit**: **adopt as the default rule shape** for v1; reach for taint mode only when structural enumeration breaks down.

- **Cons**: pre-1.0; partial language-hook parity.

## Architecture Options

(Carrying forward the four options surfaced last round; consolidating against the new findings.)

### Option α — Skill-only (no Rust crate, no xtask)
- **Trade-off**: ships fastest (~1 week), zero new Cargo infra, matches `slo-*` skill-pack pattern. Verifier gate is *prompted Bash sequence* — `semgrep --validate` and `semgrep --test` exit codes are the enforcement, but variation-coverage minimum is policy not code. No deterministic backstop against Claude skipping a step.
- **Best fit**: ship-speed wins; user accepts skill-discipline + canonical Semgrep tooling as the gate.

### Option β — Hybrid skill + `xtasks/sast-verify/` workspace member
- **Trade-off**: deterministic gate (variation-coverage minimum returns non-zero exit), reuses workspace deps, but requires net-new `.cargo/config.toml` + `xtasks/` directory + appending to `CLAUDE.md`'s baseline test list. Scope-creep risk flagged in the idea doc.
- **Best fit**: variation-coverage-minimum determinism > ship-speed.

### Option γ — Hybrid skill + `crates/sldo-sast/` first-party crate
- **Trade-off**: deepest backstop (advisory-db ingestion, GHSA-CWE join scheduled, snippet-corpus management). ~3-4 weeks. Couples the bug-found→rule-extended loop to a `cargo build` cycle — directly opposes the user's "fix the regression now" pain.
- **Best fit**: only if the project commits to a recurring advisory-db ingestion pipeline (i.e. `cwe-map-rust.md` is data-driven and auto-refreshed, not hand-curated).

### Option δ — Skill-only + upstream Semgrep `--test` runner in CI
- **Trade-off**: minimum surface area; matches upstream convention exactly (tracked corpus + `// ruleid:` + `semgrep --test`). The variation-coverage minimum becomes a Bash check in the skill, but the *correctness gate* is canonical Semgrep tooling. No in-repo deterministic fail-fast.
- **Best fit**: portability of the generated pack > in-repo gate. **Now reinforced by F9** — tracked corpus is GitHub-AUP-safe and matches upstream layout, so the generated pack runs under any consumer's `semgrep ci` with no rewriting.

### Consolidation note
The new findings push the recommended default toward **δ** (with **β** as a hardening upgrade if the variation-coverage-minimum becomes important enough to justify net-new Cargo infra). The shape `1 rule = 1 `pattern-either` with N variations = 1 paired `.rs` fixture with N `// ruleid:` annotations` (F3, F5) means the corpus-layout decision is settled and the gate can be the canonical Semgrep `--validate` + `--test` pair, no custom verifier required for v1.
