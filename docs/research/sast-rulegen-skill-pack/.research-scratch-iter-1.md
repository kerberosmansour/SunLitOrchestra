## Topic Decomposition

- **SQ1 — RustSec/OSV CWE distribution (2024-04 → 2026-04).** What does a count-by-CWE of the last 24 months of `RustSec/advisory-db` (and OSV.dev `ecosystem:crates.io`) actually look like? RustSec uses its own taxonomy (`memory-corruption`, `memory-exposure`, `crypto-failure`, `thread-safety`, `denial-of-service`, `format-injection`, …) plus optional `aliases` to CVEs/CWEs — counting requires a join, not a single query. Rationale: this is the only frequency-in-the-wild input that drives the top-10.
- **SQ2 — Where idiomatic-safe-Rust panics live (the "Rust safe ≠ no crash" gap).** Which sink shapes recur in non-`unsafe` Rust advisories: `unwrap`/`expect`/`?` on attacker-influenced input, slice indexing, integer overflow in capacity/length, recursion/stack-overflow on attacker-shaped input, denial-of-service via decompression / regex / serde untagged-recursive. Rationale: drives the per-CWE "why Rust is still susceptible" paragraph and the `references/sast/variations/` template set.
- **SQ3 — Webapp-shape CVEs in axum/actix/hyper/sqlx/reqwest/serde.** What classes (auth bypass, SQL injection via string-format on `sqlx::query`, SSRF via reqwest with attacker URL, missing output encoding, header-smuggling in hyper/h2) actually show up in advisories vs. only in audit reports? Rationale: justifies including web classes (CWE-89, CWE-79, CWE-918) in a *Rust* top-10 even though the language doesn't make them more likely.
- **SQ4 — Direct competitors in LLM-driven SAST rule generation.** What are the 2026 product and OSS positions for "bug → rule" auto-generation: Semgrep AI rule suggestion, Snyk Code custom rules + AI fix, GitHub Code Scanning custom pattern queries (CodeQL) + Copilot Autofix, SonarQube AI CodeFix, Veracode Fix, plus OSS projects like `kerberosmansour/SAST.GEN` and any LangChain/Claude-based rule synthesisers? Rationale: the gap table justifies the wedge.
- **SQ5 — Semgrep Rust frontend capabilities at HEAD (2026).** Which primitives actually work on Rust today: `pattern-either`, `pattern-inside`, `pattern-not`, `metavariable-pattern`, `metavariable-type`, taint mode (`mode: taint` with `pattern-sources`/`pattern-sinks`/`pattern-sanitizers`), `comby`/generic mode fallback, macro expansion behaviour for `#[tokio::main]`/`#[axum::handler]`/`tracing::instrument`. Rationale: rule templates must use only working primitives; rules using broken primitives become CI noise.
- **SQ6 — Adjacent Rust security tools and the integrate-vs-replace decision.** For each of `cargo-audit`, `cargo-deny`, `cargo-vet`, `cargo-geiger`, `clippy` (security-leaning lints), `miri`, `kani`, `prusti`, `creusot`, `MIRAI`, `rudra`, what is the overlap with `/slo-rulegen`? Specifically: should `/slo-rulegen` *consume* clippy JSON output (`cargo clippy --message-format=json`) instead of writing a Semgrep rule for `unwrap_used`? Rationale: avoids duplicate-tooling churn and decides whether the pack's `unwrap_used` rules are redundant.
- **SQ7 — Corpus / test layout convention across rule-pack maintainers.** How do `returntocorp/semgrep-rules`, `trailofbits/semgrep-rules`, `0xdea/semgrep-rules`, and Snyk's public rule packs lay out `(bad, good)` pairs — per-rule `.test.yaml` co-located, single `tests/` directory, inline `metadata.examples`, `// ruleid:`/`// ok:` annotation comments? Rationale: defines the schema `xtasks/sast-verify/` reads.
- **SQ8 — CI and dev-loop entry points for Semgrep on a Rust repo (2026).** Canonical CI: `returntocorp/semgrep` GitHub Action, Semgrep CI in GitLab, BuildKite plugin. Local: `pre-commit` hooks (Python-based), `lefthook`, `cargo-husky`, `rusty-hook`, `cargo-make` integration, IDE plugins (Semgrep VS Code, Semgrep Cursor extension). Rationale: drives M3 wiring and the question "is `pre-commit` still canonical for Rust repos?".
- **SQ9 — Publication-risk precedent for vulnerable-snippet corpora.** How does `returntocorp/semgrep-rules` ship deliberately-vulnerable test files (header comments, README disclaimer, branch policy)? Has any public rule-pack repo been actioned under GitHub's Acceptable Use Policy "active malware or exploits" clause? What does the AUP text actually require for educational vulnerability content? Rationale: anchors the `.gitignore`-by-default vs. tracked decision.
- **SQ10 — `cargo xtask` precedent for security-leaning verifiers.** Which large Rust projects (rust-lang, tokio, axum, ripgrep, clap, async-graphql, etc.) ship a security-themed xtask? How do they layer it under `cargo xtask <verb>` and what's the mainstream skeleton (sub-crate vs. top-level workspace member)? Rationale: locks the skeleton for `xtasks/sast-verify/`.

4. **SQ6.** Clippy integration decision. If `clippy::unwrap_used` + `clippy::indexing_slicing` + `clippy::arithmetic_side_effects` already cover ≥ 60% of CWE-248/CWE-190 sink shapes, `/slo-rulegen` should compose with clippy rather than re-implement; this changes M1 scope significantly.
5. **SQ9.** Publication-risk default. 7. **SQ8.** CI + local wiring. Lowest priority because it's M3 of Runbook A and only minor design variance is plausible (`pre-commit` vs. `lefthook`).
8. **SQ3.** Web-class inclusion in a Rust top-10. Either confirms or rejects CWE-89/CWE-79/CWE-918 in the top 10; needed for cwe-map but research-cheap once SQ1 is done.

## Initial Findings

> Confidence labels: **(H)** high — well-known and stable, **(M)** medium — believed correct but worth a deepening pass, **(L)** low / needs verification.

### CWE distribution and the "Rust safe ≠ no crash" gap (SQ1, SQ2)

- **(H)** RustSec advisory-db at `github.com/RustSec/advisory-db` uses its own `categories` taxonomy (`memory-corruption`, `memory-exposure`, `thread-safety`, `crypto-failure`, `denial-of-service`, `format-injection`, `code-execution`, `privilege-escalation`, `file-disclosure`, `directory-traversal`) defined in `osv-schema/`-aligned TOML. Many advisories carry an `aliases` field with a CVE ID; CWE mapping is *via* the CVE, not directly in the advisory. Counting by CWE requires the CVE→CWE join.
- **(H)** OSV.dev exposes the same advisories with `database_specific.cwe_ids` populated when the upstream CVE has a CWE; OSV API `POST /v1/query` with `package.ecosystem=crates.io` returns the JSON. Coverage of the CWE field on Rust ecosystem advisories is partial — many entries lack CWE — so a hybrid count (RustSec `categories` count + OSV CWE count where present) is more honest than either alone.
- **(M)** Empirically, `denial-of-service` is the most common RustSec category over the last several years (lots of `panic-on-malformed-input` advisories in serde-adjacent and parsing crates), followed by `memory-corruption` and `memory-exposure` (heavily concentrated in `unsafe`-heavy crates: `bytes`, `tokio`, `actix-*`, `nix`). Verifiable via a script that tallies `categories = ` lines in advisory TOML files. **Needs verification — exact counts.**
- **(M)** The `cargo` blog and several conference talks have made the explicit point: Rust eliminates memory-unsafety bugs but does *not* eliminate panic-on-attacker-input. The Rustonomicon's "What Unsafe Rust Can Do" chapter is the canonical source. The user's prior expectation that **CWE-248 (uncaught exception / panic-DoS), CWE-770 (resource exhaustion), CWE-190/191 (integer overflow in capacity/length), CWE-119/787 (unsafe FFI / OOB write)** dominate is consistent with the high-confidence shape of the data; the precise top-10 ranking needs the join.
- **(M)** Web-class CVEs in Rust ecosystem (CWE-89 SQL injection, CWE-79 XSS, CWE-918 SSRF, CWE-352 CSRF, CWE-22 path traversal) appear primarily in axum/actix/hyper/sqlx/reqwest advisories. They are present but lower-frequency than panic-DoS in the absolute count. Including them in a Rust top-10 is justified by the user's stated pain ("missing input sanitisation, missing output encoding"), not by a frequency tie-break.

### Direct competitors (SQ4)

- **(H)** **Semgrep AI / Semgrep Pro Engine.** Semgrep ships LLM-assisted rule generation as part of Semgrep Pro / Semgrep Code in 2024-2026. The product surface is a SaaS dashboard plus IDE plugin; rules are human-editable YAML. Pricing is per-developer SaaS (publicly listed tiers are around $40/dev/month for Code, with AppSec tiers higher). **(L)** It is unclear whether the rule-generation flow includes corpus-driven fire-on-bad/silent-on-good gating before publishing. Needs verification on the deepening pass.
- **(H)** **Snyk Code (DeepCode AI).** Snyk Code is symbolic-AI-based, not rule-based; it ships findings and AI fixes but does not produce inspectable Semgrep-shaped rules. Custom rules are supported but the auto-generation flow is narrower than Semgrep's. Pricing is per-developer SaaS.
- **(H)** **GitHub Code Scanning + Copilot Autofix.** Code Scanning runs CodeQL queries (a different language than Semgrep). Copilot Autofix generates *fix patches*, not new queries. Custom CodeQL queries are author-it-yourself; there is no "bug → query" auto-generator in GHAS as of 2026 (verify in deepening). Free for public repos; GHAS billed per active committer for private.
- **(M)** **SonarQube AI CodeFix.** Generates fixes, not new rules. Rule authoring remains manual.
- **(M)** **Veracode Fix.** Fix-suggestion AI, not rule-generation. Pricing is enterprise/custom.
- **(L)** **`kerberosmansour/SAST.GEN`** — the user's own 2024 prior art. Repo exists per the prompt; what it implemented and how it gated rules is what `/slo-research` should pull from the README. **Needs verification.**
- **(L)** Open-source LLM rule generators on GitHub: searches like "semgrep rule generator LLM" historically surface a handful of demos but nothing that integrates as a Claude Code skill with corpus-gating. **Needs deepening.**

### Adjacent Rust security tools (SQ6)

- **(H)** **cargo-audit** (RustSec): consumes `RustSec/advisory-db`, reports CVEs against `Cargo.lock`. **Integrate, not replace** — `/slo-rulegen` runs after a fixed bug; cargo-audit catches *known* vulnerabilities in deps. Different layer.
- **(H)** **cargo-deny**: license, advisory, and source policy enforcement. Adjacent; user's own project should already have it.
- **(H)** **cargo-vet** (Mozilla): supply-chain trust audit. Adjacent.
- **(H)** **cargo-geiger**: counts `unsafe` blocks. Adjacent / advisory only.
- **(H)** **clippy** ships several security-leaning lints: `clippy::unwrap_used`, `clippy::expect_used`, `clippy::indexing_slicing`, `clippy::arithmetic_side_effects`, `clippy::panic`, `clippy::integer_arithmetic` (deprecated; replaced by `clippy::arithmetic_side_effects`), `clippy::missing_panics_doc`, `clippy::unimplemented`, `clippy::todo`. **High overlap with CWE-248 and CWE-190 sink shapes.** The integrate-vs-replace decision is genuine: the pack could either (a) generate Semgrep rules that duplicate clippy, or (b) declare a clippy-config dependency and only generate Semgrep rules for sinks clippy can't express (e.g. `serde_json::from_str` followed by `.unwrap()` *in a request handler*, which is a context-sensitive pattern Clippy doesn't model). Option (b) is the honest answer; needs verification of clippy's exact 2026 lint coverage.
- **(M)** **miri** (interpreter for UB), **kani** (model checker via CBMC), **prusti** (Viper-based verifier), **creusot** (deductive verifier), **MIRAI** (abstract interpreter), **rudra** (unsafe Rust analyser). Heavyweight verification tools — orthogonal to a Semgrep-rule pack. Ignore for `/slo-rulegen` scope.
- **(M)** **`cargo xtask` precedent for security**: tokio, ripgrep, and several other large Rust projects ship `xtask/` crates. Specifically security-themed xtasks are less common but precedents exist (e.g. `tokio` has audit/release xtasks). **Needs verification of named projects.**

### Semgrep Rust frontend reality (SQ5)

- **(H)** Semgrep's Rust support reached "GA" some time around 2023-2024. `pattern`, `pattern-either`, `pattern-not`, `pattern-inside`, `metavariable-regex` work. Macro support is an ongoing weakness — Semgrep parses Rust without macro expansion, so rules over macro-generated code (axum's `#[handler]`, tokio's `#[tokio::main]`, `tracing::instrument`) match the *syntactic* form before expansion, not the expanded form. This is the documented FP/FN trade-off.
- **(M)** Taint mode for Rust: Semgrep documents taint-mode for many languages; Rust support has been added but coverage of source/sink declarations on Rust generics and trait-bound metavariables has been a friction point in the issue tracker. **Needs verification of 2026 status.**
- **(L)** `metavariable-type` for Rust generics and trait bounds: support is partial; type inference from let-bindings is limited compared to languages with simpler type systems. **Needs verification.**
- **(M)** `pattern-inside unsafe { ... }`: should work because `unsafe` blocks are syntactic. **Verify with a small test.**
- **(H)** `semgrep --validate <rule.yaml>` is the syntactic gate the verifier xtask should call before any fire/silent assertion.
- **(M)** Existing rust rules in the public registry (`semgrep.dev/r`, `semgrep/semgrep-rules` repo `rust/` directory) are sparse compared to JS/Python. The number of rules and their topical coverage gives a baseline for what's already in the wild. **Needs counting.**

### Corpus layout convention (SQ7)

- **(H)** `returntocorp/semgrep-rules` uses **inline-annotation-in-the-test-file** convention: per-rule, a file like `<rule-id>.rs` (or `.py`, `.js`, etc.) sits next to `<rule-id>.yaml`. Lines that should match are marked with `// ruleid: <rule-id>`; lines that should not match with `// ok: <rule-id>`. The runner is `semgrep --test --config <rule.yaml>`. **(H)** This is the dominant convention across `trailofbits/semgrep-rules`, `0xdea/semgrep-rules` (verify), and the official semgrep docs ("Testing rules" page).
- **(M)** Co-located `.test.yaml` per-rule and a `tests/` mono-dir are both seen but less common than the inline-annotation convention. Inline `metadata.examples` is rare and discouraged for non-trivial rules.
- **Implication:** `xtasks/sast-verify/` should default to the `// ruleid:` / `// ok:` inline-annotation format and shell out to `semgrep --test`. This avoids inventing a new schema and makes the generated pack portable to any Semgrep CI.

### CI and dev-loop entry points (SQ8)

- **(H)** **Canonical CI** for Semgrep on GitHub: the `returntocorp/semgrep` GitHub Action (or `semgrep ci` in a workflow). Pricing-free for OSS scans, paid Semgrep Cloud for centralised findings. GitLab has first-class Semgrep templates. BuildKite has a community plugin.
- **(H)** **Canonical local-dev**: `pre-commit` framework (`pre-commit-hooks`-style YAML in repo root) is still the dominant cross-language hook framework in 2026; Semgrep ships a pre-commit hook spec (`https://github.com/returntocorp/semgrep` repo's `.pre-commit-hooks.yaml`).
- **(M)** **Rust-native equivalents** to `pre-commit`: `cargo-husky` and `rusty-hook` exist but are smaller communities; `lefthook` (Go-based, language-agnostic) is gaining adoption in Rust shops. **No Rust-native hook framework has displaced `pre-commit` for cross-tool hooks**, even in `cargo`-shaped repos. Needs verification of `lefthook`'s 2026 adoption.
- **(M)** IDE plugins: VS Code Semgrep extension exists; Cursor and IntelliJ-Rust integration via Semgrep LSP. Rust-analyzer integration with Semgrep is not native.

### Publication-risk precedent (SQ9)

- **(H)** `returntocorp/semgrep-rules` is a public GitHub repo that ships **hundreds of deliberately-vulnerable test files** (`security/*/test.py`, `security/*/test.js`, `security/*/test.go`, etc.). Files use the `// ruleid:` annotation convention; many include short header comments stating the file is a test fixture. The repo has not been actioned by GitHub. The README does not contain a strident "this is intentionally vulnerable code" disclaimer beyond the rules' own metadata.
- **(M)** GitHub Acceptable Use Policy section "Active Malware or Exploits" prohibits content that *delivers* malicious payload or *attacks GitHub or other GitHub users*. Educational content (CTF challenges, security training repos like `OWASP/WebGoat`, `digininja/DVWA`, `vulhub/vulhub`) has consistently been allowed for years. **Precedent strongly favours tracked, labelled snippets** for an OSS rule pack.
- **(L)** US EAR 5D002 / "intrusion software" controls — the user's idea doc raises this. Enforcement on snippet collections is essentially zero; the precedent is that audit-tool source code (Metasploit, Nuclei, Semgrep itself) is publicly distributed. **The breach mitigation in the idea doc remains a defensible default** — `.gitignore` for the founder's repo (because that repo is not a security-tooling repo and shipping vuln snippets there is a finding) — even if the publication-risk precedent for *the rule-pack repo itself* favours tracked.
- **Implication for the design:** the snippet output dir for a generated rule pack inside a *user's* application repo defaults to `.gitignore`'d (compliance-finding mitigation); the snippet output dir for the *rule-pack repo* (this project's `references/sast/` examples) can be tracked and labelled, following `semgrep-rules` precedent.

### `cargo xtask` precedent (SQ10)

- **(H)** The `xtask` pattern is documented at `github.com/matklad/cargo-xtask`; canonical layout is a top-level `xtask/` crate (or `xtasks/<name>/` for multiple) added as a workspace member. Invoked via `cargo xtask <verb>`. `rust-lang/rust-analyzer` itself uses the pattern.
- **(M)** Adoption: `tokio`, `ripgrep`, `clap`, `wasmtime`, `bevy` and many large crates ship xtask crates for release/codegen/lint orchestration. Security-specific xtasks (e.g. running cargo-audit + cargo-deny in one command) are seen but not universal. **Verifiable list to be confirmed in deepening.**
- **CLAUDE.md constraint applies:** adding `xtasks/sast-verify/` to the workspace expands the `cargo test -p ...` baseline list (since `--workspace` is parked). The runbook must explicitly add `-p sast-verify` (or whatever the crate is named) to `CLAUDE.md`'s baseline command.
