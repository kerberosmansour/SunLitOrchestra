# Synthesis — what the research means for the design

Each paragraph ends with the load-bearing design rule and its source.

## The CWE map shifts from theoretical to data-driven

The idea doc's prior expectation was a top-7 anchored on panic-DoS (CWE-248), integer overflow (CWE-190), and unsafe-FFI (CWE-119/787). The two-hop RustSec → GHSA-OSV `cwe_ids` join over a 2024–2026 sample (n=8) produces a different ordering: **CWE-416 (use-after-free), CWE-697 (incorrect comparison), CWE-125 (OOB read), CWE-787 (OOB write), CWE-190 (integer overflow), CWE-295 (cert validation), CWE-1289 (improper validation of specified entity input), CWE-20 (improper input validation), CWE-672 (operation on resource after expiration/release)**, with CWE-755 (Improper Handling of Exceptional Conditions) imported from Trail of Bits' production rule as the canonical tag for the panic-DoS class. CWE-416 and CWE-697 were entirely absent from the user's prior expectation. The design must **publish `references/sast/cwe-map-rust.md` driven by this 2-hop join with category-only fallback for the GHSA-less informational/unmaintained tail**, because RustSec's own TOML schema has no `cwe` field and OSV does not natively expose `cwe_ids` for Rust advisories (RustSec advisory-db schema; google/osv.dev issue #3245).

## CWE-755 replaces CWE-248 as the panic-DoS tag

The user's idea doc refers to "panic-DoS" as CWE-248 / CWE-1284 / CWE-770. Trail of Bits' `panic-in-function-returning-result.yaml`, the only public production-grade Rust panic-DoS rule, tags the class as **CWE-755 — Improper Handling of Exceptional Conditions**. RustSec/GHSA do not natively tag panic-DoS at all; CWE-755 is the operational consensus tag. The design must **standardise the panic-DoS class on CWE-755 across `references/sast/` and the rule metadata**, because the canonical industry precedent for this class uses it (https://github.com/trailofbits/semgrep-rules).

## Web-class CWEs (CWE-79/CWE-89/CWE-918) need an explicit hybrid policy

Searches for axum/sqlx-specific SQLi/XSS/SSRF in 2024–2025 returned no major framework-level advisories. RUSTSEC-2024-0421 (idna/Punycode confusion in `url`/`sqlx`/`reqwest`) maps to CWE-20/CWE-1007 — phishing/SSRF-adjacent, not classic SQLi or stored XSS. The user's stated pain explicitly named XSS and missing input sanitisation, so frequency-only ranking would drop classes the user feels in production. The design must **adopt a hybrid ranking policy: top-10 = top-N frequency-confirmed CWEs ∪ user-pain-anchored web-class CWEs (CWE-79, CWE-20)**, with each entry in `cwe-map-rust.md` flagged `frequency-confirmed` or `threat-class-reasoned`, because both signals are valid but they trade off and the rule pack's user must see why a CWE is in the map (no major Rust framework SQLi/XSS advisory surfaced 2024–2025).

## Architecture default flips toward Option δ but the idea-doc course-correction holds at Option β

The deepening pass surfaced four architecture options and recommended **Option δ — skill-only + upstream `semgrep --test` runner in CI** as the v1 default, on the basis that the corpus convention is settled (paired `<rule>.yaml` + `<rule>.rs` co-located with `// ruleid:`/`// ok:` annotations, run by canonical `semgrep --validate` then `semgrep --test`). The user's idea-doc course-correction committed to **Option β — skills + `xtasks/sast-verify/` workspace member** for repo consistency with `sldo-research`/`sldo-install` and to enforce the variation-coverage minimum as a hard gate. The design must **adopt Option β with a minimal xtask body (~50–150 LOC) wrapping `semgrep --validate` + `semgrep --test` plus a YAML-parse-and-count check that every rule's `pattern-either` enumerates ≥ N variation arms (N from `references/sast/variations/cwe-XXX.md`)**, because the variation-coverage hard gate is the breach-mitigation enforcement mechanism in the idea doc and prompt-policed Bash sequences are skippable (idea-doc breach risk; Semgrep --validate exit codes 5/7/4 stable).

## Semgrep Rust frontend has known gaps the rule templates must avoid

Semgrep Rust GA in 2026 has working `pattern-either`, `pattern-inside`, `pattern-not`, taint mode (used in Semgrep's own SCA reachability for Rust), and macro-arg taint propagation since July 2023. Confirmed gaps: proc-macro expansion (`#[axum::debug_handler]`, `#[tokio::main]`, `#[tracing::instrument]`-decorated handlers do not match through the macro — issues #10471, #10362, #3600, #5221); taint through `format!` and intermediate `let` (issues #10757, #10900); `metavariable-type` semantics on Rust generics under-documented (issues #10380, #11150). `pattern-inside: unsafe { ... }` has no public example and is unverified. The design must **constrain the v1 rule templates to use only structural primitives (`pattern-either`, `pattern-not-inside`) plus basic taint sources/sinks, fall back to `metavariable-pattern` + regex over `metavariable-type` for trait-bound discrimination, and include a smoke-test for `pattern-inside: unsafe { ... }` as part of M1 before any unsafe-FFI rule is templated**, because the precedent rule (Trail of Bits `panic-in-function-returning-result`) achieves Result-type discrimination structurally without `metavariable-type` and the proc-macro / format!-taint gaps would produce false-negatives in templated rules (Semgrep open issues; Trail of Bits production rule shape).

## Corpus layout and gate are settled — re-implementing them is wasted effort

Co-located paired `<rule>.yaml` + `<rule>.rs` with inline `// ruleid:` (fire-on-bad) and `// ok:` (silent-on-good) annotations is the **uniform convention** across `semgrep/semgrep-rules` (3,966+ commits), `trailofbits/semgrep-rules`, `0xdea/semgrep-rules`. `semgrep --test` is the canonical runner (semgrep issue #2799). `semgrep --validate` exit codes are stable (5 bad config, 7 invalid rule, 4 bad pattern); `semgrep --test` has issue #10319 (returns 0 on invalid rule itself), so `--validate` must run first. The design must **adopt the upstream paired-co-located convention without modification, run `semgrep --validate` before `semgrep --test` in any verifier sequence, and surface a smoke-test of `semgrep --test` exit semantics in M1 to confirm whether issue #10319 has been fixed in the 2026 CLI**, because re-inventing the layout would break portability of the generated pack to consumer `semgrep ci` runs and the validate-before-test order is the only documented way to fail-fast on broken rules (Semgrep testing-rules docs; semgrep issue #10319).

## Clippy is a coarse fence to compose with, not replace

Clippy's security restriction lints (`unwrap_used`, `expect_used`, `indexing_slicing`, `arithmetic_side_effects`, `panic`) are all **`Allow` by default** (opt-in only) and carry **no CWE tags** (rust-clippy issue #6636 confirms the lints are debated for production opt-in). They fire on every occurrence with no taint, no source/sink, no per-handler context. The design must **compose Semgrep taint/context rules on top of Clippy restriction lints (consume Clippy as a coarse-fence findings stream, layer Semgrep for source/sink reasoning), not replace them**, because Clippy covers the sink surface broadly but lacks the contextual reasoning Semgrep provides and re-implementing Clippy's sink detection in Semgrep would duplicate effort already done in rustc's lint pipeline (Clippy lint index; arxiv "Unleashing the Power of Clippy").

## `cargo-audit` is the natural CI trigger for the per-bug extend loop

`cargo-audit` 0.22.1 (2026-02, Apache/MIT) emits findings against `Cargo.lock` in JSON and SARIF (`OutputFormat::{Json,Sarif}`); it already does the GHSA-CWE join `/slo-rulegen` needs. `cargo-deny` 0.19.4 (2026-04) emits the same formats for the advisories channel. The design must **wire `cargo-audit` (JSON or SARIF) as the M3 CI trigger that fires the per-bug extend loop: cargo-audit detects a new advisory landing in the project's `Cargo.lock`, the skill takes the advisory's GHSA URL → fix PR → fix-diff and runs `/slo-rulegen --extend` against it**, because cargo-audit is the canonical Cargo.lock-aware advisory scanner in 2026 and pairing it with `/slo-rulegen` closes the loop from "advisory dropped" to "regression rule shipped" with no manual step (cargo-audit 0.22.1 release; cargo-deny 0.19.4 release).

## Compliance default in the rule-pack repo flips to tracked-and-labelled; the user's app repo stays gitignore'd

GitHub's Acceptable Use Policy explicitly carves out educational vulnerability content. No takedown notices surfaced against semgrep-rules-style repos for deliberately-vulnerable test snippets. Upstream convention (semgrep-rules, Trail of Bits, 0xdea) is uniformly tracked-and-labelled with `// ruleid:` / `// ok:` annotations. The idea doc's compliance-fine framing was over-cautious **for the rule-pack repo itself**. However, snippets in a **user's application repo** are different — that repo is not a security-tooling repo, so tracked vuln snippets there would be a compliance finding (SOC 2 / ISO 27001 perspective) even if portable. The design must **default to tracked-and-labelled-with-disclaimer for snippets in the rule-pack repo and `.gitignore`'d for snippets generated inside a user's application repo by the extend-mode loop, with the skill detecting the repo's role and choosing the default**, because GitHub AUP permits educational snippets in security-tooling repos and upstream convention treats them as expected, but a third-party application repo should not become an exploit corpus by accident (GitHub AUP; semgrep-rules layout convention).

## `prek` is mature enough to document as a Rust-native fallback for the local hook runner

`pre-commit` (Python) remains the canonical local-hook runner in 2026; Semgrep ships a first-party hook repo at `semgrep/pre-commit`. **`prek 0.3.10` (MIT, 2026-04-21) is a drop-in `pre-commit` replacement** that reads `.pre-commit-config.yaml` unchanged; production users include CPython and Apache Airflow. `lefthook` is parallel-by-default but does not auto-install tools or pull remote hook configs. No `cargo-semgrep` subcommand wrapper exists. The design must **emit a `.pre-commit-config.yaml` that runs unchanged under both `pre-commit` and `prek`, default install instructions to `pre-commit`, document `prek` as a Rust-native alternative, and not adopt `lefthook`**, because the YAML-format compatibility means dual support is free and `prek` removes the Python-bootstrap friction in a Rust-only repo without forcing any user to migrate (j178/prek release; semgrep/pre-commit hook repo).

## Variation enumeration is one rule with N pattern-either arms, not N rules

Trail of Bits' panic-in-function-returning-result rule enumerates **4 fn-context variations inside a single `pattern-either`** (direct `Result<T1,T2>`, direct `Result<T>`, type-alias to `Result<T1,T2>`, type-alias to `Result<T>`) plus 3 `pattern-not-inside #[cfg(test)] mod tests` exclusions, paired with one fixture file containing N `// ruleid:` annotations. The idea doc had assumed "3-5 variation rules per bug" — research shows the same coverage is achievable with **1 rule + N pattern-either arms + 1 fixture file with N inline annotations**. The design must **express variation coverage as ≥ N `pattern-either` arms within a single rule with a paired single fixture file containing N `// ruleid:` annotations, not N separate rule files**, because Trail of Bits' production precedent demonstrates the structural-enumeration shape works and reduces rule-pack file count by a factor of N, simplifying the manifest and the verifier's coverage-counting step (Trail of Bits panic-in-function-returning-result.yaml).

## Trail of Bits AGPL forces clean-room re-authoring

Trail of Bits `semgrep-rules` is AGPL-3.0; copy-pasting YAML wholesale would inherit AGPL. Pattern shapes are likely uncopyrightable (functional content), but a clean re-implementation removes ambiguity. The design must **re-author all rules independently, treating ToB structural shapes (variation enumeration via `pattern-either`, `pattern-not-inside #[cfg(test)]` exclusions) as inspiration only, and document the re-authoring policy in `references/sast/AUTHORING.md`**, because the wedge's intended distribution license is unspecified but explicitly NOT AGPL, and an inadvertent AGPL contamination would block adoption in any commercial Rust project (Trail of Bits semgrep-rules license).

## Open data points that gate M1 milestone exit

Three smoke-tests must run inside M1 before any rule template is committed: (a) `pattern-inside: unsafe { ... }` to confirm the primitive works for Rust; (b) `semgrep --test` exit-code on an invalid rule to confirm whether semgrep issue #10319 has been fixed in the 2026 CLI (and therefore whether `--validate` is strictly required first); (c) the RustSec → GHSA → OSV `cwe_ids` join hit-rate over the last 24 months of advisories to size the residual category-only fallback. The design must **defer publishing `references/sast/variations/cwe-119.md` and `references/sast/variations/cwe-787.md` until smoke-test (a) confirms `pattern-inside: unsafe { ... }` works, encode the M1 verifier order as `--validate` then `--test` regardless of (b) until disproven, and bake the actual hit-rate from (c) into `cwe-map-rust.md` as the documented fallback policy**, because each of these is a documented Semgrep gap or unmeasured RustSec characteristic and proceeding without confirmation risks templating rules that silently false-negative in production (Semgrep issues #10319/#10380/#11150; google/osv.dev issue #3245).
