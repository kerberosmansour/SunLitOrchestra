## Known non-blocking limitations

- Structural-contract tests validate the documented shape of the skill, not the runtime behavior of Claude Code interpreting the skill. A malicious but well-formed SKILL.md that passes the tests could still misbehave at runtime. Mitigation: review gate via `/slo-critique` (including the rewritten security persona in M3) + `/slo-execute`'s allow-list discipline.
- The frontmatter type checker in the test file is a minimal regex-style parser; it handles the flow-style YAML lists the overview files use but would not catch multi-line list syntax (`compliance:\n  - soc2\n  - asvs`). Acceptable because the SKILL.md documents the flow-style form as the convention.
- The AI triad section in `/slo-architect`'s threat-model template embeds specific MITRE ATLAS technique IDs (`AML.T0058`, etc.). If ATLAS renumbers in a future release, the template will need a refresh. No automated drift check today.
- Dogfood `SECURITY.md` at repo root contains a placeholder-populated "Top risks" block noting that `/slo-ideate` for this runbook predated Q7. A future refresh of the idea doc could fill it properly; deferred.
- Scope-boundary judgment on the overview frontmatter update is flagged in the Evidence Log. If the user disagrees, reverting that one file (keeping `SECURITY.md` + threat-model as dogfood outputs) is clean.
