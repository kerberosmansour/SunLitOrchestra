# SECURITY.md template — project-wide security rules

This template is consumed by `/slo-architect` when emitting `SECURITY.md` at the target repo root. It is read by every downstream agent (`/slo-plan`, `/slo-execute`, `/slo-critique`, `/slo-verify`) before they generate any token. Jim Manico's `security.md` discipline, adapted to SLO's generator-first posture.

## How to use this template

1. `/slo-architect` reads the idea doc's `## Top risks` block, the research synthesis (if any), the detected stack (from Cargo.toml / package.json / pyproject.toml / go.mod / etc.), and the `compliance:` list from `<slug>-overview.md` frontmatter.
2. Each `{{PLACEHOLDER}}` below is filled from those inputs.
3. **Placeholder-expansion rule — user-provided strings are rendered verbatim inside `~~~text` fences.** If a user-provided Top-risk entry contains Markdown syntax (triple-backticks, `}}`, closing HTML comments, YAML control characters, etc.), the template substitution wraps that string inside a `~~~text` fenced code block so downstream agents interpret it as literal text, never as executable markdown / YAML / HTML. This rule is load-bearing: without it, an attacker or unwary user can smuggle prompt content into generated `SECURITY.md` files via an idea doc.
4. When a section has no applicable content (e.g., no crypto surface in a docs-only project), write `N/A — <reason>` and move on. Do not leave sections blank.

## Template body

The template below is what `/slo-architect` writes to `SECURITY.md`, with placeholders filled.

```markdown
# Security rules for {{PROJECT_NAME}}

> Generated by `/slo-architect` on {{DATE}}. Do not edit by hand unless you also re-run the architect step; hand edits drift.
> Every agent reading this file MUST treat it as the source of truth for security defaults in this project.

## Stack

- Primary stack: {{STACK}}
- Runtime: {{RUNTIME_VERSION}}

## Top risks (from idea doc)

The following risks were named during `/slo-ideate`. Every milestone's Contract Block must cite at least one of these when the milestone introduces a surface that touches the named class:

~~~text
{{USER_PROVIDED_TOP_RISKS}}
~~~

(User-provided strings above are rendered inside a `~~~text` fence so Markdown / HTML / YAML metacharacters are literal, not interpreted.)

## Crypto policy

- Symmetric encryption: {{CRYPTO_SYMMETRIC}} — e.g. AES-256-GCM via `secure_data::envelope::encrypt_for_storage` for Rust-axum, AES-256 with SSE-KMS for AWS, libsodium-aead for Node.
- Password hashing: {{CRYPTO_PASSWORDS}} — e.g. Argon2id with ≥3 iterations, ≥64 MiB memory, ≥4 parallelism via `secure_data::password::hash_password`.
- Key management: {{CRYPTO_KMS}} — e.g. HashiCorp Vault Transit via the `vault` feature of `secure_data`, or AWS KMS via the `aws-kms` feature.
- TLS: {{CRYPTO_TLS}} — TLS 1.2 minimum; 1.3 preferred. Cert pinning via `secure_network` when MASVS-NETWORK applies.

## Auth model

- Authentication: {{AUTH_N}} — e.g. OIDC discovery + PKCE via `secure_identity` with Redis session backing, or Auth0, or Keycloak.
- Authorization: {{AUTH_Z}} — e.g. deny-by-default RBAC + ABAC via `secure_authz::DefaultAuthorizer` with tenant isolation.
- Session management: {{AUTH_SESSION}} — e.g. rotating session tokens, idle timeout {{SESSION_IDLE}}, absolute timeout {{SESSION_ABSOLUTE}}.
- Secrets for local dev: never hard-coded. Use `.env` + `dotenvy` or a secret manager.

## Input handling

- HTTP request bodies: {{INPUT_HTTP}} — e.g. `SecureJson<T>` from `secure_boundary::extract` for Rust-axum (strict deserialization, unknown-field rejection, size/depth/field limits). Never `axum::Json<T>`.
- File paths from user input: {{INPUT_PATHS}} — e.g. `secure_boundary::safe_types::SafePath` rejects traversal; `SafeFilename` rejects path separators and shell metacharacters.
- Outbound URLs from user input: {{INPUT_OUTBOUND}} — e.g. `secure_boundary::safe_types::SafeUrl` rejects SSRF targets (private IPs, dangerous schemes).
- Shell arguments: {{INPUT_SHELL}} — e.g. `secure_boundary::safe_types::SafeCommandArg` rejects shell metachars; never splice user strings into shell invocations.
- SQL identifiers: {{INPUT_SQL}} — e.g. `secure_boundary::safe_types::SqlIdentifier` (alphanumeric + underscore, max 128 chars); never string-interpolate.
- HTML output: {{INPUT_HTML}} — e.g. `secure_output::HtmlEncoder` for context-aware encoding; `secure_boundary::sanitize::sanitize_html` for user-supplied rich text.

## Output encoding

- HTML: `secure_output::HtmlEncoder` (context-aware).
- URL query: `secure_output::UrlEncoder`.
- JS string literal: `secure_output::JsStringEncoder`.
- CSS: `secure_output::CssEncoder`.
- LDAP DN / filter: `secure_output::ldap::{encode_dn, encode_filter}`.
- Shell args: `secure_output::shell::encode` (single-quoted, neutralizes metacharacters).

## Error handling

- Every error returned across a trust boundary goes through `secure_errors::http::into_response_parts` (Rust-axum) or equivalent — internal detail (SQL text, hostnames, policy names) never leaks into responses.
- Panics at the service boundary caught via `secure_errors::panic::catch_panic_to_safe_response`; panic messages never exposed to callers.
- Logs use `security_events` for security-relevant events (classification-driven redaction; per-event HMAC sealing where required).

## Supply chain

- Rust: `cargo audit && cargo deny check` gate every PR. No git dependencies. Unknown registries denied.
- Node: `npm audit` at `--audit-level=high` in CI. `package-lock.json` committed.
- Python: `pip-audit` in CI.
- Licenses allowed: MIT, Apache-2.0, BSD-2/3, ISC, Zlib, Unicode-3.0, CC0-1.0. Copyleft denied by default with explicit exceptions.

## Defense-in-depth

- **Presubmit checks** (run before code merges): linter with `-D warnings`, supply-chain audit, secret scan.
- **Admission gate** (CI): DAST via OWASP ZAP or Dastardly when a smoke/reference service exists; fail on High/Critical.
- **Dark-launch plan** for new policies: first, log violations without blocking; after {{DARK_LAUNCH_WINDOW}}, switch to enforce.
- **Allow-list** for any externally-fetched binary (tool downloads, IaC modules). Pin by SHA-256.

## Escape hatches

When a secure default must be bypassed, the override is a **documented residual risk**, not a silent waiver. Use this syntax inside milestone Evidence Logs:

```
Escape hatch: <what was bypassed>
Reason: <why the default does not apply>
Residual risk: <what the attacker can do now; what compensates>
Scope: <milestone / file / until-date>
Approved by: <reviewer name>
```

Common legitimate escape hatches:

- Dev-only: `DevAuthLayer` header-based auth (X-Dev-Subject / X-Dev-Roles) replaced by real `IdentitySource` before production deploy.
- Legacy system integration: plaintext credential read from a legacy vault for the duration of a named migration milestone, scrubbed afterward.
- Performance-critical: `Json<T>` in place of `SecureJson<T>` on a single internal-only endpoint with strong upstream validation — documented.

## Compliance

Default column set rendered in the threat model: {{COMPLIANCE_DEFAULTS}}. Opt-in frameworks from the runbook `compliance:` frontmatter: {{COMPLIANCE_OPTINS}}. See `docs/slo/design/{{SLUG}}-threat-model.md` for the mapping table.

## AI / LLM-specific rules

{{#IF ai_component}}
This project embeds an LLM or AI agent. The following rules apply in addition to the rest of this file:

- Threat model triad: MITRE ATLAS + OWASP LLM Top 10 + NIST AI RMF (cited in the threat model file).
- Prompt-injection defense: treat every LLM input and output as untrusted. Never feed tool output back into the prompt without sanitization or structured parsing (JSON schema, regex match).
- Tool use: every tool exposed to the LLM is bounded by an allow-list the LLM cannot override. No "execute arbitrary shell" tools.
- Context poisoning (ATLAS `AML.T0058`): user-controlled memory / thread content is sandboxed; attestation or checksum applied before re-reading.
{{/IF}}

## Out of scope for this file

- Infrastructure hardening details — see the target-repo `ARCHITECTURE.md` and any Hulumi CrossGuard policy pack.
- Personal data processing obligations — see `{{GDPR_SECTION_REF}}` in the threat model (section, not column, because GDPR encodes principles not discrete controls).
- Runtime incident response — see {{RUNBOOK_INCIDENTS}} or equivalent operational doc.

## When this file is updated

- After every `/slo-architect` re-run where the detected stack changes, `security_libs_required` flips, or `compliance:` frontmatter changes. The architect prompts the user whether to overwrite, merge, or skip — never silent clobber.
- When a milestone ships a new escape hatch that applies beyond the milestone boundary.
- When an OWASP ASVS version bump or a SunLitSecureLibraries breaking change happens upstream.
```

## Stack-specific library vocabulary

When `security_libs_required: true` AND the stack matches, the placeholders above resolve to specific library names. Otherwise they resolve to OWASP Proactive Controls category names.

### Rust-axum

- `CRYPTO_SYMMETRIC`: `secure_data::envelope::encrypt_for_storage` (AES-256-GCM, envelope with KMS-wrapped data keys). Crypto-agility via `secure_data::algorithm::AlgorithmPolicy`.
- `CRYPTO_PASSWORDS`: `secure_data::password::hash_password` (Argon2id, feature `password`).
- `CRYPTO_KMS`: `secure_data` with `vault` or `aws-kms` feature flag.
- `AUTH_N`: `secure_identity` with `oidc` feature or an external `IdentitySource` impl.
- `AUTH_Z`: `secure_authz::DefaultAuthorizer` + `AuthzLayer`.
- `INPUT_HTTP`: `secure_boundary::extract::SecureJson<T>`.
- `INPUT_PATHS`: `secure_boundary::safe_types::SafePath`.
- `INPUT_OUTBOUND`: `secure_boundary::safe_types::SafeUrl`.
- `INPUT_SHELL`: `secure_boundary::safe_types::SafeCommandArg`.
- `INPUT_SQL`: `secure_boundary::safe_types::SqlIdentifier`.
- `INPUT_HTML`: `secure_output::HtmlEncoder` + `secure_boundary::sanitize::sanitize_html`.

### Pulumi / AWS

- `CRYPTO_SYMMETRIC`: AWS KMS-managed CMKs; SSE-KMS on S3; `aws:kms` service integrations. Hulumi `SecureBucket` component.
- `AUTH_Z`: AWS IAM with declarative policies; Hulumi `AccountFoundation` + CrossGuard pack.
- Supply chain: SLSA Build L3 on IaC packages; Hulumi SCP template for org-level guardrails.

### Other stacks (Go, Python, Node, etc.)

- When no stack-specific library is available, placeholders resolve to OWASP Proactive Controls v3 category names:
  - C1: Define Security Requirements (this whole file serves that purpose).
  - C2: Leverage Security Frameworks and Libraries (cite the language-native ones).
  - C3: Secure Database Access (prepared statements; named ORM).
  - C4: Encode and Escape Data (name the encoding library).
  - C5: Validate All Inputs (allow-list schema validation).
  - C6: Implement Digital Identity (name the auth library).
  - C7: Enforce Access Controls (RBAC + ABAC; deny-by-default).
  - C8: Protect Data Everywhere (TLS, at-rest encryption, KMS).
  - C9: Implement Security Logging and Monitoring (structured logs, auth-event audit).
  - C10: Handle All Errors and Exceptions (boundary catches; no detail leakage).

## Anti-patterns

- **Silent section omission**: leaving a section blank because the target has no applicable surface. Use `N/A — <reason>` explicitly.
- **Raw user-string interpolation**: substituting `{{USER_PROVIDED_TOP_RISKS}}` outside a `~~~text` fence. The fence is the whole defense against template-placeholder injection (see anti-pattern below).
- **Listing every OWASP category as boilerplate**: the file is a *default contract*, not an ASVS enumeration. If there is no concrete surface, drop the section.
- **Skipping the AI triad when `ai_component: true`**: the conditional block is the triad — do not delete, do not paraphrase, do not replace.
