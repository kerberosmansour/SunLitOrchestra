## Stack

- Primary stack: Rust 2021 workspace (CLI binaries + shared library) + Markdown skill pack consumed by Claude Code.
- Runtime: Rust stable (MSRV ~ latest supporting `clap 4`, `thiserror 2`, `reqwest 0.11`); no explicit rust-toolchain pin.
- Secondary stack: TypeScript + React + Tauri under `crates/sldo-tauri/` — **parked** as of 2026-04; not in scope for new security work (see CLAUDE.md).

## Crypto policy

- Symmetric encryption: N/A — SLO stores no confidential data at rest or in transit. The only cryptographic operation is SHA-256 hashing of fetched binaries in `sldo-tla-sha` (integrity, not confidentiality).
- Password hashing: N/A.
- Key management: N/A.
- TLS: `reqwest` with `rustls-tls` in `sldo-tla-sha` for HTTPS fetches; no custom TLS configuration. Host allow-list enforced in `sldo-tla-sha::allowed_hosts` pre- and post-redirect.

## Auth model

- Authentication: N/A — SLO is a local CLI / skill pack; no service endpoints, no user accounts.
- Authorization: N/A — filesystem permissions are the only authorization surface; SLO does not implement its own.
- Session management: N/A.
- Secrets for local dev: `OPENAI_API_KEY` (used only by the parked Tauri voice-transcribe feature) read from `.env` via `dotenvy` in the Tauri backend; never sent to the frontend. No secrets in source control — `.env` is gitignored.

## Input handling

- CLI argument parsing: every binary uses `clap 4` with derive. Arguments are typed (`PathBuf`, `String`, bounded integers); no manual string parsing of user input. **Allowed pattern.**
- SKILL.md / prompt files: `sldo-research` accepts a `<prompt-file>` argument and passes its contents to `claude` via stdin. The prompt file is trusted input — the user chose it. **Do not pass untrusted prompt files** (e.g., user-uploaded content on a server); this is documented in the research skill's README.
- Subprocess invocation: `Command::new("claude" | "git" | "cargo")` with explicit argument lists (no shell). Arguments derived from user input are passed as separate args, never interpolated into a shell string. `Stdio::piped()` with drain-in-separate-threads to avoid pipe-buffer deadlocks.
- Outbound URLs: `sldo-tla-sha` fetches HTTPS URLs from `skills/slo-tla/tools.toml`. Host allow-list (`github.com`, `objects.githubusercontent.com`, `raw.githubusercontent.com`, etc. — see `src/allowed_hosts.rs`) enforced pre-redirect and post-redirect. Hard byte cap (`DEFAULT_MAX_BYTES = 500 MiB`) on streamed response to prevent OOM on a malicious CDN response.
- File paths from user input: all paths used by `sldo-install` are computed from `~/.claude/` + skill name; user never provides arbitrary paths.
- SQL / LDAP / shell interpolation: N/A — no database, no LDAP, no shell interpolation.
- HTML output: N/A — no rendered HTML surface (parked Tauri app excluded).

## Output encoding

N/A for current surfaces. If future work reintroduces an HTTP API (which would require un-parking Tauri or a new service), use `secure_output` from the SunLitSecureLibraries workspace.

## Error handling

- Errors use `anyhow::Result` on glue paths and `thiserror 2` for typed errors in libraries.
- CLI errors print a concise message; no stack traces or internal paths in user-facing error output (verified by existing `sldo-common` tests).
- No panics are exposed to users in normal flows; panics in tests use `tempfile::TempDir` cleanup so no residual state.
- Logs (`.sldo-logs/`) are gitignored because they may contain proprietary source excerpts — treat as internal.

## Supply chain

- Rust: `cargo audit` + `cargo deny check` are the Phase 3 (`/slo-security-test`) contract for future CI; not currently wired. Recommended additions to CI: run on every PR, fail on High/Critical.
- No git dependencies in any workspace `Cargo.toml`; all deps from crates.io.
- Licenses: `MIT`, `Apache-2.0`, `BSD-3-Clause` across the dep graph (not currently enforced by `cargo deny`; manual audit only).
- Skill pack installation: `sldo-install` uses symlinks with no elevated privileges; uninstall preserves user-modified symlinks (warns rather than overwrites).

## Defense-in-depth

- **Presubmit checks**: `cargo fmt` (manual; not yet wired to a pre-commit hook), `cargo clippy` (ad-hoc), `cargo test` per-crate baseline. Pre-commit hooks are a Phase 3 improvement target.
- **Admission gate**: the `sldo-plan` / `sldo-run` / `sldo-research` binaries refuse to run on `main` / `master` via `preflight::check_git_safety`. Feature branches required.
- **Dark-launch plan** for new policies: N/A for this project size — new defaults land directly in a runbook milestone.
- **Allow-list** for externally-fetched binaries: `sldo-tla-sha` requires a pinned SHA-256 in `tools.toml` for every download; integrity verified before extraction.

## AI / LLM-specific rules

This project orchestrates Claude Code (`claude` CLI subprocess). The following rules apply in addition to the rest of this file:

- Threat model triad: **MITRE ATLAS + OWASP LLM Top 10 + NIST AI RMF** (cited in the threat model file).
- Prompt-injection defense: `sldo-research` dispatches user-authored prompt files to Claude Code with `WebFetch` and `WebSearch` enabled. **Do not pass untrusted prompt files** — any URL the model fetches is effectively under the prompt author's control. Documented in the research README.
- Tool use: every skill is bounded by a `toolflags::plan_allow_flags()` / `plan_deny_flags()` allow-list. Agents cannot invoke tools outside the pack's declared permissions. No `execute arbitrary shell` tools exposed.
- Context poisoning (ATLAS `AML.T0058`): skill-pack edits are review-gated by the `/slo-critique` security persona and enforced by `/slo-execute`'s allow-list rule. Tampered `SKILL.md` content is detected on re-install by `sldo-install` manifest comparison.
- Tool output handling: Claude Code's tool output is untrusted data. Parsers (e.g., future SARIF parsers in `/slo-security-test`) must reject malformed input rather than echo it back into the prompt.
