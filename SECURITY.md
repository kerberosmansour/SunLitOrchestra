## Stack

- Primary stack: Rust 2021 workspace (CLI binaries + shared library) + Markdown skill pack consumed by Claude Code.
- Runtime: Rust stable (MSRV ~ latest supporting `clap 4`, `thiserror 2`, `reqwest 0.11`); no explicit rust-toolchain pin.
- Secondary stack: TypeScript + React + Tauri under `crates/sldo-tauri/` — **parked** as of 2026-04; not in scope for new security work (see CLAUDE.md).

## Crypto policy

- Symmetric encryption: N/A — SLO stores no confidential data at rest or in transit. The only cryptographic operation is SHA-256 hashing of fetched binaries in `sldo-tla-sha` (integrity, not confidentiality).
- Password hashing: N/A.
- Key management: N/A.
- TLS: `reqwest` with `rustls-tls` in `sldo-tla-sha` for HTTPS fetches; no custom TLS configuration. Host allow-list enforced in `sldo-tla-sha::allowed_hosts` pre- and post-redirect.

## Auth model

- Authentication: N/A — SLO is a local CLI / skill pack; no service endpoints, no user accounts.
- Authorization: N/A — filesystem permissions are the only authorization surface; SLO does not implement its own.
- Session management: N/A.
- Secrets for local dev: `OPENAI_API_KEY` (used only by the parked Tauri voice-transcribe feature) read from `.env` via `dotenvy` in the Tauri backend; never sent to the frontend. No secrets in source control — `.env` is gitignored.

## Input handling

- CLI argument parsing: every binary uses `clap 4` with derive. Arguments are typed (`PathBuf`, `String`, bounded integers); no manual string parsing of user input. **Allowed pattern.**
- SKILL.md / prompt files: `sldo-research` accepts a `<prompt-file>` argument and passes its contents to `claude` via stdin. The prompt file is trusted input — the user chose it. **Do not pass untrusted prompt files** (e.g., user-uploaded content on a server); this is documented in the research skill's README.
- Subprocess invocation: `Command::new("claude" | "git" | "cargo")` with explicit argument lists (no shell). Arguments derived from user input are passed as separate args, never interpolated into a shell string. `Stdio::piped()` with drain-in-separate-threads to avoid pipe-buffer deadlocks.
- Outbound URLs: `sldo-tla-sha` fetches HTTPS URLs from `skills/slo-tla/tools.toml`. Host allow-list (`github.com`, `objects.githubusercontent.com`, `raw.githubusercontent.com`, etc. — see `src/allowed_hosts.rs`) enforced pre-redirect and post-redirect. Hard byte cap (`DEFAULT_MAX_BYTES = 500 MiB`) on streamed response to prevent OOM on a malicious CDN response.
- File paths from user input: install roots used by `sldo-install` are computed from the selected host config directory (`.claude`, `.copilot`, or `.codex`) + skill name; user never provides arbitrary target paths.
- SQL / LDAP / shell interpolation: N/A — no database, no LDAP, no shell interpolation.
- HTML output: N/A — no rendered HTML surface (parked Tauri app excluded).

## Output encoding

N/A for current surfaces. If future work reintroduces an HTTP API (which would require un-parking Tauri or a new service), use `secure_output` from the SunLitSecureLibraries workspace.

## Error handling

- Errors use `anyhow::Result` on glue paths and `thiserror 2` for typed errors in libraries.
- CLI errors print a concise message; no stack traces or internal paths in user-facing error output (verified by existing `sldo-common` tests).
- No panics are exposed to users in normal flows; panics in tests use `tempfile::TempDir` cleanup so no residual state.
- Logs (`.sldo-logs/`) are gitignored because they may contain proprietary source excerpts — treat as internal.

## Supply chain

- Rust: `cargo audit` + `cargo deny check` are the Phase 3 (`/slo-security-test`) contract for future CI; not currently wired. Recommended additions to CI: run on every PR, fail on High/Critical.
- No git dependencies in any workspace `Cargo.toml`; all deps from crates.io.
- Licenses: `MIT`, `Apache-2.0`, `BSD-3-Clause` across the dep graph (not currently enforced by `cargo deny`; manual audit only).
- Skill pack installation: `sldo-install` uses managed directory links. Linux and macOS use symlinks; Windows tries symlinks and falls back to directory junctions when symlink privileges are unavailable. Uninstall preserves user-modified links (warns rather than overwrites).

## Defense-in-depth

- **Presubmit checks**: `cargo fmt` (manual; not yet wired to a pre-commit hook), `cargo clippy` (ad-hoc), `cargo test` per-crate baseline. Pre-commit hooks are a Phase 3 improvement target.
- **Admission gate**: the `sldo-plan` / `sldo-run` / `sldo-research` binaries refuse to run on `main` / `master` via `preflight::check_git_safety`. Feature branches required.
- **Dark-launch plan** for new policies: N/A for this project size — new defaults land directly in a runbook milestone.
- **Allow-list** for externally-fetched binaries: `sldo-tla-sha` requires a pinned SHA-256 in `tools.toml` for every download; integrity verified before extraction.

## AI / LLM-specific rules

This project orchestrates Claude Code (`claude` CLI subprocess). The following rules apply in addition to the rest of this file:

- Threat model triad: **MITRE ATLAS + OWASP LLM Top 10 + NIST AI RMF** (cited in the threat model file).
- Prompt-injection defense: `sldo-research` dispatches user-authored prompt files to Claude Code with `WebFetch` and `WebSearch` enabled. **Do not pass untrusted prompt files** — any URL the model fetches is effectively under the prompt author's control. Documented in the research README.
- Tool use: every skill is bounded by a `toolflags::plan_allow_flags()` / `plan_deny_flags()` allow-list. Agents cannot invoke tools outside the pack's declared permissions. No `execute arbitrary shell` tools exposed.
- Context poisoning (ATLAS `AML.T0058`): skill-pack edits are review-gated by the `/slo-critique` security persona and enforced by `/slo-execute`'s allow-list rule. Tampered `SKILL.md` content is detected on re-install by `sldo-install` manifest comparison.
- Tool output handling: Claude Code's tool output is untrusted data. Parsers (e.g., future SARIF parsers in `/slo-security-test`) must reject malformed input rather than echo it back into the prompt.

## Biz skill pack — additional rules

> Added 2026-04-25 by `/slo-architect biz-skill-pack`; expanded to 15 skills 2026-04-25 (added `/slo-gtm`, `/slo-product`, `/slo-marketing`). Apply when authoring or executing any of the 15 biz skills: advisor cluster (`/slo-legal`, `/slo-accounting`, `/slo-equity`, `/slo-fundraise`); discovery → strategy → definition (`/slo-talk-to-users`, `/slo-gtm`, `/slo-product`, `/slo-marketing`); execution → optimization (`/slo-launch`, `/slo-sales-funnel`, `/slo-pricing`, `/slo-metrics`); team (`/slo-cofounder`, `/slo-hire`, `/slo-founder-check`).

### Founder personal data — handling discipline

The biz pack ingests founder-supplied prose that frequently contains real persons' names, counterparty names, deal values, and IP descriptions (NDAs, contractor SOWs, interview transcripts, hire pipelines). This is project-wide UK GDPR + ASVS territory and is governed by:

- **Two-tier output convention.** Confidential drafts (real names + values) write to `docs/biz/<area>/...` (gitignored by default in the founder's repo). Placeholder-only artifacts (decisions, lawyer-brief preps, public analyses) write to `docs/biz-public/<area>/...` (git-tracked). The artifact-schema reference at `references/biz/artifact-schema.md` declares each artifact category's tier.
- **`.gitignore` default for confidential tier.** Runbook A M1 ships a `.gitignore` snippet for the founder's repo that excludes `docs/biz/`. Skills warn at write-time when the target dir is git-tracked AND a remote exists AND the artifact is `tier: confidential`.
- **PII-pattern scan in `/slo-verify`.** Pass 4 (security) scans any artifact landing in `docs/biz-public/` for known PII patterns (email regex, UK NI numbers, sort codes, named-person heuristics) and fails the milestone if a confidential-looking artifact is in the public tier.
- **No WebFetch / WebSearch in biz skills.** Founder personal data lives in the prompt context. Enabling model-driven web fetching from that context would create an exfiltration surface (attacker-influenced URL pulls back content that the model interpolates with the founder's pasted data on a subsequent turn). Citations only.

### Triage gate — single source of truth

Four hard-coded predicates live in `references/biz/triage-gate.md`. The four advisor skills (legal, accounting, equity, fundraise) cite this file. Predicate IDs are stable interface (see `docs/slo/design/biz-skill-pack-interfaces.md`):

- `gate-1-regulated` — FCA / MHRA / ICO / healthcare / financial services. Routes to lawyer.
- `gate-2-deal-value-over-5k` — deal value > £5,000. Routes to lawyer.
- `gate-3-counterparty-has-lawyer-or-their-paper` — counterparty has a lawyer OR founder is signing their paper. Routes to lawyer.
- `gate-4-gdpr-document` — privacy notice / ROPA / DPA / internal data-protection policy. Routes to DPO. **`draft` mode is unconditionally refused for any GDPR document; only `translate` and `triage` are permitted** (broad-block calibration locked 2026-04-25; reversible only via fresh `/slo-architect` pass with new ICO enforcement evidence).

`/slo-critique` security persona reads `references/biz/triage-gate.md` as authoritative. Any advisor SKILL.md prose that diverges from the predicate set is a P1 finding. Adding a fifth predicate is a `/slo-architect` decision, not a per-milestone option.

### oneNDA template — verbatim rendering under CC BY-ND 4.0

`/slo-legal draft nda` renders the canonical oneNDA template body **byte-for-byte unmodified** (https://www.onenda.org/, CC BY-ND 4.0). Company-specific fields, counterparty details, and cover page emit as a separately-rendered artifact that wraps but does not edit the canonical text. Editing the body — even a "small wording fix" — breaches the licence.

- `references/biz/templates/onenda-uk.md` carries the canonical bytes.
- `sldo-install` install-time test asserts SHA-256 of the file matches the pinned hash. Mismatch fails the install and blocks `/slo-legal draft nda`.
- `/slo-verify` Pass 4 regression test runs `/slo-legal draft nda` against a fixture and asserts the rendered NDA body bytes match the canonical hash.

### Advisor mode contract — what `draft` outputs MUST carry

Every artifact emitted by an advisor skill in `draft` mode (post-gate) carries this frontmatter (full schema in `references/biz/artifact-schema.md` and `docs/slo/design/biz-skill-pack-interfaces.md`):

- `tier: confidential` (forces `docs/biz/` placement)
- `triage_gate_passed: true` (verified by `/slo-verify`)
- `lawyer_review_recommended: true` (always — drafted docs are first-cut, never final)
- `cost_baseline_ref: references/biz/cost-baseline-jpp-law-2026.md@<retrieval-date>`
- `expires_or_review_by: <date>` (drives annual refresh)

The skill ALSO writes a "LAWYER REVIEW RECOMMENDED" header at the top of the artifact body. The frontmatter is for tooling; the header is for the human who skims the file in six months.

### GDPR posture (locked 2026-04-25)

Broad hard-block on `draft` for ALL GDPR documents (privacy notice, ROPA, DPA, internal data-protection policies). Defensible on professional-negligence + upside-asymmetry grounds (DUAA 2025 Stage 3 commenced 5 February 2026; PECR fine ceiling raised to £17.5M / 4% global turnover; complaints-procedure duty for all controllers from 19 June 2026). The enforcement evidence shows PECR-direct-marketing as the small-company risk pattern, not Article 13 — a narrower posture is also defensible, but the project chose the broad block. Reversal requires fresh `/slo-architect` pass with new evidence.

### Cost baseline — JPP Law (publicly auditable)

ROI claims in `/slo-legal` outputs anchor to JPP Law's fixed-fee public pricing (https://www.jpplaw.co.uk/sectors/fixed-fee-startup/) rather than a private firm's internal price list. The reference at `references/biz/cost-baseline-jpp-law-2026.md` carries an explicit retrieval-date stamp; annual refresh expected and recommended via a `/loop` schedule.

### Out of scope for the biz pack (escalate, do not work around)

- US / EU jurisdiction. v1 is UK only; non-UK arguments emit a clear "v2 architectural pivot" error.
- Equity grant tax advice. Equity-related skills triage; tax decisions route to accountant.
- Real-time legal opinion. Skills produce templates and triage; they do not give legal advice for an active dispute.
- Founder-vs-founder dispute mediation. `/slo-cofounder` produces eval / 1:1 / trial-project artifacts only; live disputes route to a mediator.

## SAST rule-gen skill pack — additional rules

> Added 2026-04-25 by `/slo-architect sast-rulegen-skill-pack`. Apply when authoring or executing `/slo-rulegen` or `/slo-ruleverify`, or modifying `xtasks/sast-verify/` or `references/sast/`.

### Toolflag denial — WebFetch / WebSearch

`rulegen_*` and `ruleverify_*` toolflag families both DENY `WebFetch` and `WebSearch`. The CWE map and variation templates ship pre-baked in `references/sast/`; the skills do not need network access. This denial is the primary control against prompt-injection-driven exfiltration via attacker-supplied bug summaries (see threat model abuse cases tm-sast-rulegen-skill-pack-abuse-1, -3). Removing the denial requires re-running `/slo-architect` against the threat model.

### Two-tier corpus output convention

Generated vulnerable-snippet corpora follow a two-tier convention identical in spirit to the biz pack's `docs/biz/` vs `docs/biz-public/` split:

- **Rule-pack repo (this repo or any future SAST.GEN-shaped pack):** corpora are tracked-and-labelled in-tree. Per-rule `<rule-id>.rs` fixture lives next to `<rule-id>.yaml` in `.semgrep/<lang>/`, with `// ruleid:` (fire-on-bad) and `// ok:` (silent-on-good) inline annotations. Header comment marks the file test-only. This is upstream Semgrep convention; GitHub AUP "Active Malware or Exploits" carves out educational vulnerability content explicitly.
- **User application repo (extend-mode target, NOT a security-tooling repo):** corpora default to `.gitignore`'d. The skill detects via `git remote get-url origin` heuristics and forces `--target-tier confidential` unless the user explicitly opts in. The `.gitignore` snippet ships in M3 alongside the CI wiring.

Skills warn at write-time when the target dir is git-tracked AND a remote exists AND the artifact tier is `confidential`. Same warning shape as the biz pack's confidentiality warning.

### `gate` subcommand is non-bypassable

`/slo-rulegen` MUST NOT write any rule file unless `cargo xtask sast-verify gate <rule-path>` exits 0. The four sub-checks (`validate`, `test`, `check-coverage`, `check-clean`) are the breach-mitigation enforcement mechanism named in the idea doc. The `gate` subcommand is the single deterministic entry point; bypassing it (e.g., by writing rule files directly via the Write tool while the gate fails) is a P1 finding for `/slo-critique` security persona.

`check-clean` runs each rule against a "known-clean" subset of the host repo (default: the crate's own `src/` after the bug fix is applied) and rejects rules with ≥ 1 false positive — this is the prolonged-outage mitigation.

`check-coverage` enforces both a minimum N (variation floor — breach mitigation) and a maximum N (default 25 — DoS-via-pattern-explosion ceiling, threat model abuse case tm-sast-rulegen-skill-pack-abuse-5).

### Subprocess hygiene for `semgrep` invocations

The `xtasks/sast-verify` crate invokes `semgrep` via `std::process::Command::new("semgrep")` with explicit `args()`, `Stdio::piped()`, drain-on-separate-thread per the `sldo-common::copilot` precedent. Never via shell. Never with environment-variable smuggling. Always with `--timeout` (default 30s) and `--max-target-bytes` (default 1 MB) flags passed through.

Semgrep version pinned via `semgrep --version` parse against `references/sast/MIN-SEMGREP-VERSION.md`. xtask refuses to run if version is below minimum or `semgrep` is not on PATH.

### Tool output handling — `--json` over stdout

The xtask consumes `semgrep --json` structured output, parsed with `serde_yaml` / `serde_json` strict mode (rejects unknown fields). Raw stdout is logged but never re-fed to Claude Code as prompt content. This breaks the "prompt-injection-via-tool-output" loop that LLM01 / LLM05 surface in the threat model.

### Re-authoring policy — Trail of Bits AGPL clean-room

`references/sast/AUTHORING.md` documents the policy: structural shapes from public production rule packs (specifically Trail of Bits' `panic-in-function-returning-result.yaml`) are valid inspiration; rule YAML is authored fresh, never copied. Rules in `.semgrep/<lang>/` MUST NOT contain text fragments from AGPL-licensed sources. `/slo-critique` security persona is responsible for spot-checking; a cargo-deny policy on AGPL transitive deps for the workspace would be a Phase 3 hardening upgrade.

### LICENSE addendum

The SLO repo currently has no `LICENSE` file at the root (per the research dossier's noted constraint). Before the SAST rule pack is consumed by any external project, M3 of Runbook A adds a LICENSE file (recommendation: Apache-2.0 OR MIT, matching the surrounding Rust ecosystem default and explicitly NOT AGPL to keep the pack consumable in commercial projects).

### Residual risk — `references/<pack>/` files are NOT SHA-pinned by `sldo-install`

**Acknowledged residual.** `crates/sldo-install/src/install.rs::discover_skills()` requires a `<skills_dir>/<name>/SKILL.md` to discover and pin a path; as a structural consequence, sibling directories `references/biz/` and (post-M1) `references/sast/` are NOT walked, NOT manifest-tracked, and NOT SHA-verified on re-install. This is a pre-existing gap in the biz pack inherited by the SAST pack, NOT a new risk introduced by the SAST runbook.

Concrete exploit shape: a PR modifying `references/sast/variations/cwe-755.md` to lower `minimum-pattern-either-arms` from 4 to 1 (or `references/biz/triage-gate.md` predicates analogously) lands without `sldo-install` detecting the change on next install; users running the affected skill get weakened gates until someone notices.

Compensating controls in place today:
- `/slo-critique` security persona reviews changes to `references/<pack>/` (this file's discipline; review-gated, not code-gated).
- All skills cite specific file paths under `references/<pack>/`, so a removed-or-renamed file surfaces as a missing-reference error at skill-runtime (detect, not prevent).

A Phase-3 hardening runbook should extend `sldo-install`'s walker to include `references/<pack>/` and pin the contents in `~/.sldo/install.toml`. Until then, this is a documented residual that all biz / SAST skill authors and reviewers should know about.

## Scanner orchestration skill — additional rules

> Added 2026-04-26 by `/slo-architect scanner-orchestration`. Apply when authoring or executing `/slo-sast`. The full design lives at [docs/slo/design/scanner-orchestration-overview.md](docs/slo/design/scanner-orchestration-overview.md); the load-bearing safety properties below are restated here so any agent reading SECURITY.md as the source of truth sees them before generating code.

### Emitted-workflow safety contract (non-negotiable)

`/slo-sast` writes `.github/workflows/sast.yml` into the user's TARGET repo. Every emitted workflow MUST satisfy ALL of:

- `on:` block uses `pull_request` (and optionally `schedule`); MUST NOT use `pull_request_target`. Even after the 2025-12-08 GitHub default-branch sourcing mitigation, `pull_request_target` confers full secret access on PR runs and no SAST step the skill emits requires that.
- `permissions:` at workflow scope is `{}` (empty map). Per-job `permissions:` declares only what that job needs (analysis: `contents: read`; SARIF upload: adds `security-events: write`).
- Every `uses:` line resolves to a 40-character commit SHA, never a tag or branch name. CVE-2025-30066 (`tj-actions/changed-files` compromise) and the Shai Hulud v2 worm are the failure-case evidence.
- `actions/checkout` step has `with: { fetch-depth: 0 }`. Default `1` breaks `semgrep ci` diff-aware scans (canonical Semgrep KB pitfall).
- The `semgrep ci` invocation uses the `SEMGREP_RULES` env var (NOT the `--config` flag, which the CLI reference says is "not supported in ci mode" even though it works empirically).
- No `secrets.*` references in the analysis job.

A structural-contract test fixture (`emitted_workflow_has_safe_defaults`) asserts these properties at skill-author time AND at runtime before any PR is opened. Any milestone that touches the workflow template MUST also touch this fixture.

### Threat-model parser scope (defuses prompt-injection)

The skill extracts CWE references from `docs/slo/design/<slug>-threat-model.md` using regex `\bCWE-(\d+)\b` against the rendered Markdown body **only**. The parser MUST exclude:

- HTML comments (`<!-- ... -->`).
- Fenced code blocks (` ``` ` or `~~~`).
- `~~~text` user-string fences (these are user content; the fence rule from slo-security-embedding is the load-bearing defense).

Rationale: the threat-model file is user-authored prose. Treating its content as instructions (even implicitly via comment-scoped CWE references) creates a prompt-injection surface. The parser's narrow scope eliminates this class.

### `semgrep-rules` upstream SHA-pin discipline

The skill clones `github.com/semgrep/semgrep-rules` at a **commit SHA**, never a tag. SHA-pinning bounds the supply-chain trust window to "user explicitly bumps pin" events, which surface as a diff PR for human review. Bumping the pin without reviewing the upstream diff defeats the discipline. `references/sast/MIN-SEMGREP-VERSION.md` carries the floor; the pinned SHA is recorded in `.semgrep/manifest.json:semgrep_rules_sha` per emitted scan.

### Manifest schema is audit-defense, not regulatory necessity

`.semgrep/manifest.json` records `cwes_claimed` (from threat model) vs `cwes_actually_covered` (union of `metadata.cwe` from selected rules) for every scan run. The divergence surfaces gaps the user can route to manual review or to `/slo-rulegen`. **This is defensive design, not a regulatory mandate** — multiple search rounds did not surface a published audit-failure precedent fixing the "mapped-but-not-scanned" pattern as a documented requirement. Frame the field that way in any user-facing prose; do not overpromise audit acceptance.

### Compliance citation discipline — PCI DSS 6.2.3, NOT 6.3.2

Wherever `/slo-sast` (or any artifact it emits, or any documentation written about it) references PCI DSS code-review compliance, cite **PCI DSS 6.2.3 (v4.0.1)**. The v3.2.1 numbering for code review was 6.3.2; v4.0.1 renumbered it. **v4.0.1's 6.3.2 is the new bespoke / third-party software inventory mandate** (different scope; SBOM-adjacent; out of v1). Mixing the two is a substantive error, not a typo, and makes any coverage claim challengeable.

### AI / LLM surfaces specific to `/slo-sast`

Inherits the project-wide AI section. Surfaces specific to this skill:

- **Prompt injection via threat-model file content** (ATLAS AML.T0051 family / OWASP LLM01 / NIST Measure 2.6) — defused by the parser scope rule above plus the workflow YAML template-skeleton (user prose NEVER spliced into emitted YAML; only regex-validated `CWE-\d+` integers and closed-enumeration stack tags flow into output).
- **Tool poisoning via tampered semgrep-rules clone** (ATLAS AML.T0058 / OWASP LLM03 / NIST Manage 3.2) — bounded by SHA-pin; bumping the pin is a human-reviewed PR.
- **Model output hijack** (ATLAS AML.T0048 / OWASP LLM02 / NIST Map 1.1) — every emitted file passes the structural-contract test suite BEFORE the PR is opened. The PR is the human-review surface for "did Claude emit what we asked."

### Cache directory residual risk

`~/.cache/sldo/semgrep-rules/<SHA>/` grows unbounded across pin bumps. Disk-quota DoS is theoretical for solo OSS maintainers but real for shared CI environments. v1 ships no pruning; users prune by hand. A pruning skill is a Phase-2 candidate.
