# Variation template — CWE-295 Improper Certificate Validation

## Why this class

TLS configuration mistakes that disable cert validation. The most common shapes:
- `reqwest::ClientBuilder::danger_accept_invalid_certs(true)`
- `reqwest::ClientBuilder::danger_accept_invalid_hostnames(true)`
- `rustls::ClientConfig::dangerous().set_certificate_verifier(custom)` with a verifier that accepts everything.

Existing `semgrep-rules/rust/lang/security/reqwest-accept-invalid.yml` covers a subset; this pack's rule explicitly does NOT vendor that text (Trail of Bits AGPL caution is not relevant for upstream Semgrep Rules License, but our rule is authored fresh anyway).

## Sink shapes (3 minimum)

### 1. `reqwest_danger_accept_invalid_certs`

```rust
let client = reqwest::Client::builder()
    .danger_accept_invalid_certs(true)   // BAD
    .build()?;
```

### 2. `reqwest_danger_accept_invalid_hostnames`

```rust
let client = reqwest::Client::builder()
    .danger_accept_invalid_hostnames(true)   // BAD
    .build()?;
```

### 3. `rustls_dangerous_config_with_no_client_auth_or_custom_verifier`

```rust
use rustls::client::ServerCertVerifier;

struct AcceptAll;
impl ServerCertVerifier for AcceptAll { /* accepts every cert */ }

let mut config = rustls::ClientConfig::builder()
    .with_safe_defaults()
    .with_root_certificates(rustls::RootCertStore::empty())
    .with_no_client_auth();
config
    .dangerous()
    .set_certificate_verifier(std::sync::Arc::new(AcceptAll));
```

## Good (silent) shapes for the fixture

```rust
let client_safe = reqwest::Client::new();   // default validation
let client_with_pinned_root = reqwest::Client::builder()
    .add_root_certificate(my_pinned_root)
    .build()?;
```

## Composition note

This is the one CWE in the v1 pack with explicit upstream Semgrep coverage. The wedge keeps it for two reasons: (1) it's a high-confidence sink shape, (2) the user pain anchor included webapp issues which often require `reqwest` and TLS client config in test environments. Where upstream covers a sink, the rule explicitly cites the upstream rule id in `metadata.source-of-bug-shape`.
