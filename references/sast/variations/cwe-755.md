# Variation template — CWE-755 panic-DoS in functions returning `Result`

## Why this class

A function declared to return `Result<T, E>` should never panic on attacker-controlled input — the contract is "errors are values; failure is in the type signature." When a `Result`-returning function panics, the caller's error-handling path is bypassed; the request handler dies, the worker thread aborts, or — with `panic = 'abort'` — the entire process exits.

Common in HTTP handler crates (axum, actix-web) where deserialization sits on the request body: `serde_json::from_str(body).unwrap()` panics on malformed JSON, killing the handler.

## Sink shapes (all 4 MUST appear in the rule's `pattern-either`)

### 1. `unwrap`

```rust
let user: User = serde_json::from_str(body).unwrap();
```

### 2. `expect`

```rust
let user: User = serde_json::from_str(body).expect("body must be valid JSON");
```

### 3. `explicit_panic_macro`

```rust
let user: User = match serde_json::from_str::<User>(body) {
    Ok(u) => u,
    Err(_) => panic!("invalid JSON"),
};
```

### 4. `propagate_question_mark_unhandled`

This is subtle: `?` itself is fine, but inside a function returning `Result` whose caller panics on the `Err` arm, the panic propagates. The bad shape: `?`-typed expression assigned to a variable then `unwrap`ed in the same fn body.

```rust
let parsed: User = (|| -> Result<User, _> {
    let raw = serde_json::from_str(body)?;
    Ok(raw)
})().unwrap();
```

## Pattern-not-inside

The rule MUST exclude `#[cfg(test)] mod tests { ... }` blocks where panicking on bad input is intentional:

```yaml
pattern-not-inside: |
  #[cfg(test)]
  mod tests { ... }
```

Optionally also exclude `#[cfg(test)] fn $F(...) { ... }` for inline test fns.

## Good (silent) shapes for the fixture

```rust
fn handler_safe(body: &str) -> Result<User, MyError> {
    let user: User = serde_json::from_str(body)
        .map_err(|e| MyError::BadRequest(e.to_string()))?;
    Ok(user)
}

fn handler_with_explicit_default(body: &str) -> User {
    serde_json::from_str(body).unwrap_or_default()
}
```

`unwrap_or_default` is acceptable when the caller has decided that "missing is empty" is the right semantic. The rule SHOULD NOT fire on this.

## Composition note

Clippy's `clippy::unwrap_used` and `clippy::expect_used` cover the sink surface broadly but with no per-handler context (they fire on every occurrence). This Semgrep rule is more targeted: it only fires inside functions that return `Result<...>` AND outside test blocks. **Compose, don't replace.**
